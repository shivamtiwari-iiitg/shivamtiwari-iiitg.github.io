01 Framework & how to drive the round

The repeatable skeleton every system design interview follows — with what to say, when, and why.

D01The 45-minute mobile system design interview — how to drive it.Mid
In plain terms. An interviewer who gives you a blank-slate problem is testing whether you impose structure. The best candidates drive the 45 minutes like a structured conversation, not a monologue — they state assumptions, get quick buy-in, then go deep on the interesting parts.

The six-phase skeleton (approximate time allocation):

  • 1. Clarify & scope (5 min) — ask before you draw. Functional requirements (what does it do?), then non-functional (scale, offline, latency, battery, data usage, a11y). Nail down what's in vs out of scope and say it aloud. Interviewers reward candidates who ask precise questions.
  • 2. API contract (5 min) — define the surface: endpoints, request/response shapes, pagination style, error model, auth. This anchors the rest and signals systems thinking.
  • 3. High-level design / HLD (10 min) — draw the components: client layers, server, database, CDN. Describe data flow for the main use case. Keep it high-level, no classes yet.
  • 4. Client architecture deep-dive (15 min) — this is the mobile-specific part interviewers care most about. Layer structure, state model, threading, caching strategy, offline behavior. Spend the most time here.
  • 5. LLD / specific deep-dives (5 min) — key classes, interfaces, data models for the hardest part. Don't try to spec the whole thing — pick the component the interviewer seems most interested in.
  • 6. Trade-offs & non-functionals (5 min) — explicitly state what you'd do differently at larger scale, what you chose NOT to do and why, failure modes, observability. Seniority shows in trade-offs.
The mobile-specific lens: web SD rounds focus on servers and databases. Mobile rounds want the client architecture — layers, single source of truth, threading, caching, lifecycle, memory. Always answer through that lens first, even if backend comes up.
D02Offset vs cursor pagination — design and trade-offs.Mid
In plain terms. Offset says "give me items 20–40." Cursor says "give me items after this specific one." Cursor is better for feeds because live data (new inserts) breaks offset indexing.
// Offset — simple, breaks on live data
GET /api/posts?page=3&size=20
// Problem: if 5 posts are inserted before page 2, page 3 now repeats items from page 2

// Cursor — stable
GET /api/posts?after=eyJpZCI6MTA1fQ==&size=20
// Server decodes cursor → finds posts with id < 105, orders descending
// New inserts don't affect positions of older items

Client storage pattern (Android / RN): store nextCursor alongside cached pages in Room. On resume, use the stored cursor to top-up from network.

// Room DAO
@Query("SELECT * FROM post ORDER BY id DESC LIMIT :limit")
fun page(limit: Int): List<Post>

@Query("SELECT * FROM post WHERE id < :cursor ORDER BY id DESC LIMIT :limit")
fun pageAfter(cursor: Long, limit: Int): List<Post>
D03Design a multi-tier caching strategy — HLD + LLD.Hard
In plain terms. Three tiers, fastest to slowest: memory (lost on app kill) → disk (survives restarts) → network. Reads go top-down; writes go bottom-up. The policy per tier determines how fresh vs cheap each response is.

HLD:

UI → ViewModel → Repository
                    ↓
         [1] Memory LRU cache (hot objects, bounded)
                    ↓ miss
         [2] Disk / Room (structured data) + DiskLRU (blobs)
                    ↓ miss or stale
         [3] Network (OkHttp + HTTP cache layer)
                    ↓
              Server → CDN

LLD — Android:

class ArticleRepository(
    private val memCache: LruCache<String, Article>,
    private val dao: ArticleDao,
    private val api: ArticleApi
) {
    fun observe(id: String): Flow<Article> = flow {
        // 1. Emit memory cache immediately
        memCache[id]?.let { emit(it); return@flow }

        // 2. Emit disk while network fetches
        emitAll(dao.observe(id)
            .onStart {
                // 3. Kick off network refresh in parallel
                viewModelScope.launch(Dispatchers.IO) {
                    val fresh = api.article(id)
                    dao.upsert(fresh)           // disk update → Flow re-emits
                    memCache.put(id, fresh)     // warm memory tier
                }
            }
        )
    }
}

D04Offline-first with conflict resolution — HLD + LLD.Hard
In plain terms. The local database is the source of truth; the UI never waits on the network. Writes go to an "outbox" table first; a background job drains it to the server when connectivity returns.

HLD:

User action → Repository
                ↓
     1. Write to local DB (immediate, optimistic)
     2. Insert into Outbox table (pending mutation)
     3. UI observes DB → instant update
                ↓ (background)
     WorkManager drains Outbox → POST to server
     Server returns canonical id/timestamp
     Repository updates local row: pending → synced

LLD — Android (Room + WorkManager):

// Outbox entity
@Entity
data class OutboxEntry(
    @PrimaryKey val id: String = UUID.randomUUID().toString(),
    val entityType: String,    // "post", "comment", "reaction"
    val payload: String,       // JSON
    val status: Status = Status.PENDING,
    val retries: Int = 0
)

// Repository write — atomic: both in one transaction
@Transaction
suspend fun createPost(post: Post) {
    postDao.insert(post.copy(syncStatus = SyncStatus.PENDING))
    outboxDao.insert(OutboxEntry(entityType = "post", payload = post.toJson()))
}

// WorkManager sync job
class SyncWorker(ctx: Context, params: WorkerParameters) : CoroutineWorker(ctx, params) {
    override suspend fun doWork(): Result {
        val entries = outboxDao.pendingEntries()
        entries.forEach { entry ->
            try {
                api.sync(entry.payload)
                outboxDao.markSynced(entry.id)
            } catch (e: Exception) {
                outboxDao.incrementRetry(entry.id)
                if (entry.retries >= 3) outboxDao.markFailed(entry.id)
            }
        }
        return Result.success()
    }
}

Conflict resolution strategies:

  • Last-write-wins: server timestamp decides. Simple, loses concurrent edits. Good for most non-collaborative data.
  • Per-field merge: server merges changed fields independently. Better for settings/profile edits where fields are independent.
  • CRDT: data types that merge commutatively — no single winner needed. Best for collaborative editing (too complex for most apps).
D05Token auth with silent refresh — HLD + LLD.Hard
In plain terms. Short-lived access tokens (15min) + long-lived refresh tokens (30 days). A 401 triggers a token refresh transparently — the original request retries with the new token, and the user never sees a login page mid-session.

HLD:

Request → OkHttp/Axios Interceptor
    ↓ adds "Authorization: Bearer <access>"
API call → 401?
    ↓ yes
Authenticator / Response Interceptor
    ↓ single-flight mutex (only ONE refresh at a time)
POST /auth/refresh { refreshToken }
    ↓ success
Store new access + refresh tokens
Retry original request
    ↓ fail (refresh expired)
Clear session → navigate to Login

LLD — Android (OkHttp Authenticator):

class TokenAuthenticator(
    private val tokenStore: TokenStore,
    private val authApi: AuthApi
) : Authenticator {
    private val mutex = Mutex()

    override fun authenticate(route: Route?, response: Response): Request? {
        // Single-flight: only one refresh runs even if 10 requests 401 simultaneously
        return runBlocking {
            mutex.withLock {
                // Already refreshed while waiting? Skip
                val current = tokenStore.accessToken
                if (current != response.request.header("Authorization")?.drop(7)) {
                    return@withLock response.request.newBuilder()
                        .header("Authorization", "Bearer $current").build()
                }
                try {
                    val refreshed = authApi.refresh(tokenStore.refreshToken)
                    tokenStore.save(refreshed)
                    response.request.newBuilder()
                        .header("Authorization", "Bearer ${refreshed.accessToken}").build()
                } catch (e: Exception) {
                    tokenStore.clear(); // force logout
                    null
                }
            }
        }
    }
}

LLD — React/RN (Axios interceptor):

let refreshPromise: Promise<string> | null = null;   // single-flight

axios.interceptors.response.use(null, async (error) => {
    if (error.response?.status !== 401 || error.config._retry) throw error;
    error.config._retry = true;
    if (!refreshPromise) {

            .then(t => { saveTokens(t); return t.accessToken; })

D06Design an image-loading library (Glide / Coil) — HLD + LLD.Hard
Clarifications to make in the interview: bitmap in memory or vector? Max concurrent downloads? Lifecycle awareness required? Transformations (resize, crop, filters)?

HLD:

Request(url, target: ImageView/Composable, size)
    ↓
Key = hash(url + size + transformations)
    ↓
[1] Memory LRU cache → hit: set bitmap → done
    ↓ miss
[2] Disk LRU cache → hit: decode → set → update memory
    ↓ miss
[3] Deduplication layer → one in-flight request per key
    ↓
[4] Network → download bytes → write to disk
    ↓
[5] Decode thread → BitmapFactory.decodeStream(opts)
    → downsample to target size via inSampleSize
    ↓
[6] Transformation pipeline (crop, round corners, blur)
    ↓
[7] Main thread → setImageBitmap (cancel if view recycled)

LLD — Android (Kotlin):

class ImageLoader(maxMemKb: Int) {
    private val memCache = object : LruCache<String, Bitmap>(maxMemKb) {
        override fun sizeOf(k: String, v: Bitmap) = v.byteCount / 1024
    }
    private val diskCache = DiskLruCache.open(cacheDir, maxDiskKb)
    private val inFlight = ConcurrentHashMap<String, Deferred<Bitmap?>>()

    fun load(url: String, into: ImageView, size: Size) {
        val key = cacheKey(url, size)
        // Cancel any previous request on this view
        into.tag?.let { (it as? Job)?.cancel() }

        val job = ioScope.launch {
            val bmp = memCache[key]
                ?: diskCache.get(key)?.also { memCache.put(key, it) }
                ?: fetchAndDecode(url, size, key)
            withContext(Main) {
                if (into.tag == coroutineContext[Job]) into.setImageBitmap(bmp)
            }
        }
        into.tag = job
    }
}

private suspend fun fetchAndDecode(url: String, size: Size, key: String): Bitmap? {
    // Deduplicate: reuse in-flight request for same key
    return inFlight.getOrPut(key) {
        ioScope.async {
            httpClient.stream(url).use { stream ->
                val opts = BitmapFactory.Options()
                opts.inJustDecodeBounds = true
                BitmapFactory.decodeStream(stream, null, opts)
                opts.inSampleSize = calcSampleSize(opts, size)
                opts.inJustDecodeBounds = false
                BitmapFactory.decodeStream(reopened, null, opts)
                    .also { diskCache.put(key, it) }
            }
        }
    }.also { inFlight.remove(key) }.await()
}

Trade-offs: aggressive memory caching prevents OOM but risks evicting large bitmaps unexpectedly — bound to 1/8 of maxMemory. Disk dedupe saves bandwidth but the dedup map can grow — bound by size and TTL. Lifecycle tie-in (cancel on view recycle) is critical for list scroll performance.

D07Design real-time chat — HLD + LLD (Android, RN, React).Hard
Clarify: 1:1 or group? Max group size? Read receipts? Media? Offline send? Push for background? History pagination depth?

HLD:

Client ←→ WebSocket Gateway ←→ Message Service ←→ Kafka
                                       ↓
                                 Message DB (Cassandra)
                                       ↓
                                 Push (FCM/APNs) when recipient offline

Client DB (SQLite/Room):
  messages(id, conversationId, senderId, body, status, serverTimestamp, clientId)
  conversations(id, lastMessage, lastTimestamp, unreadCount)

Message flow:
1. User sends → write locally (status=SENDING, clientId=uuid)
2. POST /messages or WS send
3. Server returns {id, serverTimestamp} → update row (status=SENT)
4. Server pushes to recipient via WS → recipient stores
5. Recipient sends ACK → sender updates status=DELIVERED
6. Recipient opens conversation → sends SEEN → status=READ

LLD — Android:

@Entity
data class Message(
    @PrimaryKey val id: String,
    val clientId: String,       // idempotency key for dedup
    val conversationId: String,
    val body: String,
    val status: MessageStatus,  // SENDING, SENT, DELIVERED, READ, FAILED
    val serverTimestamp: Long?,  // null until server confirms
    val clientTimestamp: Long   // for optimistic ordering
)

class ChatRepository(
    private val dao: MessageDao,
    private val ws: WebSocketClient,
    private val api: ChatApi
) {
    fun messages(convId: String) = dao.observeMessages(convId)  // Flow<List<Message>>

    suspend fun send(convId: String, body: String) {
        val clientId = UUID.randomUUID().toString()
        // 1. Optimistic insert
        dao.insert(Message(id = clientId, clientId = clientId, status = SENDING, ...))
        try {
            // 2. Send via WS or REST
            val confirmed = api.sendMessage(convId, body, clientId)
            dao.update(clientId, serverId = confirmed.id, status = SENT)
        } catch (e: Exception) { dao.update(clientId, status = FAILED) }
    }

    fun handleIncoming(msg: WsMessage) {
        dao.insertOrIgnore(msg.toEntity())    // idempotent via clientId
        ws.send(AckFrame(msg.id))
    }
}

LLD — React / React Native:

// WebSocket management hook
function useChatSocket(conversationId) {
    const queryClient = useQueryClient();
    useEffect(() => {
        const ws = new WebSocket(`wss://api.app/ws/chat/${conversationId}`);
        ws.onmessage = ({ data }) => {
            const msg = JSON.parse(data);
            queryClient.setQueryData(
                ['messages', conversationId],

        };
        ws.onclose = () => scheduleReconnect();
        return () => ws.close();
    }, [conversationId]);
}

Ordering gotcha: never use clientTimestamp for display order in a multi-sender conversation — clocks drift. Use server timestamp as the primary sort key, falling back to client timestamp only for the sender's own optimistically-inserted messages before confirmation arrives.

// RemoteMediator handles the network→Room write
class FeedRemoteMediator(
    private val dao: FeedDao,
    private val api: FeedApi
) : RemoteMediator<Int, FeedItem>() {
    override suspend fun load(loadType: LoadType, state: PagingState<Int, FeedItem>): MediatorResult {
        val cursor = when (loadType) {
            REFRESH -> null
            PREPEND -> return MediatorResult.Success(endOfPaginationReached = true)
            APPEND -> dao.lastCursor() ?: return MediatorResult.Success(true)
        }
        return try {
            val page = api.feed(cursor)
            if (loadType == REFRESH) dao.clearAll()
            dao.insertAll(page.items)
            dao.saveCursor(page.nextCursor)
            MediatorResult.Success(endOfPaginationReached = page.nextCursor == null)
        } catch (e: Exception) { MediatorResult.Error(e) }
    }
}

D09Design resilient photo upload — HLD + LLD (Android + RN).Hard
Clarify: Max file size? Resume on kill? Progress visibility? Compression / format conversion?

HLD:

User picks photo → Client
    ↓
1. Compress/resize on IO thread (target ~1MB for photos)
2. POST /uploads/initiate → server returns uploadId + presigned S3 URL
3. Split into 5MB chunks
4. Upload chunks to S3 via presigned URLs (parallel, max 3 at once)
5. Record uploaded chunk IDs locally (persist to survive kill)
6. POST /uploads/complete { uploadId, chunks } → server assembles
7. POST /posts { mediaUrl } → create the post

LLD — Android (WorkManager + Room):

@Entity
data class UploadJob(
    @PrimaryKey val id: String = UUID.randomUUID().toString(),
    val localPath: String,
    val uploadId: String? = null,
    val completedChunks: List<Int> = emptyList(),
    val status: UploadStatus = UploadStatus.QUEUED,
    val idempotencyKey: String = UUID.randomUUID().toString()
)

class UploadWorker(ctx: Context, params: WorkerParameters) : CoroutineWorker(ctx, params) {
    override suspend fun doWork(): Result {
        val jobId = inputData.getString("jobId") ?: return Result.failure()
        val job = uploadDao.get(jobId)

        // Resume: skip already-uploaded chunks
        val uploadId = job.uploadId ?: api.initiate(job.idempotencyKey).also {
            uploadDao.update(job.copy(uploadId = it))
        }

        chunks(job.localPath)
            .filter { it.index !in job.completedChunks }
            .forEach { chunk ->
                api.uploadChunk(uploadId, chunk)
                uploadDao.markChunkDone(jobId, chunk.index)  // persist immediately
            }

        api.complete(uploadId)
        uploadDao.update(job.copy(status = UploadStatus.DONE))
        return Result.success()
    }
}

// Schedule — survives app kill, retries on network
WorkManager.getInstance(context).enqueue(
    OneTimeWorkRequestBuilder<UploadWorker>()
        .setConstraints(Constraints(NetworkType.CONNECTED))
        .setBackoffCriteria(BackoffPolicy.EXPONENTIAL, 15, TimeUnit.SECONDS)
        .setInputData(workDataOf("jobId" to job.id))
        .build()
)

class AppMessagingService : FirebaseMessagingService() {
    override fun onNewToken(token: String) {
        // Token rotates — always re-register
        lifecycleScope.launch { api.registerDevice(token, Platform.ANDROID) }
    }

    override fun onMessageReceived(msg: RemoteMessage) {
        when (msg.data["type"]) {
            "new_message" -> {
                // Data push: sync the conversation silently
                WorkManager.getInstance(this).enqueue(SyncConversationRequest(msg.data["convId"]!!))
            }
            "promo" -> {

                    .setContentText(msg.notification?.body)
                    .setContentIntent(deepLinkIntent(msg.data["target"]))  // ← deep link
                    .setAutoCancel(true)
                    .build()
                    .let { nm.notify(System.currentTimeMillis().toInt(), it) }

        }

}

object Analytics {
    private val queue = Channel<Event>(capacity = 100, onBufferOverflow = DROP_OLDEST)
    private val scope = CoroutineScope(Dispatchers.IO + SupervisorJob())

    fun init() {
        scope.launch {
            val batch = mutableListOf<Event>()
            while (isActive) {
                withTimeoutOrNull(30_000) {         // flush every 30s or on size
                    while (batch.size < 50) {
                        batch.add(queue.receive())
                    }
                }
                if (batch.isNotEmpty()) {
                    eventDao.insertAll(batch.map { it.toEntity() })
                    batch.clear()
                }
            }
        }
        // Periodic flush via WorkManager
    }

    fun track(name: String, props: Map<String, Any> = emptyMap()) {
        queue.trySend(Event(
            id = UUID.randomUUID().toString(),
            name = name,
            props = props + session.context(),    // auto-enrich
            timestamp = System.currentTimeMillis()
        ))
    }
}

// FlushWorker
class AnalyticsFlushWorker(ctx: Context, params: WorkerParameters) : CoroutineWorker(ctx, params) {
    override suspend fun doWork(): Result {
        val batch = eventDao.unsynced(limit = 200)
        if (batch.isEmpty()) return Result.success()
        return try {
            api.postEvents(batch)
            eventDao.delete(batch.map { it.id })
            Result.success()
        } catch (e: Exception) { Result.retry() }
    }
}

LLD — React Native:

class Analytics {
    queue = []; flushTimer = null;

    track(name, props = {}) {
        this.queue.push({ name, props: { ...props, ...this.context() }, ts: Date.now() });
        if (this.queue.length >= 50) this.flush();
        else if (!this.flushTimer) this.flushTimer = setTimeout(() => this.flush(), 30000);
    }

    async flush() {
        clearTimeout(this.flushTimer); this.flushTimer = null;
        const batch = this.queue.splice(0, 100);    // drain up to 100
        if (!batch.length) return;
        try { await api.postEvents(batch); }
        catch { this.queue.unshift(...batch); }    // re-queue on failure
    }

D12Design an app-wide overlay / intervention framework (my ShareChat work) — HLD + LLD.Hard
The problem: 20+ features each want to show overlays (popups, tooltips, snackbars). Without a central arbiter they stack, collide, and fire at bad moments. Goal: one system that decides what shows, when, and in what order.

HLD:

Feature modules → InterventionManager.enqueue(Intervention)
                        ↓
              Priority Queue (ordered by priority + time)
                        ↓
              Arbitration rules:
                - max one overlay at a time
                - only show in allowed screens/states
                - honor per-intervention frequency caps (Room: shown_at, shown_count)
                - honor global pacing (max N overlays per session)
                - remote-config gate (feature flag)
                        ↓
              InterventionHost (Activity/Fragment/ComposeHost)
              renders current intervention from StateFlow
                        ↓
              User dismisses → notify queue → dequeue next

LLD — Android (Kotlin Coroutines Actor):

data class Intervention(
    val id: String,
    val priority: Int,
    val type: InterventionType,
    val eligibleScreens: Set<String>,
    val cooldownMs: Long = 0,
    val maxPerSession: Int = Int.MAX_VALUE,
)

class InterventionManager @Inject constructor(
    private val remoteConfig: RemoteConfig,
    private val dao: InterventionDao,
    private val lifecycleState: StateFlow<LifecycleState>
) {
    private val _current = MutableStateFlow<Intervention?>(null)
    val current = _current.asStateFlow()

    // Actor: serializes all enqueue/dismiss operations — no locks needed
    private val actor = CoroutineScope(Dispatchers.Default).actor<Cmd> {
        val queue = PriorityQueue<Intervention>(compareByDescending { it.priority })
        for (cmd in channel) {
            when (cmd) {
                is Cmd.Enqueue -> if (isEligible(cmd.item)) queue.add(cmd.item)
                is Cmd.Dismiss -> { dao.record(cmd.id); _current.value = null }
            }
            if (_current.value == null && lifecycleState.value == RESUMED) {
                _current.value = queue.poll()
            }
        }
    }

    fun enqueue(item: Intervention) { actor.trySend(Cmd.Enqueue(item)) }
    fun dismiss(id: String) { actor.trySend(Cmd.Dismiss(id)) }

    private suspend fun isEligible(item: Intervention): Boolean {
        val record = dao.get(item.id) ?: return remoteConfig.isEnabled(item.id)
        return record.shownCount < item.maxPerSession
            && System.currentTimeMillis() - record.lastShownAt > item.cooldownMs
    }
}

Company-wide adoption pattern: each feature registers its interventions via @IntoSet multibinding — the manager collects them without knowing any individual feature. Features don't know each other exists; the manager arbitrates centrally. This is what enabled adoption across 20+ features at ShareChat without coupling.

class FeatureFlags @Inject constructor(
    private val store: DataStore<Preferences>,
    private val api: ConfigApi,
    private val defaults: FlagDefaults
) {

    fun <T> get(flag: Flag<T>): T {
        val cached = store.data.first().get(flag.key)
        return flag.deserialize(cached) ?: flag.default
    }

        val remote = api.fetchConfig()
        store.edit { prefs ->
            remote.flags.forEach { (k, v) -> prefs[stringPreferencesKey(k)] = v }
        }
    }

    // Apply flags at a safe boundary — next screen transition, not mid-flow
}

function useAutocomplete(minLength = 2) {
    const [query, setQuery] = useState('');
    const [suggestions, setSuggestions] = useState<string[]>([]);
    const [selected, setSelected] = useState(-1);
    const cacheRef = useRef(new Map<string, string[]>());
    const abortRef = useRef<AbortController>();

    const search = useMemo(
        () => debounce(async (q: string) => {
            if (q.length < minLength) { setSuggestions([]); return; }
            if (cacheRef.current.has(q)) { setSuggestions(cacheRef.current.get(q)!); return; }
            abortRef.current?.abort();
            abortRef.current = new AbortController();
            try {
                const res = await fetch(`/suggest?q=${q}`, { signal: abortRef.current.signal });
                const data: { suggestions: string[] } = await res.json();
                cacheRef.current.set(q, data.suggestions);
                setSuggestions(data.suggestions);
            } catch (e) { if ((e as Error).name !== 'AbortError') throw e; }
        }, 200),
        [minLength]
    );

    const handleKeyDown = (e: React.KeyboardEvent) => {
        if (e.key === 'ArrowDown') setSelected(s => Math.min(s + 1, suggestions.length - 1));
        if (e.key === 'ArrowUp') setSelected(s => Math.max(s - 1, -1));
        if (e.key === 'Escape') setSuggestions([]);
    };

    return { query, setQuery, suggestions, selected, handleKeyDown, onChange: search };
}

// Accessible render
<div role="combobox" aria-expanded={suggestions.length > 0} aria-haspopup="listbox">

        aria-autocomplete="list"
        aria-controls="suggestions"
        aria-activedescendant={selected >= 0 ? `option-${selected}` : undefined}
        onKeyDown={handleKeyDown}
    />
    <ul id="suggestions" role="listbox">
        {suggestions.map((s, i) => (
            <li key={s} id={`option-${i}`} role="option" aria-selected={i === selected}>
                <Highlight text={s} query={query} />
            </li>
        ))}
    </ul>
</div>
D15Design a rate limiter — HLD + LLD (client + server).Hard
Clarify: Per-user, per-IP, or per-API-key? Distributed (multi-server)? Hard reject or queue?

HLD:

Algorithms:
  Token bucket: tokens refill at rate R, each request spends 1, burst up to bucket size
  Sliding window: count requests in rolling time window, reject if over limit

Server (distributed):
  Redis key: "rl:{userId}:{windowStart}"
  INCR atomically → compare to limit → reject with 429 + Retry-After header

Client:
  Respect Retry-After header
  Exponential backoff with jitter on 429
  Proactive client-side throttle to avoid hitting the limit at all

LLD — Server (Node.js + Redis):

async function slidingWindowLimit(userId: string, limit = 100, windowMs = 60000) {
    const now = Date.now();
    const windowStart = now - windowMs;
    const key = `rl:${userId}`;

    const [count] = await redis.pipeline()
        .zremrangebyscore(key, '-inf', windowStart)  // remove old entries
        .zadd(key, now, `${now}-${Math.random()}`)   // add this request
        .zcard(key)                                     // count in window
        .expire(key, windowMs / 1000)
        .exec();

    if (count > limit) {
        const oldest = await redis.zrange(key, 0, 0, 'WITHSCORES');
        const retryAfter = Math.ceil((+oldest[1] + windowMs - now) / 1000);
        throw { status: 429, retryAfter };
    }
}

LLD — Client React (proactive throttle + 429 handling):

class RateLimitedClient {
    private tokens = 10;
    private lastRefill = Date.now();
    private readonly refillRate = 10 / 60000;  // 10 tokens per minute

    async request<T>(fn: () => Promise<T>): Promise<T> {
        this.refill();
        if (this.tokens < 1) throw new Error('client rate limited');
        this.tokens--;
        try { return await fn(); }
        catch (e: any) {
            if (e.status === 429) {
                const wait = (e.retryAfter ?? 60) * 1000;
                await sleep(wait);
                return this.request(fn);   // retry once after waiting
            }
            throw e;
        }
    }

    private refill() {
        const elapsed = Date.now() - this.lastRefill;
        this.tokens = Math.min(10, this.tokens + elapsed * this.refillRate);
        this.lastRefill = Date.now();
    }
}
D16Design a collaborative editor — HLD + LLD (React).Hard
Clarify: Rich text or plain? Max concurrent editors? Offline editing? Conflict strategy — OT or CRDT?

HLD:

import * as Y from 'yjs';
import { WebsocketProvider } from 'y-websocket';
import { IndexeddbPersistence } from 'y-indexeddb';
import { useEditor } from '@tiptap/react';
import Collaboration from '@tiptap/extension-collaboration';
import CollaborationCursor from '@tiptap/extension-collaboration-cursor';

function CollabEditor({ docId, userId }) {
    const ydoc = useMemo(() => new Y.Doc(), []);

    useEffect(() => {
        // Persist to IndexedDB for offline editing
        const idb = new IndexeddbPersistence(`doc-${docId}`, ydoc);
        // Sync via WebSocket when online
        const ws = new WebsocketProvider(`wss://collab.api/docs/${docId}`, docId, ydoc, {
            connect: navigator.onLine,
        });
        const onOnline = () => ws.connect();
        const onOffline = () => ws.disconnect();
        window.addEventListener('online', onOnline);
        window.addEventListener('offline', onOffline);
        return () => { ws.destroy(); idb.destroy(); };
    }, [docId, ydoc]);

    const editor = useEditor({
        extensions: [
            Collaboration.configure({ document: ydoc }),
            CollaborationCursor.configure({
                provider: wsProviderRef.current,
                user: { name: userId, color: '#FFB257' },    // remote cursors
            }),
        ],
    });

    return <EditorContent editor={editor} />;
}

Why CRDT (Yjs) over OT: OT requires a central server that's always available to sequence ops; CRDT merges any two states in any order, so offline editing then sync just works. The trade-off is more memory (each CRDT document carries history for merging) and less fine-grained conflict resolution for complex semantics (hard to express "don't insert inside a deleted range").

function InfiniteFeed() {
    const { data, fetchNextPage, hasNextPage } = useInfiniteQuery({
        queryKey: ['feed'],
        queryFn: ({ pageParam }) => api.feed({ after: pageParam }),
        getNextPageParam: last => last.nextCursor,
    });

    const items = useMemo(() =>
        data?.pages.flatMap(p => p.items) ?? [], [data]
    );

    // Virtualizer — only renders visible rows in the DOM
    const rowVirtualizer = useVirtualizer({
        count: items.length + (hasNextPage ? 1 : 0),  // +1 for loader sentinel
        getScrollElement: () => containerRef.current,
        estimateSize: () => 200,   // estimated row height
        overscan: 3,               // render 3 extra rows above/below viewport
    });

    useEffect(() => {
        const lastRow = rowVirtualizer.getVirtualItems().at(-1);
        if (lastRow?.index >= items.length - 1 && hasNextPage) fetchNextPage();
    }, [rowVirtualizer.getVirtualItems(), hasNextPage]);

    return (
        <div ref={containerRef} style={{ overflow: 'auto', height: '100vh' }}>
            <div style={{ height: rowVirtualizer.getTotalSize() }}>
                {rowVirtualizer.getVirtualItems().map(vRow => (
                    <div key={vRow.key} style={{ transform: `translateY(${vRow.start}px)`, position: 'absolute', width: '100%' }}>
                        {vRow.index < items.length
                            ? <FeedCard item={items[vRow.index]} />
                            : <LoadingSpinner />}
                    </div>
                ))}
            </div>
        </div>
    );
}

Scroll position restoration: when users navigate back, React Query's cache means data is instantly available. Restore scroll position by storing scrollTop in a ref or sessionStorage before navigation, and calling scrollTo(0, savedPos) after re-mount.
