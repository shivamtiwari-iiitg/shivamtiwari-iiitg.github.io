01 Framework & fundamentals

How to drive the round, plus the cross-cutting decisions every mobile design needs.

D01A repeatable framework for a mobile system-design round.Mid

Mobile design rounds reward client-side reasoning, not just boxes-and-arrows backend. Drive it in this order:

  1. Clarify & scope — functional requirements, then non-functional (offline, latency, battery, data usage, app size, accessibility). State assumptions and constraints (platforms, scale, API ownership).
  2. Define the API contract — endpoints, pagination style, payload shape, error model. This anchors everything.
  3. Client architecture — layers (UI → domain → data), state model, threading, single source of truth.
  4. Data & caching — memory/disk tiers, persistence (Room/SQLite), eviction, freshness/TTL, offline-first sync.
  5. Deep dives — the hard part the interviewer cares about (real-time, conflict resolution, media, list performance).
  6. Non-functional polish — performance, observability/metrics, security, rollout, failure modes.
Spend the most time on the deep dive, and always close on trade-offs — "I chose X over Y because…". Seniority shows in the trade-offs, not the diagram.
D02Offset vs cursor pagination — which for an infinite feed?Mid

Offset (?page=3&size=20) is simple but breaks on live data — inserts/deletes shift offsets, causing duplicates or skipped items, and deep offsets are slow on the DB. Cursor/keyset (?after=<opaque_token>) encodes the last item's sort key, so the next page is "everything after this point" — stable under inserts and O(log n) on an indexed column. Use cursor pagination for feeds and chat. The client stores the next cursor with the cached page and requests the next on scroll.

D03Design a multi-tier client caching strategy.Hard

Three tiers, read top-down, write bottom-up:

  • Memory (LRU) — hot objects/bitmaps, fastest, lost on process death. Bound by size.
  • Disk — SQLite/Room for structured data, a disk LRU for blobs/images. Survives restarts.
  • Network — source of truth, with HTTP caching (ETag/Cache-Control) at the OkHttp layer.

Freshness policies: cache-first (speed, may be stale), network-first (freshness, slower), and stale-while-revalidate (show cache instantly, refresh in background, emit the update) — usually the best UX. Drive it through the repository so the UI just observes a Flow. Add per-entity TTLs and invalidation on writes.

D04How do you build offline-first with conflict resolution?Hard

Make the local DB the source of truth; the UI never waits on the network. Writes go through an outbox pattern:

  1. Write optimistically to the local DB and mark the row pending (with a client-generated id).
  2. Enqueue a mutation in an outbox table; a WorkManager job drains it when connectivity returns, with exponential backoff and idempotency keys.
  3. On success, reconcile the server id and clear pending; on permanent failure, surface a retry/needs-attention state.

Conflict resolution: simplest is last-write-wins via server timestamps/versions; better is per-field merge or, for collaborative data, CRDTs/OT. Always send a version/updatedAt so the server can detect conflicts. Show clear pending/failed indicators rather than silently dropping edits.

D05Design token auth with silent refresh on mobile.Hard

Short-lived access token + long-lived refresh token. The access token is attached by an interceptor; on a 401, an authenticator uses the refresh token to get a new access token and retries the request.

  • Store refresh tokens in secure storage — Android Keystore / iOS Keychain, never plain prefs.
  • Single-flight the refresh: if many requests 401 at once, only one refresh runs (a mutex), others await its result — otherwise you stampede the refresh endpoint and may rotate tokens out of sync.
  • On refresh failure (revoked/expired), clear the session and route to login.
  • Support token rotation and server-side revocation; consider certificate pinning for the auth endpoints.

02 Mobile system designs

Worked client-side designs — the bread and butter of mobile rounds.

D06Design an image-loading library (Glide / Coil).Hard

Requirements: load a URL into an ImageView/Composable fast, cache aggressively, downsample to target size, cancel on view recycle, never OOM, support transformations.

Pipeline (request → result):

  1. Request — key = URL + target size + transformations. Check memory LRU (decoded bitmaps) → return instantly on hit.
  2. Miss → check disk LRU (encoded bytes). Hit → decode with sampling → put in memory cache.
  3. Miss → fetch over the network (dedupe concurrent requests for the same key), write bytes to disk.
  4. Decode off the main thread, downsampling via inSampleSize to the target dimensions to cap memory.
  5. Deliver on the main thread; apply a placeholder/error and a crossfade.

Lifecycle: tie each request to the view/lifecycle so scrolling a row off-screen cancels its in-flight request (critical for list performance — the RecyclerView/FlashList connection). Bound both caches; size the memory cache from maxMemory. Use a bitmap pool to reduce GC churn. Trade-offs: memory cache size vs OOM risk; aggressive prefetch vs data usage.

D07Design a real-time chat / messaging feature.Hard

Requirements: send/receive in real time, offline send, correct ordering, delivery/read receipts, history pagination, media.

Transport: a persistent WebSocket (or platform push for wake-ups) for low-latency bidirectional messages; fall back to long-poll. Heartbeats + reconnect with backoff; resume by passing the last-seen message id.

Client model: local SQLite is the source of truth, UI observes a Flow per conversation (cursor-paginated). Sending uses the outbox pattern: insert with a client id and sending status → optimistic render → socket/REST send → server returns canonical id + timestamp → update status to sent/delivered/read.

Ordering: order by server timestamp + a tiebreaker; use a monotonic sequence per conversation so a late-arriving message inserts correctly. Receipts are just lightweight events updating message rows. Sync on reconnect: pull everything after the last synced cursor. Media uploads go through the upload design (D09) and reference a blob id.

D08Design an infinite feed / timeline (e.g. ShareChat/Moj).Hard

Requirements: smooth infinite scroll, fresh content, offline view of last feed, mixed media, low jank at 180M-MAU scale.

  • Pagination: cursor-based; server returns a page + next cursor + ranking already applied (push-vs-pull fanout is a backend concern — note it but keep client focus).
  • Paging 3 + RemoteMediator: network writes pages into Room; the UI's PagingSource reads from Room → offline-first, single source of truth, automatic load states and retry.
  • Prefetch: trigger the next page a few items before the end; preload media for upcoming rows (bounded) so images/video are ready.
  • List performance: recycled rows (RecyclerView/FlashList), stable keys, view-type-specific holders, image requests cancelled on recycle, no work in onBind.
  • Freshness: pull-to-refresh resets the cursor; show "new posts" pill when newer content exists. Dedupe by id across pages.
  • Video: autoplay only the most-visible item, preload ±1, release players off-screen (player pool).
D09Design a resilient photo / file upload system.Hard

Requirements: upload large media reliably on flaky networks, survive app/process death, show progress, allow retry, don't block the UI.

  • Pre-process: compress/resize on a background thread before upload to save bytes and time.
  • Resumable, chunked upload: split into chunks; track which chunks succeeded so a drop resumes from the last good chunk (tus/S3 multipart style). The server assembles on completion.
  • Durability: run the upload in WorkManager (or a foreground service for user-initiated, must-finish-now uploads) so it survives backgrounding/reboot, with network constraints and exponential backoff.
  • Direct-to-storage: client requests a pre-signed URL, uploads straight to object storage (offload from app servers), then notifies the backend with the blob id.
  • UX: optimistic local entry with a progress indicator; idempotency key so retries don't create duplicates; clear failed state with manual retry.
D10Design a push notification system (client side).Mid

Registration: get an FCM/APNs token on launch, send it to the backend mapped to the user+device; refresh on token rotation and on logout invalidate it. Delivery: backend → FCM/APNs → device. Handle two types: display notifications (system shows them) and data/silent pushes (wake the app to sync, subject to OS background limits — coalesce, don't abuse battery).

Client handling: a messaging service receives the payload, builds a channel-categorized notification, and sets a deep link/PendingIntent so a tap routes to the right screen with the right back stack. Group/bundle related notifications, respect per-channel user prefs, and track delivered/opened for analytics. Add a server-side notification center fetch so users who had pushes off still see history.

D11Design an analytics / event-tracking SDK.Hard

Requirements: capture events with low overhead, never lose events (offline), never block the UI, batch efficiently, be resilient to crashes.

  • API: simple track(event, props); enrich automatically with session, device, app version, timestamp.
  • Persist first: write each event to a local queue (SQLite/Room) before anything else, off the main thread — so a crash or kill doesn't lose data.
  • Batching & flush: flush on size threshold, time interval, or app background; send compressed batches. Use WorkManager for guaranteed delivery with backoff; delete rows only after a 2xx.
  • Reliability: idempotency/dedupe ids, bounded queue with drop-oldest policy to cap disk, retry with jitter, sampling for high-volume events.
  • Privacy: consent gating, PII scrubbing, opt-out. Versioned event schema.

This is a registry-friendly design — multibinding a set of trackers (the Hilt @IntoSet pattern) lets features add events without coupling.

D12Design an app-wide overlay / intervention framework (my work).Hard

Problem: many features want to show overlays (popups, tooltips, coach-marks, snackbars, bottom sheets, force-update dialogs) across the app. Without coordination they collide, stack, or fire at bad moments. Goal: one system that arbitrates what shows, when, and in what order, lifecycle-aware and remotely controllable.

Design:

  • Central orchestrator — a single coordinator (I used a Kotlin Coroutines Actor to serialize requests without locks) owns a priority queue of intervention requests. Producers enqueue(Intervention); the actor decides the next to display.
  • Intervention contract — id, priority, type, eligibility predicate, display constraints (which screens/states it's allowed on), cooldown/frequency caps, and a remote-config gate.
  • Lifecycle awareness — only show when a valid host surface is in RESUMED; pause/resume the queue with the foreground surface; never show over an incompatible screen.
  • Arbitration rules — at most one overlay at a time; higher priority preempts or queues; honor per-intervention frequency caps and global pacing so users aren't spammed.
  • State & UI — MVI state drives a host that renders the current intervention; results (shown/dismissed/clicked) feed back for analytics and caps. Dagger-Hilt wires producers via multibinding so features register interventions without the core knowing them.

Trade-offs: centralization gives consistency and control but every feature must route through it (governance cost); serializing via an actor avoids race conditions at the cost of one logical owner. Adopted company-wide at ShareChat for exactly this reliability/UX consistency.

D13Design video playback for a short-video / livestream feed.Hard

Requirements: instant start, smooth scroll-to-next, low rebuffering on poor networks, battery/data conscious.

  • Adaptive bitrate (HLS/DASH) — the player picks a rendition matching current bandwidth, stepping down to avoid stalls. Server provides multiple renditions.
  • Player pool — reuse a small set of ExoPlayer instances across the list (like view recycling); never one player per row.
  • Prefetch — warm the next 1–2 videos (initial segments) while the current plays, so swiping is instant; cap prefetch on cellular.
  • Autoplay policy — play only the most-visible item; pause/release off-screen; mute by default.
  • Caching — a bounded media cache for recently seen segments; resume position tracking.
  • Live specifics — low-latency HLS, sync to live edge, graceful degradation, and a separate real-time channel (Agora/WebRTC) for interactive low-latency streams.
D14Design a feature-flag / remote-config system for the client.Mid

Goal: toggle features, run A/B tests, and kill-switch bad code without a release. Client: fetch a config blob on launch (and periodically), cache it locally, and expose typed accessors with safe defaults baked into the app (so first launch / offline still works). Evaluation can be server-side (server returns the resolved variant per user — consistent, simpler client) or client-side (rules + user attributes evaluated locally — works offline). Apply changes at a safe boundary (next screen/launch) to avoid mid-session flips. Add bucketing by stable user id for sticky A/B assignment, exposure logging for analytics, and a remote kill switch evaluated early in startup. Trade-off: fetch-on-launch freshness vs a flash of default behavior — mitigate with a short cached TTL and background refresh.

03 Web system designs

Frontend-leaning web problems that round out a full-stack mobile profile.

D15Design a typeahead / autocomplete (web).Hard

Client: debounce input (~200–300ms), require a min length, distinctUntilChanged, and cancel stale requests (AbortController) so only the latest query renders — the web twin of flatMapLatest. Cache results per prefix in an LRU so backspacing is instant. Handle keyboard navigation (↑/↓/Enter), ARIA combobox roles, and a loading/empty state.

Server: a prefix index — a trie or an inverted index (Elasticsearch) — returns top-K suggestions ranked by popularity/personalization, served from an edge cache/CDN since suggestions are highly cacheable. Precompute top completions per prefix for hot terms. Trade-off: client cache freshness vs server load; debounce latency vs request volume.

D16Design a rate limiter (and the client-side throttle).Hard

Algorithms:

  • Token bucket — tokens refill at a fixed rate; each request spends one; allows bursts up to bucket size. Most common.
  • Leaky bucket — requests drain at a constant rate; smooths bursts.
  • Fixed window — count per window; simple but allows 2× burst at window edges.
  • Sliding window log/counter — accurate, more memory.

Distributed: keep counters in Redis (atomic INCR/Lua) keyed by user/IP/API-key so all servers share state. Return 429 with Retry-After. Client side: respect Retry-After, back off with jitter, and throttle/debounce outgoing calls so you don't self-DDoS — the same token-bucket idea applied locally.

D17Design an image CDN / delivery service.Mid

Goal: deliver the right-sized, right-format image fast worldwide. Origin stores the master; an image service does on-the-fly transforms from URL params (?w=400&format=webp&q=80) — resize, crop, format negotiation (WebP/AVIF by Accept header), DPR-aware sizing. Results cache at the edge (CDN) keyed by the transform URL, so the transform runs once per variant. Add responsive delivery (srcset/sizes), lazy loading, LQIP/blur-up placeholders, and signed URLs to prevent transform abuse. Trade-off: generate-on-demand (flexible, first-hit latency) vs pre-generate common sizes (fast, storage cost) — usually do both: pre-warm hot sizes, generate the long tail on demand.

D18Design a collaborative editor (Google-Docs-style) — the sync core.Hard

Challenge: many users edit the same document concurrently; everyone must converge to the same state despite latency and offline edits.

Two approaches:

  • Operational Transformation (OT) — clients send operations (insert/delete at index); a server transforms each op against concurrent ones so they apply consistently. Proven (Google Docs) but the transform logic is famously tricky.
  • CRDTs — data types (e.g. RGA/Yjs) that merge commutatively, so order doesn't matter and you can merge peer-to-peer and offline without a central transformer. More memory/metadata, simpler convergence — the modern default.

Transport: WebSocket for low-latency op broadcast; presence/cursors as ephemeral metadata. Offline: buffer local ops and merge on reconnect (CRDTs shine here). Persist with periodic snapshots + op log for fast load and history. Show remote cursors and selections for awareness.
