D01The 45-minute mobile system design interview — how to drive itMid▸

In plain terms

An interviewer who gives you a blank-slate problem is testing whether you impose structure. The best candidates drive the 45 minutes like a structured conversation, not a monologue — state assumptions, get quick buy-in, then go deep on the interesting parts.

The six-phase skeleton

  • 1. Clarify & scope (5 min) — ask before you draw. Functional requirements, then non-functional (scale, offline, latency, battery, data usage). Say what's in/out of scope aloud.
  • 2. API contract (5 min) — endpoints, request/response shapes, pagination style, error model, auth. Anchors the rest.
  • 3. High-level design (10 min) — draw the components: client layers, server, database, CDN. Describe the main data flow. No classes yet.
  • 4. Client architecture deep-dive (15 min) — the mobile-specific part interviewers care most about. Layer structure, state model, threading, caching, offline behavior.
  • 5. LLD / deep-dive (5 min) — key classes/interfaces for the hardest part. Pick what the interviewer seems most interested in.
  • 6. Trade-offs & non-functionals (5 min) — what you'd do differently at scale, what you chose not to do and why, failure modes. Seniority shows in trade-offs.

The mobile-specific lens: web rounds focus on servers/DBs. Mobile rounds want client architecture — layers, single source of truth, threading, caching, lifecycle, memory. Answer through that lens first, even if backend comes up.

D02Offset vs cursor pagination — design and trade-offsMid▸

In plain terms

Offset says "give me items 20–40." Cursor says "give me items after this specific one." Cursor is better for live feeds because inserts/deletes shift offset-based positions and cause duplicates or skips; a cursor anchored to an actual row is immune to that.

Wire format

// Offset — breaks on live data
GET /api/posts?page=3&size=20
// If 5 posts insert before page 2 loads, page 3 now repeats page-2 items

// Cursor — stable, encodes (sortValue, id) as a composite key
GET /api/posts?after=eyJpZCI6MTA1fQ==&size=20
// Server decodes → WHERE (start_time, id) > (cursorTime, cursorId) ORDER BY start_time, id LIMIT 20

Why include the ID in the cursor, not just a timestamp

Two rows can share the same sort value (e.g. two meetings starting at 9am). A composite (sortValue, id) key gives a strictly total order so the range query never double-returns or skips a same-timestamp row. Requires a composite index on (sortKey, id) or you're back to a scan despite cursor semantics.

Client — Room + Paging 3 RemoteMediator

@Query("SELECT * FROM post ORDER BY id DESC LIMIT :limit")
fun page(limit: Int): List<Post>

@Query("SELECT * FROM post WHERE id < :cursor ORDER BY id DESC LIMIT :limit")
fun pageAfter(cursor: Long, limit: Int): List<Post>

// RemoteMediator persists the cursor in Room so pagination survives process death mid-scroll
class FeedRemoteMediator(...) : RemoteMediator<Int, Post>() {
    override suspend fun load(loadType: LoadType, state: PagingState<Int, Post>): MediatorResult {
        val cursor = when (loadType) {
            REFRESH -> null
            PREPEND -> return MediatorResult.Success(endOfPaginationReached = true)
            APPEND -> remoteKeyDao.getLastCursor()
        }
        val page = api.getPosts(cursor, state.config.pageSize)
        db.withTransaction {
            if (loadType == REFRESH) dao.clearAll()
            dao.insertAll(page.items)
            remoteKeyDao.saveCursor(page.nextCursor)
        }
        return MediatorResult.Success(endOfPaginationReached = page.nextCursor == null)
    }
}

Edge cases worth naming unprompted

  • Item deleted after being cursor-referenced: fine — the cursor is just a comparison value, not a lookup that requires the row to still exist.
  • Bidirectional scrolling (e.g. a calendar paging both forward and backward) needs a startCursor/endCursor pair per page — the Relay-style connection shape (pageInfo{ startCursor, endCursor, hasNextPage, hasPreviousPage }).
  • "Jump to page 7" UIs genuinely don't fit pure cursor pagination without a hybrid offset-estimate — say this is a known, legitimate trade-off rather than forcing a fit.
D03Design a multi-tier caching strategyHard▸

In plain terms

Three tiers, fastest to slowest: memory (lost on app kill) → disk (survives restarts) → network. Reads go top-down; writes go bottom-up. The policy per tier determines freshness vs cost.

HLD

UI → ViewModel → Repository
                    ↓
         [1] Memory LRU cache (hot objects, bounded)
                    ↓ miss
         [2] Disk / Room (structured) + DiskLRU (blobs)
                    ↓ miss or stale
         [3] Network (OkHttp + HTTP cache layer)
                    ↓
              Server → CDN

D04Offline-first with conflict resolutionHard▸

In plain terms

The local database is the source of truth; the UI never waits on the network. Writes go to an outbox table first; a background job drains it when connectivity returns.

HLD

User action → Repository
                ↓
     1. Write to local DB (immediate, optimistic)
     2. Insert into Outbox table (pending mutation)
     3. UI observes DB → instant update
                ↓ (background)
     WorkManager drains Outbox → POST to server
     Server returns canonical id/timestamp
     Repository updates local row: pending → synced

LLD — Room + WorkManager

@Entity
data class OutboxEntry(
    @PrimaryKey val id: String = UUID.randomUUID().toString(),
    val entityType: String, val payload: String,
    val status: Status = Status.PENDING, val retries: Int = 0
)

@Transaction
suspend fun createPost(post: Post) {
    postDao.insert(post.copy(syncStatus = SyncStatus.PENDING))
    outboxDao.insert(OutboxEntry(entityType = "post", payload = post.toJson()))
}

class SyncWorker(ctx: Context, params: WorkerParameters) : CoroutineWorker(ctx, params) {
    override suspend fun doWork(): Result {
        outboxDao.pendingEntries().forEach { entry ->
            try { api.sync(entry.payload); outboxDao.markSynced(entry.id) }
            catch (e: Exception) {
                outboxDao.incrementRetry(entry.id)
                if (entry.retries >= 3) outboxDao.markFailed(entry.id)
            }
        }
        return Result.success()
    }
}

Conflict resolution strategies

  • Last-write-wins: server timestamp decides. Simple, loses concurrent edits. Fine for most non-collaborative data.
  • Per-field merge: independent fields merge independently — better for settings/profile edits.
  • CRDT: commutative merge, no single winner needed. Best for collaborative editing, most complex to implement.
D05Token auth with silent refreshHard▸

In plain terms

Short-lived access tokens (15 min) + long-lived refresh tokens (30 days). A 401 triggers a transparent refresh — the original request retries with the new token, the user never sees a login page mid-session.

HLD

Request → OkHttp/Axios Interceptor
    ↓ adds "Authorization: Bearer <access>"
API call → 401?
    ↓ yes
Authenticator / Response Interceptor
    ↓ single-flight mutex (only ONE refresh at a time)
POST /auth/refresh { refreshToken }
    ↓ success
Store new access + refresh tokens → Retry original request
    ↓ fail (refresh expired)
Clear session → navigate to Login

LLD — Android OkHttp Authenticator

class TokenAuthenticator(
    private val tokenStore: TokenStore, private val authApi: AuthApi
) : Authenticator {
    private val mutex = Mutex()
    override fun authenticate(route: Route?, response: Response): Request? = runBlocking {
        mutex.withLock {
            val current = tokenStore.accessToken
            if (current != response.request.header("Authorization")?.drop(7)) {
                return@withLock response.request.newBuilder()
                    .header("Authorization", "Bearer $current").build()
            }
            try {
                val refreshed = authApi.refresh(tokenStore.refreshToken)
                tokenStore.save(refreshed)
                response.request.newBuilder()
                    .header("Authorization", "Bearer ${refreshed.accessToken}").build()
            } catch (e: Exception) { tokenStore.clear(); null }
        }
    }
}

Storage: Android Keystore (EncryptedSharedPreferences), iOS Keychain — never plain SharedPreferences or localStorage for tokens.

    ↓
[1] Memory LRU cache → hit: set bitmap → done
    ↓ miss
[2] Disk LRU cache → hit: decode → set → update memory
    ↓ miss

LLD — Android

class ImageLoader(maxMemKb: Int) {
    private val memCache = object : LruCache<String, Bitmap>(maxMemKb) {
        override fun sizeOf(k: String, v: Bitmap) = v.byteCount / 1024
    }
    private val diskCache = DiskLruCache.open(cacheDir, maxDiskKb)
    private val inFlight = ConcurrentHashMap<String, Deferred<Bitmap?>>()

    fun load(url: String, into: ImageView, size: Size) {
        val key = cacheKey(url, size)
        into.tag?.let { (it as? Job)?.cancel() }
        val job = ioScope.launch {
            val bmp = memCache[key]
                ?: diskCache.get(key)?.also { memCache.put(key, it) }
                ?: fetchAndDecode(url, size, key)
            withContext(Main) {
                if (into.tag == coroutineContext[Job]) into.setImageBitmap(bmp)
            }
        }
        into.tag = job
    }
}

Trade-offs: aggressive memory caching prevents redundant decode but risks evicting large bitmaps unexpectedly — bound to 1/8 of maxMemory. Disk dedupe saves bandwidth; the dedup map is bound by size + TTL. Lifecycle tie-in (cancel on view recycle) is critical for scroll performance.

D07Design real-time chatHard▸

Clarify

Clarifying questions — 1:1 or group? Max group size? Read receipts? Media? Offline send? History pagination depth?

HLD

Client ←→ WebSocket Gateway ←→ Message Service ←→ Kafka
                                       ↓
                                 Message DB (Cassandra)
                                       ↓
                                 Push (FCM/APNs) when recipient offline

Message flow:
1. User sends → write locally (status=SENDING, clientId=uuid)
2. POST /messages or WS send
3. Server returns {id, serverTimestamp} → update row (status=SENT)
4. Server pushes to recipient via WS → recipient stores
5. Recipient ACKs → sender updates status=DELIVERED
6. Recipient opens conversation → sends SEEN → status=READ

LLD — Android

@Entity
data class Message(
    @PrimaryKey val id: String,
    val clientId: String,        // idempotency key for dedup
    val conversationId: String,
    val body: String,
    val status: MessageStatus,   // SENDING, SENT, DELIVERED, READ, FAILED
    val serverTimestamp: Long?,
    val clientTimestamp: Long
)

class ChatRepository(private val dao: MessageDao, private val ws: WebSocketClient, private val api: ChatApi) {
    fun messages(convId: String) = dao.observeMessages(convId)

    suspend fun send(convId: String, body: String) {
        val clientId = UUID.randomUUID().toString()
        dao.insert(Message(id = clientId, clientId = clientId, status = SENDING, ...))
        try {
            val confirmed = api.sendMessage(convId, body, clientId)
            dao.update(clientId, serverId = confirmed.id, status = SENT)
        } catch (e: Exception) { dao.update(clientId, status = FAILED) }
    }
}

Ordering gotcha: never use clientTimestamp for display order in a multi-sender conversation — clocks drift. Use server timestamp as primary sort key, falling back to client timestamp only for the sender's own optimistic message before confirmation.

@Entity
data class UploadJob(
    @PrimaryKey val id: String = UUID.randomUUID().toString(),
    val localPath: String, val uploadId: String? = null,
    val completedChunks: List<Int> = emptyList(),
    val status: UploadStatus = UploadStatus.QUEUED,
    val idempotencyKey: String = UUID.randomUUID().toString()
)

class UploadWorker(ctx: Context, params: WorkerParameters) : CoroutineWorker(ctx, params) {
    override suspend fun doWork(): Result {
        val job = uploadDao.get(inputData.getString("jobId")!!)
        val uploadId = job.uploadId ?: api.initiate(job.idempotencyKey).also { uploadDao.update(job.copy(uploadId = it)) }
        chunks(job.localPath).filter { it.index !in job.completedChunks }.forEach { chunk ->
            api.uploadChunk(uploadId, chunk)
            uploadDao.markChunkDone(job.id, chunk.index)   // persist immediately
        }
        api.complete(uploadId); uploadDao.update(job.copy(status = UploadStatus.DONE))
        return Result.success()
    }
}

D12Design an app-wide overlay / intervention frameworkHard▸

The problem

20+ features each want to show overlays (popups, tooltips, snackbars). Without a central arbiter they stack, collide, and fire at bad moments. One system decides what shows, when, and in what order.

HLD

Feature modules → InterventionManager.enqueue(Intervention)
                        ↓
              Priority Queue (ordered by priority + time)
                        ↓
              Arbitration: max one at a time, allowed screens/states only,
              per-intervention frequency caps, global pacing, remote-config gate
                        ↓
              InterventionHost renders current from StateFlow
                        ↓
              User dismisses → notify queue → dequeue next

LLD — Kotlin Coroutines Actor

class InterventionManager(private val remoteConfig: RemoteConfig, private val dao: InterventionDao) {
    private val _current = MutableStateFlow<Intervention?>(null)
    val current = _current.asStateFlow()
    private val actor = CoroutineScope(Dispatchers.Default).actor<Cmd> {
        val queue = PriorityQueue<Intervention>(compareByDescending { it.priority })
        for (cmd in channel) {
            when (cmd) {
                is Cmd.Enqueue -> if (isEligible(cmd.item)) queue.add(cmd.item)
                is Cmd.Dismiss -> { dao.record(cmd.id); _current.value = null }
            }
            if (_current.value == null) _current.value = queue.poll()
        }
    }
    fun enqueue(item: Intervention) { actor.trySend(Cmd.Enqueue(item)) }
}

Company-wide adoption pattern: each feature registers its interventions via @IntoSet multibinding — the manager collects them without knowing any individual feature. Features don't know each other exists; the manager arbitrates centrally.

LLD — request de-duplication

class RequestDeduper {
    private val inFlight = ConcurrentHashMap<String, Deferred<Response>>()
    suspend fun <T> dedupe(key: String, block: suspend () -> T): T {
        val deferred = inFlight.getOrPut(key) { scope.async { block() } }
        return deferred.await().also { inFlight.remove(key, deferred) }
    }
}

LLD — single-flight token refresh (the part most candidates fumble)

class TokenAuthenticator(private val tokenRepo: TokenRepository) : Authenticator {
    private val refreshMutex = Mutex()
    override fun authenticate(route: Route?, response: Response): Request? = runBlocking {
        refreshMutex.withLock {
            val current = tokenRepo.getAccessToken()
            // Someone else already refreshed while we waited for the lock — reuse it, don't refresh again
            if (current != extractTokenFromRequest(response.request)) {
                return@withLock response.request.newBuilder().header("Authorization", "Bearer $current").build()
            }
            val newToken = tokenRepo.refreshToken() ?: return@withLock null   // refresh failed → logout
            response.request.newBuilder().header("Authorization", "Bearer $newToken").build()
        }
    }
}

Exponential backoff — the formula and why jitter matters

delay = min(baseDelay * 2^attempt, maxDelay) + jitter

fun backoffDelay(attempt: Int, base: Long = 1000L, max: Long = 30_000L): Long {
    val exp = min(base * (1L shl attempt), max)
    return exp + Random.nextLong(0, exp / 2)   // "equal jitter" — guarantees a minimum wait while spreading load
}

Without jitter, thousands of clients that all fail at once (server recovering from an outage) retry in perfectly synchronized waves — jitter breaks that synchronization so the server actually gets a chance to recover.

What to retry vs not — the distinction that separates senior answers

  • Retry: timeouts, 5xx, connection resets on idempotent methods (GET, PUT, DELETE) only.
  • Never blindly retry: POST, unless the endpoint has an idempotency key — you'd risk double-charging a payment or double-creating a resource.
  • Don't retry 4xx — the problem is the request itself, not a transient condition. 401 is special: it triggers refresh + one retry, not backoff-style repeated retrying.
  • Respect Retry-After if the server sends one — it overrides your own computed backoff.

Cancellation

Tie the call to viewModelScope (survives rotation, cancels on clear) or lifecycleScope depending on whether data should survive config change. Cancelling the coroutine should call okHttpCall.cancel() — Retrofit's suspend support does this automatically via suspendCancellableCoroutine + invokeOnCancellation.

D20Design a file upload/download manager — pause/resume, background continuationHard▸

Clarify

        ↓

        ↓

        ↓
Background worker — Foreground service / WorkManager

Resume bypass: pause reads back the last saved byte offset instead of a separate code path

Why one field, bytesTransferred, is the entire design

Collapsing "displayed progress" and "actual resume point" into a single durable field removes an entire class of bugs where the two drift out of sync. Resume IS start — there's no separate resume function; start() always reads wherever bytesTransferred currently is.

LLD — chunked download with correct cancellation ordering

fun download(entity: TransferEntity): Flow<TransferProgress> = flow {
    var offset = entity.bytesTransferred   // resume point IS the start point — no separate resume path
    try {
        while (offset < entity.totalBytes) {
            currentCoroutineContext().ensureActive()   // cooperative cancellation checkpoint
            val end = minOf(offset + chunkSize - 1, entity.totalBytes - 1)
            val request = Request.Builder().url(entity.remoteUrl)
                .header("Range", "bytes=$offset-$end").build()
            // ...stream response body, write to file, advance offset...
            if (offset - lastPersisted >= 256 * 1024) { dao.updateProgress(entity.transferId, offset) }
        }
        dao.updateStatus(entity.transferId, Status.COMPLETE)
    } catch (e: CancellationException) {
        dao.updateProgress(entity.transferId, offset)   // persist BEFORE propagating — the line that makes pause correct
        dao.updateStatus(entity.transferId, Status.PAUSED)
        throw e
    }
}

Background continuation

Under ~10 min: WorkManager with setExpedited() suffices. For long-running/user-visible transfers, use a genuine foreground service with ForegroundServiceType.dataSync (required on Android 14+) — WorkManager jobs can still be deferred under battery restrictions; a foreground service is what actually survives Doze.

Optimizations

  • Parallel chunk transfer — fire multiple byte-range requests concurrently when the server supports it; big win on high-latency links, at the cost of progress becoming a sum over N in-flight offsets.
  • Adaptive chunk sizing — measure first-chunk throughput, size later chunks proportionally instead of a fixed 5MB.
  • Incremental checksum — feed bytes into a running MessageDigest as chunks stream, turning O(file size) verification into O(1) marginal cost.

LLD — debounced autosave with the one line that actually matters

class DraftAutosaveController(private val draftDao: DraftDao, private val screenKey: String, scope: CoroutineScope) {
    private val textFlow = MutableStateFlow("")
    init {
        textFlow.debounce(500).onEach { persist(it) }.launchIn(scope)
    }
    fun onTextChanged(text: String) { textFlow.value = text }

    // Called from onStop() — bypasses debounce; up to 500ms of typing could be unsaved otherwise
    suspend fun flushImmediately() { persist(textFlow.value) }

    private suspend fun persist(text: String) {
        if (text.isBlank()) draftDao.discard(screenKey)
        else draftDao.save(DraftEntity(screenKey, text, updatedAt = System.currentTimeMillis()))
    }
}

The debounce buffer (up to 500ms of the user's most recent typing) is lost forever if the process dies before it flushes — forcing a synchronous flush on onStop() closes that gap. Most candidates miss this.

Discard vs back-press — a distinct code path

fun onSendClicked() { sendMessage(text); draftDao.discard(key) }
fun onDiscardClicked() { draftDao.discard(key) }
fun onBackPressed() { /* do nothing — draft persists, this is NOT a discard */ }

Treating "user pressed back" the same as "user discarded" is the #1 complaint users have about compose screens that get this wrong.

Drafts are keyed by an unbounded, dynamic set of conversation IDs — a relational lookup pattern, not a fixed small set of app-wide settings. If it were a single "new post" draft with no per-conversation keying, DataStore would be the simpler, correct choice.

D23Design a background job scheduler (WorkManager internals-level)Hard▸

Clarify

Clarifying questions — Periodic sync, one-off deferred work, or both? Do jobs have dependencies on each other? Minimum OS version? Battery-sensitive app?

Why naive approaches fail — the actual thing being tested

A repeating Handler.postDelayed or AlarmManager.setRepeating fights the OS's Doze/App Standby battery model instead of cooperating with it. WorkManager's entire design is built around cooperating with that state machine — it delegates to the real JobScheduler on API 23+, which batches your job's wake-up with other apps' jobs of similar constraints.

HLD

        ↓ enqueue() — written to WorkManager's own internal Room DB immediately (survives process death)
        ↓
Internal Scheduler → delegates to OS JobScheduler
        ↓ OS decides WHEN, batched with other apps' jobs
Constraints re-checked at dispatch time (state may have changed since scheduling)
        ↓
doWork() executes → result updates persisted state → retry re-enqueues per backoff policy

LLD — unique periodic work, and the OS-enforced floor

val syncRequest = PeriodicWorkRequestBuilder<SyncWorker>(15, TimeUnit.MINUTES)   // 15 min is the OS-enforced MINIMUM, not arbitrary
    .setConstraints(Constraints.Builder().setRequiredNetworkType(NetworkType.CONNECTED).setRequiresBatteryNotLow(true).build())
    .setBackoffCriteria(BackoffPolicy.EXPONENTIAL, 10, TimeUnit.SECONDS)
    .build()

workManager.enqueueUniquePeriodicWork("periodic_sync", ExistingPeriodicWorkPolicy.KEEP, syncRequest)
// KEEP: don't reset the schedule if already enqueued — calling enqueue() on every app launch without
// this dedup would create a NEW periodic job every launch

Chained/dependent work

workManager.beginWith(refreshTokenRequest).then(syncDataRequest).then(uploadPendingChangesRequest).enqueue()
// If refreshTokenRequest fails permanently, the whole chain is marked FAILED — dependent steps never attempt

Optimizations

  • Expedited work (setExpedited()) for near-real-time needs the 15-min floor can't satisfy.
  • Constraint tuning per job criticality — over-constraining low-priority sync (e.g. requiresCharging) can starve it indefinitely on a phone that's rarely charging.
  • Batch multiple logical tasks into fewer WorkRequests — each enqueued job is a separate wake-up negotiation; fewer, larger batched wake-ups is what the OS's Doze model rewards.
D24Handle multiple concurrent API calls with dependencies (A before B)Hard▸

Clarify

Clarifying questions — Simple linear chain or a true DAG (multiple nodes depend on the same parent)? Fail-fast per branch, or do independent branches continue? Screen-load scenario or durable background orchestration? Results aggregated at the end?

The core idea

Don't hand-wire nested callbacks or a long sequential await chain — model dependencies as data (a graph), and let a generic executor run anything whose dependencies are satisfied, in parallel where possible. Nested callbacks either create callback hell or accidentally serialize independent work that could run concurrently.

LLD — the executor

class TaskNode<T>(val id: String, val dependsOn: List<String> = emptyList(),
                   val action: suspend (results: Map<String, Any?>) -> T)

class DagExecutor(private val scope: CoroutineScope) {
    suspend fun execute(nodes: List<TaskNode<*>>): Map<String, NodeState> {
        val stateMap = ConcurrentHashMap<String, NodeState>()
        val resultsMap = ConcurrentHashMap<String, Any?>()
        nodes.forEach { stateMap[it.id] = NodeState.Pending }

        fun isReady(n: TaskNode<*>) = n.dependsOn.all { stateMap[it] is NodeState.Success }
        fun isBlocked(n: TaskNode<*>) = n.dependsOn.any { stateMap[it] is NodeState.Failed || stateMap[it] is NodeState.Skipped }

        return coroutineScope {
            val launched = mutableSetOf<String>()
            suspend fun tryLaunchReady() {
                nodes.filter { it.id !in launched }.forEach { node ->
                    when {
                        isBlocked(node) -> { stateMap[node.id] = NodeState.Skipped; launched += node.id }
                        isReady(node) -> {
                            launched += node.id; stateMap[node.id] = NodeState.Running
                            launch {
                                try {
                                    val result = node.action(resultsMap)
                                    resultsMap[node.id] = result; stateMap[node.id] = NodeState.Success(result)
                                } catch (e: Exception) { stateMap[node.id] = NodeState.Failed(e) }
                                tryLaunchReady()   // completion may unlock others — re-check the graph
                            }
                        }
                    }
                }
            }
            tryLaunchReady()
        }.let { stateMap }
    }
}

Cancellation — falls out of structured concurrency for free

Because everything runs inside one coroutineScope { }, cancelling the parent (e.g. viewModelScope on onCleared()) automatically cancels every in-flight node and prevents any not-yet-started node from launching — no manual bookkeeping needed.

Optimizations

  • Reverse-adjacency map instead of full graph rescan on each completion — matters once graphs grow beyond a handful of nodes.
  • Per-branch timeout via withTimeout() so one slow branch doesn't hold up the whole graph; combine with partial-result rendering.
  • Expose progressive state as a Flow<Map<String, NodeState>> so the UI can render nodes as they complete rather than blocking on the whole DAG.

    val minSessionCount: Int = 0,

class RateUsEligibility(private val dao: InterventionDao, private val userStats: UserStatsProvider) {

LLD — pool assignment on scroll

class ExoPlayerPool(context: Context, poolSize: Int = 3) {
    private val players = List(poolSize) { ExoPlayer.Builder(context).build() }
    private val assignments = mutableMapOf<String, ExoPlayer>()   // itemId -> player

    fun attach(itemId: String, mediaUrl: String, surface: PlayerView) {
        val player = assignments[itemId] ?: players.firstOrNull { it !in assignments.values }
            ?: reclaimLeastRecentlyUsed()
        player.setMediaItem(MediaItem.fromUri(mediaUrl))
        player.prepare(); player.playWhenReady = true
        surface.player = player
        assignments[itemId] = player
    }

    fun release(itemId: String) {
        assignments[itemId]?.let { it.stop(); it.clearMediaItems() }
        assignments.remove(itemId)
    }
}

Lifecycle safety

Tie player.pause() to onStop() and full release to onDestroy() — never let a player instance keep decoding off-screen. Combine with the same visibility-threshold logic used for image preloading in D08's feed.

Optimizations

  • Preload the next item's first few seconds while the current one plays, so autoplay feels instant on scroll.
  • Adaptive bitrate (DASH/HLS) via DefaultTrackSelector constraints tuned to current network type.
  • Surface reuse via SurfaceView pooling at the Compose/View layer to avoid surface creation/teardown cost on every scroll.

D30Handle deep links safely across cold / warm / foreground startMid▸

Clarify

Clarifying questions — Universal links (verified domain) or custom scheme? Does the target screen need auth first? Should a deep link received while already on that screen refresh it or no-op?

HLD — the three entry states need three different handling paths

Cold start (app not running):
  Intent → Application.onCreate() → nav graph not yet built
  → store pending deep link → consume AFTER nav graph + auth state are ready

Warm start (app in background, process alive):
  Intent delivered to existing Activity via onNewIntent()
  → nav graph already exists → navigate immediately, but check current back stack first

Already in foreground (user actively on some screen):
  Same onNewIntent() path → must decide: interrupt current flow, or queue until safe
  (e.g. don't yank the user out of an in-progress checkout for a promo deep link)

LLD — a single resolver that all three paths funnel through

class DeepLinkResolver(private val navController: NavController, private val authState: StateFlow<AuthState>) {
    private var pendingLink: Uri? = null

    fun handle(uri: Uri, isColdStart: Boolean) {
        if (isColdStart && authState.value !is AuthState.Ready) {
            pendingLink = uri; return   // defer until nav graph + auth are both ready
        }
        if (requiresAuth(uri) && authState.value !is AuthState.LoggedIn) {
            navController.navigate("login?redirect=${Uri.encode(uri.toString())}")
            return
        }
        navController.navigate(resolveRoute(uri))
    }

    fun onAuthReady() { pendingLink?.let { handle(it, isColdStart = false) }; pendingLink = null }
}

The failure mode worth naming

Handling the deep link directly in onCreate() without checking whether the nav graph and auth state are actually ready is the most common bug — it works fine in testing (fast emulator, warm cache) and then crashes or silently no-ops for real users on a cold start with a slow network, because the target screen's dependencies genuinely aren't ready yet.

suspend fun expand(master: EventMasterEntity, rangeStart: Long, rangeEnd: Long): List<EventInstance> {
    if (master.rrule == null) return if (master.startTime in rangeStart..rangeEnd) listOf(EventInstance.from(master)) else emptyList()

        when (val ex = exceptions[t]) {
            null -> EventInstance.from(master, instanceTime = t)
            else -> if (ex.isDeleted) null else EventInstance.from(master, instanceTime = ex.newStartTime!!, title = ex.newTitle)
        }
    }
}

"This and following" edits

Truncate the original master's recurrenceEndTime to just before the edited instance, and create a new master starting from that instance forward. Splits the series rather than requiring a more complex partial-RRULE override mechanism.

Fetching data — API request format at scale

// Cold sync — bounded range, sparse fields, server-side expansion if requested
GET /calendars/{id}/events?timeMin=2026-03-01T00:00:00Z&timeMax=2026-04-15T00:00:00Z
    &maxResults=250&singleEvents=true&fields=items(id,title,start,end,recurringEventId),nextPageToken

// Every subsequent sync — delta only, via opaque syncToken
GET /calendars/{id}/events?syncToken=CigreiLA1234...&maxResults=250
// Deleted event → tombstone: { "id": "evt_456", "status": "cancelled" } — client must REMOVE the local row
// syncToken expired (410 Gone) → discard token, fall back to a fresh range-based full sync

UI fetch strategy — local-first, network never blocks render

Month view always queries and expands local Room data first (instant, works offline). A separate Fetched-Range Tracker table decides whether to fire a background network fetch — bucketed by calendar month, with a staleness TTL — so swiping back to an already-viewed month never re-hits the network.

Why Room over DataStore for drafts/exceptions here

Same reasoning as D22 and D21 — exceptions are keyed by an unbounded set of occurrences, a relational lookup pattern, not fixed app-wide settings.

D34Design Google Docs — collaborative editing on mobileHard▸

Clarify

Clarifying questions — Plain text or rich formatting? Offline editing that merges cleanly on reconnect? Live cursor/presence of other editors required? Comment threads in scope?

Why this is a fundamentally different problem from D04's offline sync

D04's last-write-wins is fine for a notes app where concurrent edits are rare and losing one is tolerable. A collaborative doc has frequent, fine-grained concurrent edits to the same content — LWW would constantly clobber other people's keystrokes. This needs a data structure that merges automatically, commutatively, regardless of the order edits arrive in.

HLD — CRDT over OT for a mobile client

Local edit → CRDT document (Yjs-style) mutated immediately — UI always responsive, no waiting on network
        ↓
IndexedDB/Room-backed persistence provider — offline edits accumulate in the CRDT structure locally
        ↓
WebSocket/relay provider — broadcasts local ops to other connected clients when online
        ↓
Remote ops arrive → merged into the local CRDT automatically — no server-side transform step required
        ↓
Editor view (Compose/TipTap-equivalent) re-renders from the merged CRDT state

Why CRDT, not Operational Transform: OT requires a central server that's always available to sequence and transform concurrent ops — hard to reconcile with "edit fully offline for hours, then reconnect." CRDTs merge any two states in any order, so offline-then-sync just works as a natural consequence of the data structure, not a special case.

LLD — the client-side shape (conceptual, Android/Kotlin analog of Yjs)

class CollabDocument(private val docId: String, private val persistDao: DocOpsDao) {
    private val crdt = YDocLike()   // a CRDT text structure — each character/insertion carries a stable, comparable position

    fun applyLocalEdit(op: TextOp) {
        crdt.apply(op)
        persistDao.appendOp(docId, op.serialize())   // durable BEFORE broadcasting — same discipline as D04's outbox
        connectionManager.broadcastIfOnline(op)
    }

    fun applyRemoteOp(op: TextOp) {
        crdt.apply(op)   // CRDT merge is commutative — order of arrival doesn't matter for final consistency
        persistDao.appendOp(docId, op.serialize())
    }
}

Trade-offs to state explicitly

  • Memory cost: a CRDT document carries enough history/metadata to merge correctly — heavier than a plain string, worth compacting periodically (garbage-collect tombstoned deletions once all peers have acknowledged them).
  • Expressiveness limit: CRDTs handle character-level merge well but struggle to express some complex intents ("don't insert inside a range someone just deleted") — real products layer additional semantic rules on top for rich-text edge cases.
  • Presence/cursors are NOT part of the CRDT — they're ephemeral, broadcast-only state (who's where right now) that doesn't need to persist or merge; keep them on a separate lightweight channel, same principle as D07 chat separating presence from message delivery.

Reusing earlier patterns

Comment threads on a doc are exactly D07's nested/threaded data shape; document sharing/permissions is exactly D32's invite+ACL model; the local-first "always responsive, sync in background" feel is exactly D04's principle, just with a different merge strategy at the core.

function useAutocomplete(minLength = 2) {
    const [suggestions, setSuggestions] = useState<string[]>([]);
    const cacheRef = useRef(new Map<string, string[]>());
    const abortRef = useRef<AbortController>();

    const search = useMemo(() => debounce(async (q: string) => {
        if (q.length < minLength) return setSuggestions([]);
        if (cacheRef.current.has(q)) return setSuggestions(cacheRef.current.get(q)!);
        abortRef.current?.abort(); abortRef.current = new AbortController();
        try {
            const res = await fetch(`/suggest?q=${q}`, { signal: abortRef.current.signal });
            const data = await res.json();
            cacheRef.current.set(q, data.suggestions); setSuggestions(data.suggestions);
        } catch (e) { if ((e as Error).name !== 'AbortError') throw e; }
    }, 200), [minLength]);

    return { suggestions, onChange: search };
}

async function slidingWindowLimit(userId: string, limit = 100, windowMs = 60000) {
    const now = Date.now(); const windowStart = now - windowMs; const key = `rl:${userId}`;
    const [, , count] = await redis.pipeline()
        .zremrangebyscore(key, '-inf', windowStart)
        .zadd(key, now, `${now}-${Math.random()}`)
        .zcard(key).expire(key, windowMs / 1000).exec();
    if (count > limit) {
        const oldest = await redis.zrange(key, 0, 0, 'WITHSCORES');
        throw { status: 429, retryAfter: Math.ceil((+oldest[1] + windowMs - now) / 1000) };
    }
}
