01 Android extras

Framework breadth — services, IPC, security, deep linking, OS-level constraints.

E01Service types: started, bound, foreground.Low

A started service runs until it stops itself (or is killed); used for fire-and-forget background work — mostly replaced by WorkManager now. A bound service exposes an IPC interface (Binder/Messenger/AIDL) and lives while clients are bound. A foreground service shows a persistent notification and is the only way to do user-visible, must-run-now background work (media, navigation, calls) without being killed; on Android 14+ you must declare a foreground service type.

E02BroadcastReceiver: manifest vs runtime, ordered broadcasts.Low

Receivers handle system or app broadcasts. Manifest-declared receivers can be woken even when the app isn't running (limited since Android 8). Runtime-registered receivers live with their Context — typical for in-app events. onReceive runs on the main thread and must finish <10s. Ordered broadcasts deliver sequentially by priority and let receivers abort or modify the payload — used to be common for SMS interception, now rare. Prefer LocalBroadcastManager alternatives like SharedFlow or LiveData for in-app fan-out.

E03ContentProvider: when (and when not) to use one.Low

The official cross-process data sharing surface — exposes data by URI with permission control, and what backs Contacts/MediaStore/Calendar. Internally it implements query/insert/update/delete over a Cursor, with a UriMatcher routing URIs to handlers. Today you only build one when you genuinely need to share data with other apps; for in-app DB work, Room is the right tool. Most "do I need a ContentProvider?" interview answers are: no — unless you're publishing data to other processes.

E04Binder & AIDL — how Android IPC actually works.Mid

Binder is the kernel driver that backs nearly all Android IPC: service calls, startActivity, content providers — everything ultimately marshals through Binder transactions. AIDL is a DSL that generates a Binder stub/proxy pair for you: clients call methods that look local; the framework serializes args via Parcel, hops processes, and dispatches on a worker thread on the server side. Use Messenger (built on AIDL) for simple message passing, AIDL when you need a typed multi-method interface. Watch for: 1MB transaction size limit, main-thread blocking on bound calls, oneway methods for fire-and-forget.

E05Intent flags & launch modes you actually use.Low

Most "back stack misbehaves" bugs come down to these. FLAG_ACTIVITY_NEW_TASK + CLEAR_TASK is the classic "reset to home after login" combo. SINGLE_TOP reuses the top activity if it matches; CLEAR_TOP pops everything above it. Launch modes (standard, singleTop, singleTask, singleInstance) are the manifest-level version. With NavController you mostly let Navigation handle it via popUpTo/inclusive, but you'll still be asked when the launch-mode names appear in interviews.

E06Deep links vs App Links — and why verification matters.Mid

A deep link is any intent-filter that handles a URI; the OS shows a disambiguation chooser. Android App Links are HTTPS deep links the OS has verified via /.well-known/assetlinks.json on your domain — they open directly in your app with no chooser, which is critical for marketing/email flows. iOS equivalent is Universal Links. Test with adb shell am start -a android.intent.action.VIEW -d "https://…" and the App Links assistant. For React Native, configure both platforms and use Linking/React Navigation.

E07Doze, App Standby buckets & background restrictions.Mid

Why your "every minute" sync isn't every minute on real devices. Doze kicks in when the device is idle and unplugged, batching network/jobs into infrequent maintenance windows. App Standby Buckets (Active → Working → Frequent → Rare → Restricted) limit job/alarm frequency based on user engagement. You don't fight these; you cooperate: WorkManager with constraints, foreground services for must-run-now work, high-priority FCM for time-critical wake-ups. Test on real devices — emulators don't enforce realistically.

E08Network Security Config & certificate pinning.Mid

An XML resource that declares your TLS posture without code changes: opt out of cleartext, restrict trusted CAs per domain, pin certificates (<pin-set>) so a compromised CA can't MITM you, and define a debug-overrides for local proxies. Pinning is great defence-in-depth but a hard pin can brick an app if certs rotate — pair pins with a server-driven kill switch or short-lived pin sets. OkHttp also supports programmatic pinning, useful when you need rotation logic.

E09BiometricPrompt, Keystore & CryptoObject.Low

Use BiometricPrompt for fingerprint/face auth. To gate sensitive crypto on biometric success, generate a key in the Android Keystore with setUserAuthenticationRequired(true) and pass a CryptoObject wrapping a Cipher/Signature: the OS only unlocks the key when the user authenticates. Key attestation lets your server verify a key originated on real hardware (StrongBox/TEE), defeating emulator/keystore swap attacks. Almost never the headline question, often a follow-up.

E10Scoped storage & the Storage Access Framework.Low

Since Android 10, apps no longer have free-roam access to external storage. You use MediaStore for media in shared collections, the app-private dirs for everything internal, and the SAF (ACTION_OPEN_DOCUMENT/ACTION_CREATE_DOCUMENT) when the user picks a file or folder — the OS hands you a persistent URI you can re-open later via ContentResolver. WRITE_EXTERNAL_STORAGE is effectively dead. Photo Picker on 13+ avoids permissions entirely for image/video picking.

E11Notification channels & user controls.Low

Since Android 8, every notification belongs to a channel; importance, sound, vibration are user-controlled per channel. Design channels by user intent ("New messages", "Promotions"), not feature names, so users can mute one without losing others. Android 13+ requires runtime POST_NOTIFICATIONS permission. For lots of notifications use group + summary; for high-signal chat consider Bubbles.

E12Macrobenchmark, Perfetto & startup tracing.Mid

Macrobenchmark is the Jetpack library that runs your real release APK through scripted journeys and reports cold/warm/hot startup, frame timing and journey duration — feeds Baseline Profile generation. Perfetto is the system tracer: cross-process timeline, binder calls, choreographer frames, CPU scheduling. The combo is how you turn "startup feels slow" into a number with a cause. Pair with Android Vitals (Play Console) for field metrics.

E13When R8 breaks things — common keep-rule patterns.Low

R8 can't see code reached through reflection or string-based lookups. Symptoms: NoSuchFieldException, fields rename to a/b, models deserialize as nulls. Standard fixes: keep model classes (-keep class com.foo.model.** { *; }), keep JNI methods, keep classes named in XML, rely on library consumer rules (Gson, Moshi, Retrofit ship their own). Always smoke-test release builds, not just debug.

E14Play Asset / Feature Delivery & DFM.Low

Dynamic Feature Modules ship as part of the AAB but install on-demand (or conditionally by device/locale), shrinking the base APK. Useful for big rarely-used surfaces — at ShareChat we DFM'd FFMPEG and Agora for 15MB+ savings. Asset Delivery does the same for large media (games). Trade-off: install latency on first use; you have to handle install state, download progress, and fall-back UI. Test edge cases like immediate uninstall and missing modules at runtime.

02 JS & React extras

The async/concurrency catalogue and React features that get follow-up'd.

E15Promise.any, race, allSettled — when each?Mid

Beyond Promise.all: any resolves with the first fulfilled value and only rejects if all fail (an AggregateError) — perfect for "fastest mirror wins". race settles with the first to either resolve or reject — used for timeouts (race against a delayed reject). allSettled waits for every promise to settle and returns {status, value/reason} tuples — use when partial success is OK (analytics flush, multi-endpoint dashboard).

// timeout pattern
const withTimeout = (p, ms) =>
    Promise.race([p, new Promise((_, r) => setTimeout(() => r(new Error('timeout')), ms))]);
E16Implement an async pool (concurrency limit).Mid

"Run N tasks at most K at a time" — a classic follow-up to Promise.all when you can't fire 1000 requests at once.

async function pool(items, limit, fn) {
    const out = []; const active = new Set();
    for (const [i, item] of items.entries()) {
        const p = Promise.resolve(fn(item, i)).then(v => { out[i] = v; active.delete(p); });
        active.add(p);
        if (active.size >= limit) await Promise.race(active);
    }
    await Promise.all(active);
    return out;
}

Key idea: track active promises in a Set, await race when full so one slot frees up. Write results by index to preserve order.

E17Retry with exponential backoff & jitter.Mid 
async function retry(fn, { tries = 4, base = 300, cap = 5000 } = {}) {
    let err;
    for (let i = 0; i < tries; i++) {
        try { return await fn(); }
        catch (e) {
            err = e;
            const delay = Math.min(cap, base * 2 ** i);
            const jitter = Math.random() * delay;
            await new Promise(r => setTimeout(r, jitter));
        }
    }
    throw err;
}

The jitter matters: without it every retrying client hits the server in lockstep ("thundering herd"). Pair with AbortController so a navigating-away user cancels in-flight retries.

E18AbortController — cancellable fetch.Low

JS doesn't have first-class cancellation; AbortController is the universal opt-in. Create one, pass controller.signal to fetch / event listeners / streams, call abort() to cancel. Reads on signal.aborted let you bail out mid-loop. In React, abort the in-flight request in useEffect cleanup so a fast-typing user doesn't render stale results.

E19Sketch a tiny virtual-DOM diff.Mid

Compare two node trees and produce DOM mutations. Two trees, same position: if types differ → replace; if same → patch attributes, then recurse over children.

function diff(prev, next, dom) {
    if (!next) return dom.remove();
    if (!prev || prev.type !== next.type) return dom.replaceWith(render(next));
    if (typeof next === 'string') if (dom.nodeValue !== next) dom.nodeValue = next;
    else {
        patchProps(dom, prev.props, next.props);
        const a = prev.children || [], b = next.children || [];
        for (let i = 0; i < Math.max(a.length, b.length); i++)
            diff(a[i], b[i], dom.childNodes[i]);
    }
}

This is the toy version — React's reconciler additionally uses keys to match list children by identity (not index) and Fiber for time-slicing.

E20useTransition & useDeferredValue.Mid

Concurrent React lets you mark updates as non-urgent so React keeps the input responsive while heavy work renders in the background. useTransition wraps a state update: keystrokes stay snappy while a big list re-renders. useDeferredValue hands you a lagged copy of a value to feed into expensive children. Trade-off: the deferred UI is a frame or two behind reality — show a subtle "pending" indicator so users notice.

E21React Compiler (auto-memoization).Low

The compiler that statically analyses components and emits the equivalent of useMemo/useCallback/React.memo automatically — meaning much of what you hand-roll for performance becomes a build artefact. You still need to write correct, side-effect-free render code; the compiler can opt out of components it can't prove safe. Worth a sentence in interviews to signal you're tracking React's direction.

E22React Server Components — what changes.Low

Components run on the server, render to a special wire format, stream to the client where they merge with client components. Result: data fetching co-located with components, smaller JS bundles, async/await directly in components. Only relevant for web (Next.js etc.) — React Native still uses traditional client rendering. Talk to it as a web architectural shift, not a mobile concern.

E23React Query / SWR fetcher patterns.Mid

"Server state" libraries solve the same problem the Android repository pattern solves: cache, dedupe, refetch on focus/network, optimistic updates, mutation invalidation. Mental model: a query key identifies remote data; the library owns its lifecycle. Maps cleanly to the offline-first repo design from the system-design page — same single-source-of-truth idea, different syntax.

E24Web Workers (and the RN equivalents).Low

Workers run JS off the main thread; you communicate via postMessage. Use for parsing, crypto, image processing — anything that would block the UI. On React Native there's no Worker; the equivalents are Reanimated worklets (UI thread) and native modules (heavy work in Kotlin/Swift). Same goal, different mechanics: keep the JS thread free.

E25IndexedDB & Service Workers (web offline).Low

IndexedDB is the browser's asynchronous, indexed, large key-value/object store — the persistent layer for offline-first web. Service Workers are background scripts that intercept fetches and serve cached responses, enabling true offline UX and Progressive Web Apps. The combo is the web analogue of Room + WorkManager. Wrap IndexedDB with idb-keyval / Dexie; raw API is verbose.

03 System design awareness

Two concepts that show up as follow-ups to the main system-design designs.

E26CDN cache invalidation — the hard part.Mid

"There are only two hard problems in CS…" applies hard here. Strategies, in order of preference: (1) versioned URLs — new content gets a new URL, caches are immutable; (2) short TTLs + stale-while-revalidate — accept a small staleness window for write-through caching; (3) explicit purge — call the CDN API to invalidate paths/tags, slow and rate-limited but exact. Most production systems combine #1 for assets and #2 for API responses. Mention Cache-Control: s-maxage for CDN-only TTLs.

E27Circuit breakers & graceful degradation.Mid

A circuit breaker wraps a dependency call: after N failures it "opens" and fails fast for a cooldown window, sparing the dependency from a stampede and your callers from waiting. Half-open state probes recovery. On mobile, the equivalent is "if the search API is down, fall back to local results and show a banner" — gracefully degrading instead of spinning. Pairs well with retry/jitter; without a breaker, retries amplify outages.
