01 JavaScript core

The engine-level questions every JS/RN interview opens with.

Q01Explain the event loop, micro- vs macro-tasks.Hard

JS is single-threaded. The call stack runs synchronous code; async callbacks wait in queues. After each stack frame empties, the loop drains all microtasks (Promise .then, queueMicrotask, MutationObserver) before taking one macrotask (setTimeout, I/O, UI events), then repeats.

console.log('A');
setTimeout(() => console.log('B'), 0);
Promise.resolve().then(() => console.log('C'));
console.log('D');
// A D C B  — microtask C beats macrotask B

This is why a flood of resolved promises can starve setTimeout/rendering.

Q02What is a closure? Give a real use.Mid

A closure is a function plus the lexical environment it was created in — it keeps access to outer variables after the outer function returns. Used for private state, factories, memoization and event handlers.

function counter() {
    let n = 0;            // private
    return () => ++n;
}
const c = counter();
c(); c();              // 1, 2

Classic gotcha: var in a loop closes over one shared binding; use let (per-iteration binding) so each callback captures its own value.

Q03How is this determined? Why do arrow functions differ?Mid

this is set by how a function is called, not where it's defined: a method call binds to the object before the dot; a bare call is undefined (strict) or global; new binds to the new instance; call/apply/bind set it explicitly. Arrow functions have no own this — they capture it lexically from the enclosing scope, which is why they're ideal for callbacks where you want the surrounding this (and why you can't bind them).

Q04How does prototypal inheritance work?Mid

Every object has an internal [[Prototype]] link. On a property miss, the engine walks up the prototype chain until it finds the property or hits null. class is syntactic sugar over this: methods live on Class.prototype, shared by all instances. Object.create(proto) makes an object with a chosen prototype; obj.__proto__/Object.getPrototypeOf read the link.

Q05var vs let/const: hoisting and the TDZ.Mid

var is function-scoped and hoisted, initialized to undefined. let/const are block-scoped and hoisted too but live in the Temporal Dead Zone — referencing them before declaration throws ReferenceError. const forbids rebinding (the value can still mutate). Function declarations are fully hoisted; function expressions follow their variable's rules.

Q06== vs ===, and surprising coercions.Easy

=== compares type + value with no coercion. == coerces, with thorny rules. Use === by default. Worth memorizing: null == undefined is true but null === undefined is false; NaN !== NaN (use Number.isNaN); [] == ![] is true; 0 == '' is true. The one acceptable == is x == null to catch both null and undefined.

Q07Promises vs async/await; how do errors flow?Mid

A Promise is a value that settles once (fulfilled/rejected); .then/.catch/.finally chain transformations. async/await is sugar that lets you write the chain like synchronous code — await pauses until the promise settles. Errors: a rejected promise skips .then to the nearest .catch; with await, wrap in try/catch. Run independent work in parallel with Promise.all (fail-fast) or Promise.allSettled (collect all outcomes). An unhandled rejection is a bug — always handle or propagate.

Q08Shallow vs deep copy in JS — the safe options.Easy

Spread {...obj} and Object.assign copy one level — nested objects are still shared references. For a deep copy use structuredClone(obj) (built-in, handles Dates/Maps/Sets/cycles), or write a recursive clone. JSON.parse(JSON.stringify(x)) works only for plain JSON and silently drops functions, undefined and breaks on cycles.

02 JS coding challenges

The implement-from-scratch problems (bigfrontend-style) interviewers love.

Q09Implement debounce(fn, wait).Mid

Delay invoking fn until wait ms after the last call — resets the timer on every call. Use for search-as-you-type, resize.

function debounce(fn, wait) {
    let t;
    return function (...args) {
        clearTimeout(t);
        t = setTimeout(() => fn.apply(this, args), wait);
    };
}

Preserve this and args with a regular function + apply. A common follow-up adds a leading-edge option and a .cancel().

Q10Implement throttle(fn, wait) — and how it differs from debounce.Mid

Throttle runs fn at most once per wait window (good for scroll/drag). Debounce runs once after activity stops.

function throttle(fn, wait) {
    let last = 0, t = null, lastArgs;
    return function (...args) {
        const now = Date.now();
        const remaining = wait - (now - last);
        lastArgs = args;
        if (remaining <= 0) {
            last = now; fn.apply(this, args);
        } else if (!t) {
            t = setTimeout(() => {
                last = Date.now(); t = null;
                fn.apply(this, lastArgs);
            }, remaining);
        }
    };
}
Q11Implement your own Promise.all.Hard

Resolve with an array of results in input order, or reject on the first failure.

function promiseAll(items) {
    return new Promise((resolve, reject) => {
        const out = []; let done = 0;
        if (items.length === 0) return resolve(out);
        items.forEach((p, i) => {
            Promise.resolve(p).then(
                (v) => { out[i] = v; if (++done === items.length) resolve(out); },
                reject
            );
        });
    });
}

Key details: wrap each item in Promise.resolve (inputs may be plain values), write results by index not push order, and handle the empty array.

Q12Implement flatten(arr, depth).Mid 
function flatten(arr, depth = 1) {
    return arr.reduce((acc, x) => {
        if (Array.isArray(x) && depth > 0)
            acc.push(...flatten(x, depth - 1));
        else acc.push(x);
        return acc;
    }, []);
}
flatten([1, [2, [3, [4]]]], 2);  // [1, 2, 3, [4]]

For full depth, an iterative stack version avoids deep recursion on huge arrays.

Q13Implement curry(fn).Hard

Return a function that collects args until it has as many as the original's arity, then calls it.

function curry(fn) {
    return function curried(...args) {
        if (args.length >= fn.length) return fn.apply(this, args);
        return (...rest) => curried.apply(this, [...args, ...rest]);
    };
}
const add = curry((a, b, c) => a + b + c);
add(1)(2)(3); add(1, 2)(3); add(1)(2, 3);  // all 6
Q14Implement a generic memoize(fn).Mid 
function memoize(fn, keyFn = (...a) => JSON.stringify(a)) {
    const cache = new Map();
    return function (...args) {
        const key = keyFn(...args);
        if (cache.has(key)) return cache.get(key);
        const result = fn.apply(this, args);
        cache.set(key, result);
        return result;
    };
}

Discuss the key strategy (stringify is simple but lossy for objects/order), and that an unbounded cache leaks — bound it with LRU for hot paths.

Q15Implement an EventEmitter (on / emit / off).Mid 
class EventEmitter {
    constructor() { this.map = new Map(); }
    on(ev, cb) {
        (this.map.get(ev) ?? this.map.set(ev, new Set()).get(ev)).add(cb);
        return () => this.off(ev, cb);   // unsubscribe handle
    }
    off(ev, cb) { this.map.get(ev)?.delete(cb); }
    emit(ev, ...args) { this.map.get(ev)?.forEach((cb) => cb(...args)); }
}

Using a Set dedupes listeners and makes off O(1). Returning an unsubscribe function is the ergonomic touch interviewers like.

Q16Implement deepClone handling cycles.Hard 
function deepClone(obj, seen = new WeakMap()) {
    if (obj === null || typeof obj !== 'object') return obj;
    if (seen.has(obj)) return seen.get(obj);   // cycle
    const copy = Array.isArray(obj) ? [] : {};
    seen.set(obj, copy);
    for (const k of Reflect.ownKeys(obj))
        copy[k] = deepClone(obj[k], seen);
    return copy;
}

The WeakMap of already-cloned objects breaks circular references. Mention that structuredClone is the built-in production answer.

03 TypeScript

Types that make a large RN codebase safe to refactor.

Q17When and how do you reach for generics?Mid

Generics preserve the relationship between input and output types instead of widening to any. Constrain them with extends to keep them safe.

function prop<T, K extends keyof T>(obj: T, key: K): T[K] {
    return obj[key];
}
prop({ id: 1, name: 'x' }, 'name');  // inferred string
Q18Which built-in utility types do you use most?Mid

Partial<T> (all optional, for patches), Required<T>, Pick<T,K> / Omit<T,K> (shape subsets, e.g. a DTO from a model), Record<K,V> (typed maps), Readonly<T>, and ReturnType/Parameters/Awaited for deriving types from functions. They keep one source of truth — derive variants instead of duplicating interfaces.

Q19How do you narrow a union type safely?Mid

TS narrows via typeof, instanceof, in, equality checks, and discriminated unions — a shared literal kind field the compiler keys off:

type Result =
  | { kind: 'ok'; value: number }
  | { kind: 'err'; message: string };

function show(r: Result) {
    if (r.kind === 'ok') return r.value;  // narrowed
    return r.message;
}

For custom guards, write a function returning x is Foo. This mirrors Kotlin sealed-class when exhaustiveness.

04 React

Render model, hooks and performance — shared by React and React Native.

Q20What is reconciliation and why are keys important?Mid

On a state change React builds a new element tree and diffs it against the previous one (the Fiber tree), computing the minimal set of real mutations. It compares by element type and position; matching types are updated in place, different types are torn down and rebuilt.

Keys identify list items across renders so React can match them by identity instead of index — without stable keys, inserting at the top reuses the wrong DOM/native nodes, loses input focus/animation state, and can corrupt component state. Never use array index as key for reorderable lists.

Q21Why must hooks be called unconditionally at the top level?Mid

React tracks hook state by call order, not by name — it's an index into a per-component list. Calling a hook inside a condition/loop shifts indices between renders, so state gets associated with the wrong hook. Hence the two rules: only call hooks at the top level, and only from React functions/custom hooks. The ESLint plugin enforces this.

Q22useMemo vs useCallback — and when NOT to use them.Hard

useMemo caches a computed value; useCallback(fn, deps) caches a function reference (it's useMemo(() => fn, deps)). The point is referential stability: a stable prop/dep prevents memoized children (React.memo) from re-rendering and keeps useEffect deps from re-firing.

Don't sprinkle them everywhere. Memoization has its own cost and stale-deps risk. Reach for them when (1) a child is wrapped in React.memo and needs stable props, (2) the value is genuinely expensive, or (3) it's an effect dependency. React 19's compiler auto-memoizes much of this.
Q23Mental model for useEffect deps and cleanup.Hard

Think "synchronize with an external system", not "lifecycle". The effect re-runs whenever a dependency changes; the returned cleanup runs before the next run and on unmount. Every value from render used inside must be in the deps — omitting them causes stale closures (you read an old value). Patterns: empty deps = run once (mount); cleanup unsubscribes listeners/timers/requests.

useEffect(() => {
    const sub = api.subscribe(id, setData);
    return () => sub.unsubscribe();   // cleanup on id change/unmount
}, [id]);

If you only use the latest value in a callback, a ref avoids re-subscribing. Don't put derived data in effects — compute it during render.

Q24What's the performance pitfall of Context, and fixes?Hard

Every consumer re-renders when the Provider's value changes by reference — and an inline object/array makes a new reference every render, re-rendering all consumers. Fixes: memoize the value, split contexts (state vs dispatch, or by domain) so unrelated consumers don't wake up, or use a selector-based store (Zustand/Redux/Jotai) for high-frequency state. Context is great for low-frequency, app-wide values (theme, auth, locale); it's not a state manager.

Q25Write a custom hook — and what's the rule for one?Mid

A custom hook is a function starting with use that composes other hooks to share logic (not markup). Each call gets isolated state.

function useDebouncedValue(value, delay = 300) {
    const [v, setV] = useState(value);
    useEffect(() => {
        const t = setTimeout(() => setV(value), delay);
        return () => clearTimeout(t);
    }, [value, delay]);
    return v;
}
Q26Controlled vs uncontrolled components?Easy

Controlled: React state is the single source of truth — the input's value comes from state and onChange updates it. Predictable, easy validation, but re-renders per keystroke. Uncontrolled: the DOM/native holds the value and you read it via a ref when needed — less code, fewer renders, harder to validate live. Default to controlled; use uncontrolled (or refs) for performance-sensitive or simple one-shot reads.

Q27What do Suspense and error boundaries give you?Mid

Suspense lets a subtree "wait" for something (lazy-loaded code via React.lazy, or data with a Suspense-enabled fetcher) and shows a fallback until ready — declarative loading states without scattered flags. Error boundaries are class components implementing getDerivedStateFromError/componentDidCatch that catch render-time errors in their subtree and show a fallback instead of crashing the whole tree. They don't catch errors in event handlers or async code — handle those with try/catch.

05 React Native

The bridge, the New Architecture, threading and performance.

Q28What threads does React Native run on?Mid
  • JS thread — runs your React code, business logic, the render pass.
  • Native/UI (main) thread — draws native views, handles gestures and platform UI.
  • Shadow thread — computes layout with Yoga (Flexbox) off the main thread.

Jank usually comes from blocking the JS thread (heavy work, too many re-renders) or chatty JS↔native traffic. Keep the JS thread free; move animations to the UI thread (Reanimated).

Q29How did the old "bridge" work and what was wrong with it?Hard

The legacy bridge passed messages between JS and native as asynchronous, batched, JSON-serialized payloads. Three costs: everything was async (no synchronous calls), every message paid serialize/deserialize overhead, and high-frequency traffic (fast lists, gestures, animations driven from JS) congested the bridge → dropped frames. It was also eager — all native modules initialized at startup.

Q30Explain the New Architecture: JSI, Fabric, TurboModules.Hard
  • JSI (JavaScript Interface) — a lightweight C++ layer that lets JS hold references to native (C++/host) objects and call them directly and synchronously, no JSON bridge. It's the foundation everything else sits on.
  • Fabric — the new rendering system. The shadow tree lives in C++ and is shared, enabling synchronous layout reads and better interop; concurrent-React-friendly.
  • TurboModules — native modules exposed over JSI and lazily loaded on first use (faster startup), with type-safe interfaces.
  • Codegen — generates the native↔JS type-safe glue from your TS spec files.
  • Hermes — the default JS engine, optimized for RN (fast startup, low memory, bytecode precompilation).

Net effect: synchronous, type-safe, lower-overhead JS↔native communication and smoother high-frequency UI.

Q31FlatList vs FlashList — how do you make long lists smooth?Hard

FlatList virtualizes (renders only items near the viewport) but creates and destroys item views as you scroll, which can drop frames on heavy rows. FlashList (Shopify) recycles views — conceptually like Android's RecyclerView — for far less work per frame.

Tuning levers: stable keyExtractor, getItemLayout / estimatedItemSize to skip measurement, memoized renderItem rows (React.memo), avoid anonymous functions/inline styles in rows, removeClippedSubviews, and downsized images. This is the RN analogue of the RecyclerView jank work I did on Android.

Q32Why does Reanimated run animations on the UI thread?Hard

Driving animations from JS means every frame crosses to native — if the JS thread stalls, the animation stutters. Reanimated runs animation logic in worklets: small JS functions (marked 'worklet') executed on the UI thread via JSI, reading/writing shared values there. The animation keeps hitting 60fps even while the JS thread is busy. Pair with react-native-gesture-handler so gestures are also handled natively for instant response.

Q33How do you expose a native module to JS? (your Kotlin/Swift work)Hard

You write platform code (Kotlin/Java on Android, Swift/Obj-C on iOS) implementing a module interface, mark methods callable from JS, and return results via promises/callbacks (old arch) or directly via JSI (TurboModules). On the New Architecture you declare a TS spec and Codegen produces type-safe bindings.

Concretely I bridged on-device ML inference and a dual-engine Speech-to-Text/Text-to-Speech layer this way — the heavy native work (audio capture, model inference) runs natively and only the results cross to JS, keeping the JS thread light. The mental model is identical to writing an Android library, just with a JS-facing surface.

Q34How does CodePush / OTA update work, and its limits?Mid

RN apps ship a JS bundle that the native shell loads at runtime. OTA tools (CodePush/Expo Updates) host a new bundle; the app downloads it and swaps it on next launch — so you can push JS/asset fixes without an app-store review. Limits: you can only update JS and assets, not native code (a new native module or RN upgrade needs a store release), and store policies require updates not to materially change the app's purpose. Use staged rollouts and mandatory-vs-optional flags. I set up CodePush + GitHub Actions pipelines for CriQ's multi-market releases.

Q35What does Hermes change vs JSC?Mid

Hermes is a JS engine built for RN. It precompiles JS to bytecode at build time, so the device skips parse/compile on launch — meaningfully faster cold start, lower memory and smaller TTI, especially on low-end Android. It also has RN-aware GC and tooling. It's the default in modern RN. JSC (JavaScriptCore) was the older default and still works, but Hermes wins on startup/memory for typical apps.

Q36React Navigation: native-stack vs JS stack, and performance.Mid

native-stack uses the platform's native navigators (UINavigationController / Fragment) for true-native transitions, gestures and memory behavior — prefer it. The JS stack renders transitions in JS (more customizable, but can jank). Use react-native-screens so off-screen routes are detached from the native view hierarchy (less memory). Lazy-load heavy screens, and keep navigation params small/serializable.

06 Android ↔ React Native

My differentiator — mapping native concepts to their RN equivalents.

Q37RecyclerView ↔ FlashList — same idea?Mid

Yes — both solve "render a huge list cheaply by reusing views". RecyclerView pools ViewHolders and rebinds; FlashList recycles cells the same way (unlike FlatList, which mounts/unmounts). DiffUtil's role maps to React's keyed reconciliation. Knowing the native original makes the RN tuning intuitive: stable keys ≈ stable item ids, estimatedItemSize ≈ getItemLayout, memoized rows ≈ light onBindViewHolder.

Q38Kotlin coroutines ↔ JS async/await — where do they diverge?Hard

Both write async code in sequential style. But Kotlin coroutines run on a real multi-threaded dispatcher pool with structured concurrency and cooperative cancellation; JS is single-threaded with a microtask-based event loop and no built-in cancellation (you reach for AbortController). So Dispatchers.Default for CPU work has no direct JS analogue — heavy JS work blocks the one thread, which is exactly why RN offloads to native modules or worklets. Coroutine scopes ≈ a discipline you must hand-roll in JS.

Q39JNI / native bindings ↔ RN native modules & JSI?Hard

Both are FFI boundaries between a managed runtime and native code. JNI bridges JVM↔C/C++; RN's JSI bridges JS↔C++ (and from there to Kotlin/Swift). The shared lessons transfer directly: minimize boundary crossings, batch data, do heavy work on the native side, and watch object lifetimes/threading across the boundary. Having written native modules in Kotlin and Swift, the JSI model felt like the same problem with a different glue layer.

Q40Jetpack Compose ↔ React — declarative UI parallels.Mid

Near one-to-one: recomposition ≈ re-render (both re-run UI functions when observed state changes and diff the result). remember ≈ useState/useMemo; LaunchedEffect ≈ useEffect; state hoisting is the same unidirectional pattern in both. Compose stability (skipping) ≈ React.memo + referential equality. Coming from Compose, React's render model needs almost no re-learning — which is why I pitch Android depth as a React-Native accelerant, not a context switch.
