01 JavaScript core

The engine-level questions every JS/RN interview opens with — explained from the ground up.

Q01Explain the event loop, micro- vs macro-tasks.Hard
In plain terms. JavaScript runs on one thread. It runs synchronous code first, then processes async callbacks from queues. There are two queue "tiers": microtasks (from Promises) get drained completely between every step, while macrotasks (setTimeout, I/O) get processed one at a time.

Picture a barista who handles all in-line orders (sync code) first, then before taking the next customer walk-in (macrotask), fully clears the pickup shelf of paid mobile orders (microtasks). Mobile orders always beat the next walk-in.

After each stack frame empties, the loop drains all microtasks (Promise .then, queueMicrotask, MutationObserver), then takes one macrotask (setTimeout, I/O, UI events), then repeats.

console.log('A');
setTimeout(() => console.log('B'), 0);   // macrotask
Promise.resolve().then(() => console.log('C'));   // microtask
console.log('D');
// Output: A D C B  — microtask C runs before macrotask B

Why it matters: a runaway chain of resolved promises can starve setTimeout and even rendering, since microtasks keep the loop from advancing. It also explains why "delay 0" is not actually zero — you're joining the back of a macrotask queue.

Q02What is a closure? Give a real use.Mid
In plain terms. A function "remembers" the variables from where it was born, even after that surrounding function has returned. That memory is the closure — it's how JavaScript gives you private state without classes.

Like taking a snapshot of your desk when you leave the room: the papers you had out (outer variables) come with you inside the photo, and you can keep referring to them later.

function counter() {
    let n = 0;              // private — outside code can't touch it
    return () => ++n;       // this arrow function closes over 'n'
}
const c = counter();
c(); c();                // 1, 2 — same 'n' persists between calls

A second real example — closures are how you build a rate-limited API key without exposing the counter:

function createRateLimiter(max) {
    let calls = 0;
    return function () {
        if (calls >= max) throw new Error('rate limited');
        calls++;
        return fetch('/api');
    };
}
const callApi = createRateLimiter(5);   // 'calls' is private to this instance

Classic trap: a var in a loop shares one binding across iterations, so all callbacks see the final value; let creates a fresh per-iteration binding, so each callback captures its own:

for (var i = 0; i < 3; i++) setTimeout(() => console.log(i), 0);
// 3, 3, 3 — all three closures share the same 'i'
for (let j = 0; j < 3; j++) setTimeout(() => console.log(j), 0);
// 0, 1, 2 — each iteration gets its own 'j'
Q03How is this determined? Why do arrow functions differ?Mid
In plain terms. In a regular function, this depends on how you called it, not where you wrote it. In an arrow function there's no own this at all — it borrows from the surrounding code, which is usually what you actually want in callbacks.
const user = {
    name: 'Sara',
    greetRegular: function () { return `Hi, ${this.name}`; },
    greetArrow: () => { return `Hi, ${this?.name}`; }
};
user.greetRegular();   // "Hi, Sara" — 'this' = user, the caller
user.greetArrow();     // "Hi, undefined" — arrow's 'this' comes from outer scope, not user

For regular functions, this is set by the call site: a method call (obj.fn()) binds this to obj; a bare call (fn()) gives undefined in strict mode; new Fn() binds to the new instance; fn.call(x)/apply(x)/bind(x) set it explicitly.

class Counter {
    n = 0;
    inc = () => { this.n++; };   // arrow: 'this' = the Counter instance always
}
const counter = new Counter();
button.addEventListener('click', counter.inc);   // works — no .bind(counter) needed

Why it matters: this is exactly why React class components historically needed this.handleClick = this.handleClick.bind(this) in the constructor for regular methods, but class fields with arrow functions sidestep the problem entirely.

Q04How does prototypal inheritance work?Mid
In plain terms. Every object has a hidden link to another object — its "prototype." When you access a property that doesn't exist on the object, JavaScript follows that link up the chain until it finds one, or reaches the end (null).

Like a lookup at a big company: if your team doesn't have the answer, they check with the department, then the division, then HQ, then give up. Each level is a prototype in the chain.

const animal = { eats: true };
const rabbit = Object.create(animal);   // rabbit.[[Prototype]] = animal
rabbit.jumps = true;
rabbit.eats;    // true — not on rabbit itself, found on 'animal' via the chain
Object.getPrototypeOf(rabbit) === animal;   // true

The class keyword is sugar over exactly this: methods live on Class.prototype and are shared by all instances — one copy in memory, not one per object.

class Animal {
    constructor(name) { this.name = name; }
    speak() { return `${this.name} makes a sound`; }   // lives on Animal.prototype
}
const a1 = new Animal('Rex'), a2 = new Animal('Fido');
a1.speak === a2.speak;   // true — same function object, shared via the prototype

Why it matters: this demystifies "why does my object have toString?" and explains patterns like mixins and delegation without deep class hierarchies.

Q05var vs let/const: hoisting and the TDZ.Mid
In plain terms. Both are "hoisted" — the engine knows about them before their line runs — but var is initialized to undefined, so using it early is silently weird; let/const throw a clear error if you touch them before their declaration.
console.log(a);   // undefined  (var: hoisted & initialized to undefined)
console.log(b);   // ReferenceError: Cannot access 'b' before initialization (TDZ)
var a = 1;
let b = 2;

That error zone between the top of the block and the actual declaration is the Temporal Dead Zone — it exists so accidentally using a variable before you meant to is a loud crash instead of a subtle bug.

function scopeDemo() {
    if (true) {
        var x = 1;    // leaks to function scope
        let y = 2;    // contained to this block
    }
    console.log(x);   // 1 — var escaped the if-block
    // console.log(y);   // ReferenceError — y doesn't exist here
}

Rule of thumb today: default to const, use let when reassigning, avoid var entirely in new code.

Q06== vs ===, and surprising coercions.Easy
In plain terms. === compares strictly — same type, same value. == tries to convert one side to match the other first, and its rules are famously weird. Default to ===.
null == undefined       // true  (only these two are == equal to each other)
null === undefined      // false — different types
NaN === NaN              // false — use Number.isNaN(x) instead
[] == ![]                // true  — both sides coerce to 0
0 == ''                 // true  — empty string coerces to 0
'0' == 0                // true  — string coerces to number
{} == {}                 // false — object identity, not contents

The one legitimate ==: x == null checks for both null and undefined in one expression — a genuinely useful shorthand. Otherwise, always ===.

Q07Promises vs async/await; how do errors flow?Mid
In plain terms. A Promise is a placeholder for a value that will settle later — either successfully (resolved) or with an error (rejected). async/await is prettier syntax for the same thing: it lets you write async code that reads top-to-bottom like normal code.
// same logic, two styles
function loadPromise() {
    return api.user()
        .then(user => api.feed(user.id))
        .catch(e => { reportError(e); throw e; });
}
async function loadAsync() {
    try {
        const user = await api.user();
        return await api.feed(user.id);
    } catch (e) { reportError(e); throw e; }
}

For running independent calls in parallel, awaiting sequentially wastes time. Use Promise.all to run in parallel and fail-fast if any fails; Promise.allSettled if you want every result regardless of outcome:

async function loadDashboard() {
    const [user, feed] = await Promise.all([
        api.user(), api.feed()      // both start immediately, run concurrently
    ]);
    return { user, feed };
}

Gotcha: an unhandled promise rejection is a bug (a hard crash in modern Node/RN). Always handle or explicitly propagate it.

Q08Shallow vs deep copy in JS — the safe options.Easy
In plain terms. A shallow copy only clones the top level — nested objects are still shared references, so mutating them affects both copies. A deep copy walks the whole tree and clones everything.
const a = { name: 'x', tags: ['js'] };
const shallow = { ...a };            // or Object.assign({}, a)
shallow.tags.push('ts');
a.tags;   // ['js', 'ts'] — same array, mutation leaked through to the original

const deep = structuredClone(a);   // modern built-in, handles cycles
deep.tags.push('go');
a.tags;   // still ['js', 'ts'] — deep copy was independent
  • structuredClone(obj) — the right default today. Built-in, handles Dates, Maps, Sets, typed arrays, circular references.
  • A hand-rolled recursive clone — when you need custom rules (skip functions, transform certain keys).
  • Avoid JSON.parse(JSON.stringify(x)): silently drops functions and undefined, mangles Dates into strings, and throws on cycles.

The implement-from-scratch problems (bigfrontend-style) interviewers love — with the intuition first, then the code.

Q09Implement debounce(fn, wait).Mid
In plain terms. "Wait until they stop, then act." Debounce postpones running your function until wait milliseconds have passed with no new calls — every new call restarts the clock. Perfect for search-as-you-type: you don't want to fire a request for "c", "ca", "cat".

Like an elevator door: every time someone new steps in, the door timer resets. It only actually closes once there's been a lull.

function debounce(fn, wait) {
    let t;
    return function (...args) {
        clearTimeout(t);                        // cancel any pending call
        t = setTimeout(() => fn.apply(this, args), wait);
    };
}

// usage
const onSearch = debounce(query => api.search(query), 300);
input.addEventListener('input', e => onSearch(e.target.value));

Two subtle things senior candidates get right: use a regular function (not arrow) for the returned wrapper so this flows from the caller, and forward arguments with apply. Common follow-up — add a leading option and a .cancel():

function debounce(fn, wait, { leading = false } = {}) {
    let t, called = false;
    function wrapper(...args) {
        if (leading && !called) { fn.apply(this, args); called = true; }
        clearTimeout(t);
        t = setTimeout(() => { called = false; if (!leading) fn.apply(this, args); }, wait);
    }
    wrapper.cancel = () => clearTimeout(t);
    return wrapper;
}
Q10Implement throttle(fn, wait) — and how it differs from debounce.Mid
In plain terms. "Run at most once per window." Throttle guarantees the function fires no more than once every wait ms — great for events that fire in a stream (scroll, mousemove, drag) where you want updates but not every update.

A metered water tap: no matter how fast you tap the handle, water only comes out once per fixed interval. Debounce, by contrast, is a tap that only opens after you've stopped touching it.

function throttle(fn, wait) {
    let last = 0, t = null, lastArgs;
    return function (...args) {
        const now = Date.now();
        const remaining = wait - (now - last);
        lastArgs = args;
        if (remaining <= 0) {              // window elapsed — fire now
            last = now; fn.apply(this, args);
        } else if (!t) {                   // schedule the trailing call
            t = setTimeout(() => {
                last = Date.now(); t = null;
                fn.apply(this, lastArgs);
            }, remaining);
        }
    };
}

// usage — scroll handler that runs at most every 200ms
window.addEventListener('scroll', throttle(updateProgressBar, 200));

The trailing call matters — without it, the last event in a burst is dropped, so a slow-moving cursor never updates its final position. Interview tip: mention both leading (fire immediately) and trailing (fire the last one) as configurable options.

Q11Implement your own Promise.all.Hard
In plain terms. Take an array of promises, wait for them all in parallel, and give back their results in the same order. If any one rejects, the whole thing rejects immediately with that error (fail-fast).
function promiseAll(items) {
    return new Promise((resolve, reject) => {
        const out = []; let done = 0;
        if (items.length === 0) return resolve(out);
        items.forEach((p, i) => {
            Promise.resolve(p).then(          // wrap non-promise values
                (v) => {
                    out[i] = v;                // write by index, not push
                    if (++done === items.length) resolve(out);
                },
                reject                         // first rejection wins
            );
        });
    });
}

// test
promiseAll([1, Promise.resolve(2), new Promise(r => setTimeout(() => r(3), 100))])
    .then(console.log);   // [1, 2, 3] — in order, after 100ms

Three details interviewers grade on: wrap each input in Promise.resolve because the array might contain plain values, not just promises; write to out[i] by index because promises resolve out of order but the caller needs original ordering; handle the empty-array case explicitly, otherwise it never resolves.

Q12Implement flatten(arr, depth).Mid
In plain terms. Take a nested array and "un-nest" it by one or more levels. flatten([1, [2, [3]]], 1) gives [1, 2, [3]]. Recursion is the natural fit — if an item is an array and we still have depth budget, flatten it and spread the result.
function flatten(arr, depth = 1) {
    return arr.reduce((acc, x) => {
        if (Array.isArray(x) && depth > 0)
            acc.push(...flatten(x, depth - 1));    // consume one level
        else acc.push(x);
        return acc;
    }, []);
}
flatten([1, [2, [3, [4]]]], 1);   // [1, 2, [3, [4]]]  — one level only
flatten([1, [2, [3, [4]]]], 2);   // [1, 2, 3, [4]]
flatten([1, [2, [3, [4]]]], Infinity);   // [1, 2, 3, 4]

Follow-up worth knowing: for arbitrary-depth flattening on huge inputs, recursion can blow the call stack. An iterative version using an explicit work stack avoids that:

function flattenIterative(arr) {
    const stack = [...arr], out = [];
    while (stack.length) {
        const next = stack.pop();
        if (Array.isArray(next)) stack.push(...next);
        else out.unshift(next);
    }
    return out;
}

In modern JS the built-in arr.flat(Infinity) does this — mention it after showing your implementation.

Q13Implement curry(fn).Hard
In plain terms. Turn a function that takes N arguments at once into one you can feed arguments piece by piece. add(1, 2, 3) becomes add(1)(2)(3) — or any mix like add(1, 2)(3). It calls the real function only once you've supplied enough arguments.
function curry(fn) {
    return function curried(...args) {
        if (args.length >= fn.length)                // enough args → invoke
            return fn.apply(this, args);
        return (...rest) =>                          // not enough → return a new collector
            curried.apply(this, [...args, ...rest]);
    };
}
const add = curry((a, b, c) => a + b + c);
add(1)(2)(3);   // 6
add(1, 2)(3);   // 6
add(1)(2, 3);   // 6

Why anyone uses this: partial application — building specialized functions out of general ones.

const applyTax = curry((rate, price) => price * (1 + rate));
const applyIndiaGst = applyTax(0.18);   // specialized, reusable
applyIndiaGst(1000);   // 1180
Q14Implement a generic memoize(fn).Mid
In plain terms. Cache a function's result the first time it's called with a particular set of args, and return the cached answer on subsequent calls. Trades memory for speed on pure functions.
function memoize(fn, keyFn = (...a) => JSON.stringify(a)) {
    const cache = new Map();
    return function (...args) {
        const key = keyFn(...args);
        if (cache.has(key)) return cache.get(key);
        const result = fn.apply(this, args);
        cache.set(key, result);
        return result;
    };
}

// example: memoized fibonacci — turns exponential into linear
const fib = memoize(n => (n <= 1 ? n : fib(n - 1) + fib(n - 2)));
fib(40);   // instant, vs. billions of calls unmemoized

Two things to discuss aloud: the cache key — JSON.stringify is simple but has holes (object key order matters, can't handle cycles, drops functions), so for serious use accept a custom keyFn. And the cache size — an unbounded cache is a memory leak; bound it with LRU eviction in production. React's useMemo is a special case: a cache of size 1, keyed by the deps array.

Q15Implement an EventEmitter (on / emit / off).Mid
In plain terms. The classic publish/subscribe: many parts of your code can register callbacks for a named event ("user:login"), and someone else can fire that event to notify them all. Underpins Node streams, DOM events, RN's device event bus, and a lot of your own custom bus code.
class EventEmitter {
    constructor() { this.map = new Map(); }
    on(ev, cb) {
        (this.map.get(ev) ?? this.map.set(ev, new Set()).get(ev)).add(cb);
        return () => this.off(ev, cb);           // unsubscribe handle
    }
    off(ev, cb) { this.map.get(ev)?.delete(cb); }
    emit(ev, ...args) {
        this.map.get(ev)?.forEach((cb) => cb(...args));
    }
}

// usage
const bus = new EventEmitter();
const unsubscribe = bus.on('user:login', user => console.log(`${user} logged in`));
bus.emit('user:login', 'sara');   // "sara logged in"
unsubscribe();                          // clean removal, no need to keep the callback reference

Two small polish moves that matter: use a Set instead of an array so duplicate registrations are natural no-ops and off is O(1); return an unsubscribe function from on, exactly the pattern React's useEffect cleanup uses. Follow-ups: add once(ev, cb), and guard so a throwing listener doesn't stop the rest from running.

Q16Implement deepClone handling cycles.Hard
In plain terms. Walk the whole object tree and make a fresh copy of everything. The tricky bit is cycles — an object referencing itself, or two objects referencing each other — because naive recursion would loop forever. The trick is to keep a map of "already cloned" objects and reuse them.
function deepClone(obj, seen = new WeakMap()) {
    if (obj === null || typeof obj !== 'object') return obj;
    if (seen.has(obj)) return seen.get(obj);        // cycle — reuse the copy
    const copy = Array.isArray(obj) ? [] : {};
    seen.set(obj, copy);                            // register before recursing
    for (const k of Reflect.ownKeys(obj))
        copy[k] = deepClone(obj[k], seen);
    return copy;
}

// test with a cycle
const a = { name: 'a' };
a.self = a;                    // self-reference
const clone = deepClone(a);
clone.self === clone;          // true — cycle preserved correctly, no infinite loop
clone !== a;                   // true — genuinely a separate object

Why a WeakMap: it doesn't prevent garbage collection of clone-source objects and allows objects as keys. Note the order — register the copy in seen before recursing into children, so a self-reference finds the (still-empty) copy already there instead of recursing forever.

In production: structuredClone(obj) is the built-in — handles Dates/Maps/Sets/typed arrays/cycles for you. Show the manual version to prove you understand it, then mention the built-in.

03 TypeScript

Types that make a large RN codebase safe to refactor.

Q17When and how do you reach for generics?Mid
In plain terms. A generic is a placeholder type — "whatever type you give me in, I'll give the matching type back out." It lets one function stay type-safe for many different types instead of either duplicating it or falling back to any and losing safety.
// without generics — loses the specific type
function first(arr: any[]): any { return arr[0]; }
const x = first([1, 2, 3]);   // x: any — TypeScript can't help you anymore

// with generics — the relationship is preserved
function first<T>(arr: T[]): T { return arr[0]; }
const y = first([1, 2, 3]);   // y: number — inferred automatically

Constrain generics with extends when you need to guarantee some capability of the type:

function prop<T, K extends keyof T>(obj: T, key: K): T[K] {
    return obj[key];
}
prop({ id: 1, name: 'x' }, 'name');   // inferred as string
// prop({ id: 1 }, 'nope');   // compile error — 'nope' isn't a key

Why it matters: generics are what let a single useState<T>(), FlatList<T>, or API-client function stay fully typed across every different data shape it's used with in your app.

Q18Which built-in utility types do you use most?Mid
In plain terms. Instead of writing near-duplicate interfaces by hand for every variant of a type you need (an update payload, a public-facing DTO), these transform an existing type automatically — one source of truth, many derived shapes.
interface User { id: string; name: string; email: string; }

type UserPatch = Partial<User>;              // { id?: string; name?: string; email?: string }
type UserPreview = Pick<User, 'id' | 'name'>;  // { id: string; name: string }
type UserNoEmail = Omit<User, 'email'>;    // everything except email
type UsersById = Record<string, User>;        // { [id: string]: User }
type ReadonlyUser = Readonly<User>;           // all fields become immutable

// derive from a function's own signature — one source of truth
function fetchUser(id: string) { return api.get<User>(`/users/${id}`); }
type FetchUserReturn = ReturnType<typeof fetchUser>;   // Promise<User>

Why it matters: when your API response type changes, every derived type (form state, patch payloads, previews) stays in sync automatically instead of drifting out of date across the codebase.

Q19How do you narrow a union type safely?Mid
In plain terms. A union type says "this could be one of several shapes." Narrowing is proving to the compiler which shape you actually have at a given point, so it lets you safely access fields specific to that shape.

The most reliable pattern is a discriminated union — every variant shares one literal field the compiler can key off:

type Result =
  | { kind: 'ok'; value: number }
  | { kind: 'err'; message: string };

function show(r: Result) {
    if (r.kind === 'ok') return r.value;      // narrowed to the 'ok' branch — .value is valid
    return r.message;                       // narrowed to the 'err' branch — .message is valid
    // r.value here would be a compile error — TS knows it's not 'ok'
}

Other narrowing tools: typeof for primitives, instanceof for classes, in for checking a property exists, and custom type guards that return x is Foo:

function isError(x: unknown): x is Error {
    return x instanceof Error;
}
try { risky(); } catch (e) {
    if (isError(e)) console.log(e.message);   // e is narrowed to Error inside this block
}

This mirrors Kotlin's sealed-class when exhaustiveness — the same "make impossible states unrepresentable" idea, expressed with structural types instead of a class hierarchy.

04 React

Render model, hooks and performance — shared by React and React Native.

Q20What is reconciliation and why are keys important?Mid
In plain terms. When state changes, React doesn't rebuild the real UI from scratch — it builds a lightweight description of what the UI should look like, compares it to the previous description, and applies only the differences. That comparison process is reconciliation.

React compares elements by type and position. Matching types get updated in place (fast); different types get torn down and rebuilt from scratch (expensive — loses state, focus, animation).

// without a key, reordering this list confuses React
{items.map(item => <Row data={item} />)}          // ✗ no key — React falls back to matching by position

// with a stable key, React tracks each row by identity
{items.map(item => <Row key={item.id} data={item} />)}   // ✓

Concretely what breaks without keys: if you insert a new item at the top of a list without stable keys, React (matching by position) thinks item 1 changed into item 2, item 2 changed into item 3, and so on — every row updates, any input focus or animation state on those rows is lost, and you pay for far more DOM work than the single actual insertion needed. Never use array index as a key for a reorderable list — index is position, so it defeats the whole point.

Q21Why must hooks be called unconditionally at the top level?Mid
In plain terms. React doesn't know your hooks by name — it tracks them by the order they were called in, like a numbered list. If that order changes between renders, React hands the wrong stored state to the wrong hook.
// ✗ breaks the rule — the call order can change between renders
function Bad({ show }) {
    if (show) {
        const [x, setX] = useState(0);   // only called sometimes — shifts every hook after it
    }
    const [y, setY] = useState(1);
}

// ✓ correct — always called, condition moves inside
function Good({ show }) {
    const [x, setX] = useState(0);
    const [y, setY] = useState(1);
    if (!show) return null;   // condition affects render output, not hook order
}

In the broken version, if show flips from true to false between renders, React's internal hook list shifts by one slot — y's state now gets read from where x's used to be. The rule ("only call hooks at the top level, only from React functions") exists specifically to keep that numbered list stable. The ESLint plugin catches this automatically.

Q22useMemo vs useCallback — and when NOT to use them.Hard
In plain terms. Both are "don't redo this unless the inputs actually changed." useMemo caches a computed value; useCallback caches a function reference so it doesn't count as "new" on every render.
function ProductList({ items, onSelect }) {
    // without useMemo: this expensive sort reruns on EVERY render,
    // even ones caused by unrelated state elsewhere in the app
    const sorted = items.slice().sort(expensiveCompare);

    // with useMemo: only reruns when 'items' actually changes
    const sortedMemo = useMemo(() => items.slice().sort(expensiveCompare), [items]);

    // without useCallback: a NEW function object every render —
    // breaks React.memo on , which re-renders anyway
    return sortedMemo.map(item =>
        <Row key={item.id} item={item} onSelect={() => onSelect(item.id)} />
    );
}

The point is referential stability: a stable prop/dependency prevents a memoized child (React.memo) from re-rendering unnecessarily, and keeps a useEffect dependency array from re-firing when nothing meaningful changed.

Don't sprinkle them everywhere. Memoization has its own cost (comparing deps, holding cached values) and a real risk of stale closures if you get the deps array wrong. Reach for them when a child is genuinely wrapped in React.memo and needs stable props, when the computation is genuinely expensive, or when it's an effect dependency. React 19's compiler auto-memoizes much of this, reducing how often you need to reach for these manually.
Q23Mental model for useEffect deps and cleanup.Hard
In plain terms. Think "synchronize with an external system," not "lifecycle method." The effect re-runs whenever a dependency changes; the returned cleanup function runs before the next run and on unmount — it's how you keep an outside system (a subscription, a timer) in sync with your component's current props/state.
useEffect(() => {
    const sub = api.subscribe(id, setData);
    return () => sub.unsubscribe();   // cleanup: unsubscribe the OLD id before subscribing to the new one
}, [id]);   // re-run whenever 'id' changes

Every value from render that's used inside the effect must be listed in the deps array — omitting one causes a stale closure, where the effect keeps reading an old value forever:

// ✗ stale closure — 'count' is captured once and never updates
useEffect(() => {
    const id = setInterval(() => console.log(count), 1000);
    return () => clearInterval(id);
}, []);   // missing 'count' — this always logs the initial value

// ✓ fixed — depend on count, or use the functional update form
useEffect(() => {
    const id = setInterval(() => console.log(count), 1000);
    return () => clearInterval(id);
}, [count]);

If you only need the latest value inside a callback without re-subscribing every time it changes, a ref sidesteps the problem. Don't compute derived data inside an effect — compute it directly during render instead.

Q24What's the performance pitfall of Context, and fixes?Hard
In plain terms. Every consumer of a Context re-renders whenever the Provider's value changes by reference. If you build that value inline on every render, it's a "new" object every time even if the actual data didn't change — so every consumer re-renders for no reason.
// ✗ new object literal every render — every consumer re-renders every time
<ThemeContext.Provider value={{ color, setColor }}>

// ✓ memoized — same reference unless color/setColor actually changed
const value = useMemo(() => ({ color, setColor }), [color]);
<ThemeContext.Provider value={value}>

Beyond memoizing the value, split contexts by how often they change — separate state from dispatch, or split by domain — so a consumer that only needs "dispatch" doesn't re-render every time unrelated "state" changes:

const StateContext = createContext(null);
const DispatchContext = createContext(null);   // dispatch never changes — zero re-renders from this one

For genuinely high-frequency state (form fields, animation values), reach for a selector-based store (Zustand, Redux, Jotai) instead — they let a component subscribe to just the slice it needs. Context is great for low-frequency, app-wide values (theme, auth, locale); it's not a general-purpose state manager.

Q25Write a custom hook — and what's the rule for one?Mid
In plain terms. A custom hook is just a function starting with use that composes other hooks to share logic across components — not markup, just behaviour. Each component that calls it gets its own, isolated copy of that state.
function useDebouncedValue(value, delay = 300) {
    const [v, setV] = useState(value);
    useEffect(() => {
        const t = setTimeout(() => setV(value), delay);
        return () => clearTimeout(t);   // cancel the stale timeout on each new keystroke
    }, [value, delay]);
    return v;
}

// usage — debounces search input without duplicating the effect logic per screen
function SearchBox() {
    const [query, setQuery] = useState('');
    const debounced = useDebouncedValue(query, 300);
    useEffect(() => { if (debounced) api.search(debounced); }, [debounced]);
    return <input value={query} onChange={e => setQuery(e.target.value)} />;
}

Why it matters: this is the JS analogue of a Kotlin extension function or a shared utility — logic gets reused without duplicating the underlying useState/useEffect wiring in every screen that needs it.

Q26Controlled vs uncontrolled components?Easy
In plain terms. Controlled: React state is the source of truth for the input's value. Uncontrolled: the DOM/native view itself holds the value, and you only reach in with a ref when you actually need to read it.
// controlled — React re-renders on every keystroke, but validation/formatting is trivial
const [name, setName] = useState('');
<input value={name} onChange={e => setName(e.target.value)} />

// uncontrolled — no re-render per keystroke, read the value only when needed
const ref = useRef();
<input ref={ref} defaultValue={''} />
const handleSubmit = () => console.log(ref.current.value);

Default to controlled — predictable, easy to validate live, easy to reset. Reach for uncontrolled (or a ref-based read) in performance-sensitive forms with many fields, or simple one-shot reads where you don't need live validation on every keystroke.

Q27What do Suspense and error boundaries give you?Mid
In plain terms. Suspense lets part of the UI "wait" for something — like a chunk of code still downloading — and shows a fallback until it's ready, without you managing a loading flag by hand. Error boundaries catch a crash in part of the tree and show a fallback instead of taking down the whole app.
// Suspense — code-splitting a heavy screen
const Settings = React.lazy(() => import('./Settings'));
<Suspense fallback={<Spinner />}>
    <Settings />
</Suspense>

// Error boundary — must be a class component (no hook equivalent yet)
class ErrorBoundary extends React.Component {
    state = { hasError: false };
    static getDerivedStateFromError() { return { hasError: true }; }
    componentDidCatch(error, info) { logToService(error, info); }
    render() {
        if (this.state.hasError) return <FallbackUI />;
        return this.props.children;
    }
}

Important limit: error boundaries only catch errors during rendering — not inside event handlers or async code. A throw inside an onClick handler won't be caught; wrap that logic in a normal try/catch instead.

05 React Native

The bridge, the New Architecture, threading and performance.

Q28What threads does React Native run on?Mid
In plain terms. RN splits work across three lanes: one thread runs your JavaScript, one draws the native UI, and one computes layout. Jank almost always traces back to one of these being blocked or overloaded.
  • JS thread — runs your React code, business logic, and the render pass that decides what the UI should look like.
  • Native/UI (main) thread — actually draws native views, handles touch gestures, and runs platform UI.
  • Shadow thread — computes layout with Yoga (RN's Flexbox engine), off the main thread.
// this blocks the JS thread — every component update queues behind it
function processHugeList(items) {
    return items.map(expensiveTransform);   // synchronous, thousands of items — freezes JS
}
// fix: chunk the work, or move it to a native module / worklet

Why it matters: jank usually comes from blocking the JS thread (heavy synchronous work, too many re-renders) or from chatty JS↔native traffic. Keep the JS thread free of heavy computation, and move animations to the UI thread with Reanimated instead of driving them from JS.

Q29How did the old "bridge" work and what was wrong with it?Hard
In plain terms. JS and native code live in separate worlds and can't call each other directly. The old "bridge" was a message-passing system between them — every call became a JSON message queued up and sent across, asynchronously, in batches.

Like two offices communicating only by mailed letters, batched once a day, rather than being able to walk down the hall and ask a question directly. Every "call" had to be written up, sent, and waited on — even for something as simple as "what's the width of this view right now?"

Three concrete costs of this design: everything was async — you couldn't ask native "what's this view's size?" and get an answer back in the same tick; every message paid serialize/deserialize overhead (converting to and from JSON); and high-frequency traffic (fast scrolling lists, gestures, animations driven from JS) could congest the bridge and drop frames. It was also eager — every native module initialized at app startup whether you used it or not, hurting cold start.

Q30Explain the New Architecture: JSI, Fabric, TurboModules.Hard
In plain terms. The New Architecture replaces the mail-by-letter bridge with a direct phone line: JS can hold a reference to a native object and call it synchronously, no JSON round-trip, no queue. Everything else (Fabric, TurboModules) is built on top of that one change.
  • JSI (JavaScript Interface) — a lightweight C++ layer letting JS hold direct references to native (C++/host) objects and call them synchronously. This is the foundation everything else sits on.
  • Fabric — the new rendering system. The shadow tree lives in C++ and is shared between JS and native, enabling synchronous layout reads and tighter interop — and it's built to work well with concurrent React.
  • TurboModules — native modules exposed over JSI, lazily loaded on first actual use instead of all at startup, with type-safe interfaces.
  • Codegen — generates the native↔JS type-safe glue code automatically from your TypeScript spec files, so the types on both sides can't drift apart.
  • Hermes — the default JS engine, optimized specifically for RN (fast startup, low memory, precompiled bytecode).
// TurboModule spec — Codegen generates the native glue from this
export interface Spec extends TurboModule {
    getDeviceName(): string;
    playSound(path: string): Promise<void>;
}
export default TurboModuleRegistry.getEnforcing<Spec>('DeviceInfo');

Net effect: synchronous, type-safe, lower-overhead JS↔native communication — smoother high-frequency UI (scrolling, gestures, animations) and faster startup since modules load lazily instead of eagerly.

Q31FlatList vs FlashList — how do you make long lists smooth?Hard
In plain terms. Both only render items near the visible viewport instead of the entire list ("virtualization"). The difference is what happens as you scroll: FlatList creates and destroys row views as they enter and leave the screen; FlashList reuses a fixed pool of views and just refills them — far less work per frame.
import { FlashList } from '@shopify/flash-list';

<FlashList
    data={items}
    renderItem={({ item }) => <Row item={item} />}
    keyExtractor={item => item.id}          // stable identity across renders
    estimatedItemSize={80}                  // skip a measurement pass — biggest perf win
/>

Tuning levers, in rough order of impact: give estimatedItemSize (FlashList) / getItemLayout (FlatList) so the list skips measuring every row; use a stable keyExtractor; memoize renderItem rows with React.memo so scrolling doesn't re-render unrelated rows; avoid creating new inline functions/styles inside each row (they defeat memoization, same trap as Q22); downsize images before rendering them at list scale.

This is the RN analogue of RecyclerView's view-recycling — estimatedItemSize plays the same role as skipping onMeasure passes on Android, and memoized rows play the same role as a lean onBindViewHolder.

Q32Why does Reanimated run animations on the UI thread?Hard
In plain terms. If you compute each animation frame in JS, every frame has to cross the JS↔native gap — and if the JS thread is ever busy (loading data, re-rendering something else), the animation stutters or freezes mid-motion. Reanimated instead runs the animation logic directly on the UI thread, so it keeps going smoothly no matter what JS is doing.
import Animated, { useSharedValue, useAnimatedStyle, withSpring } from 'react-native-reanimated';

function Card() {
    const offset = useSharedValue(0);
    const style = useAnimatedStyle(() => ({
        transform: [{ translateX: offset.value }],   // runs on the UI thread, not JS
    }));
    const onPress = () => { offset.value = withSpring(100); };
    return <Animated.View style={style} />;
}

The mechanism: useAnimatedStyle is a worklet — a small JS function marked to run on the UI thread via JSI, reading/writing shared values that live there too. The animation keeps hitting 60fps even while the JS thread is fully occupied elsewhere. Pair with react-native-gesture-handler so gestures are also recognized natively for instant response, not routed through JS first.

Q33How do you expose a native module to JS? (your Kotlin/Swift work)Hard
In plain terms. You write real platform code — Kotlin on Android, Swift on iOS — implementing a defined module interface, and expose specific methods for JS to call. The heavy lifting happens natively; only the results cross over to JS.
// Android — Kotlin native module (legacy bridge style, simplified)
class SpeechModule(reactContext: ReactApplicationContext) : ReactContextBaseJavaModule(reactContext) {
    override fun getName() = "SpeechModule"

    @ReactMethod
    fun speak(text: String, promise: Promise) {
        try {
            ttsEngine.speak(text)
            promise.resolve(null)
        } catch (e: Exception) { promise.reject("TTS_ERROR", e) }
    }
}
// JS side — call it like any async function
import { NativeModules } from 'react-native';
await NativeModules.SpeechModule.speak('Hello');

On the New Architecture you additionally declare a typed TS spec and Codegen produces the type-safe bindings automatically, instead of hand-writing the bridging by convention.

How this maps to my actual work: I bridged on-device ML inference and a dual-engine Speech-to-Text/Text-to-Speech layer this way at Dream11 — the expensive work (audio capture, model inference) runs natively in Kotlin/Swift, and only the final result crosses to JS, which is exactly why the JS thread stayed responsive despite the heavy processing underneath. The mental model is identical to writing a standalone Android library, just with a JS-facing entry point bolted on.

Q34How does CodePush / OTA update work, and its limits?Mid
In plain terms. An RN app ships a JS bundle that the native shell loads and runs at startup. OTA update tools let you swap that JS bundle for a newer one without going through the app store — the app just downloads the new bundle and uses it on the next launch.
// conceptual flow, e.g. with react-native-code-push
codePush.sync({
    updateDialog: false,
    installMode: codePush.InstallMode.ON_NEXT_RESTART,
});
// on next app open: new JS bundle is already in place, no store review needed

The key limit: you can only update JS and assets, not native code — a new native module, a native dependency bump, or an RN version upgrade still requires a full store release. Store policies also require that updates don't materially change the app's fundamental purpose (that's a store-review evasion risk, not a technical one). Use staged rollouts and a mandatory-vs-optional flag so a bad update doesn't hit 100% of users at once.

Applied directly on the resume: I set up CodePush together with GitHub Actions pipelines for CriQ's multi-market releases, so JS-level fixes could reach users in both the Indian and international builds without waiting on two separate store review cycles.

Q35What does Hermes change vs JSC?Mid
In plain terms. A JS engine has to turn your source code into something runnable. JSC (the old default) does that parsing/compiling on the device at launch. Hermes does it ahead of time, at build, so the device just runs pre-compiled bytecode — skipping a step that used to happen on every cold start.

That precompilation is why Hermes meaningfully improves cold-start time, lowers memory usage, and shrinks time-to-interactive — especially noticeable on low-end Android devices where parsing JS on-device was a real bottleneck. It also has GC tuned for RN's usage pattern and better debugging tooling. Hermes is the default in modern RN; JSC still works but loses on startup and memory for typical apps.

Q36React Navigation: native-stack vs JS stack, and performance.Mid
In plain terms. native-stack hands screen transitions to the real platform navigator (UINavigationController on iOS, Fragment-based on Android), so they look and feel truly native and don't compete with the JS thread. The plain JS stack renders and animates transitions itself in JS — more customizable, but can jank under load.
import { createNativeStackNavigator } from '@react-navigation/native-stack';
const Stack = createNativeStackNavigator();

<Stack.Navigator>
    <Stack.Screen name="Home" component={HomeScreen} />
    <Stack.Screen name="Detail" component={DetailScreen} />
</Stack.Navigator>

Default to native-stack. Also use react-native-screens underneath so off-screen routes are fully detached from the native view hierarchy instead of just hidden — meaningfully less memory as your navigation stack grows deep. Lazy-load heavy screens (don't import them until navigated to), and keep navigation params small and serializable rather than passing large objects through routes.

My differentiator — mapping native concepts to their RN equivalents, with side-by-side code.

Q37RecyclerView ↔ FlashList — same idea?Mid
In plain terms. Both solve "render a huge list cheaply by reusing views instead of creating new ones for every row." Once you understand one, the other's tuning knobs are immediately familiar — they're the same problem in two ecosystems.
// Android — RecyclerView + DiffUtil
class Diff : DiffUtil.ItemCallback<Item>() {
    override fun areItemsTheSame(a: Item, b: Item) = a.id == b.id
}
// getItemLayout equivalent: RecyclerView measures once per view TYPE, reuses it
// React Native — FlashList, same underlying idea
<FlashList
    data={items}
    keyExtractor={item => item.id}       // ≈ DiffUtil's areItemsTheSame
    estimatedItemSize={80}               // ≈ skipping RecyclerView's measure pass
    renderItem={({ item }) => <Row item={item} />}   // ≈ onBindViewHolder
/>

Stable keys ≈ stable item ids; a memoized renderItem row ≈ a lean onBindViewHolder that does minimal work per bind; estimatedItemSize/getItemLayout both exist to let the list skip an expensive measurement pass. Knowing the native original makes the RN tuning intuitive rather than cargo-culted from documentation.

Q38Kotlin coroutines ↔ JS async/await — where do they diverge?Hard
In plain terms. Both let you write asynchronous code that reads top-to-bottom instead of nested callbacks. The real difference: Kotlin coroutines run across a real multi-threaded pool with built-in structured cancellation; JS is fundamentally single-threaded, so "concurrency" there means interleaving, not parallel execution.
// Kotlin — CPU-bound work has a real dispatcher for it
viewModelScope.launch(Dispatchers.Default) {
    val result = heavyComputation()   // genuinely runs on another thread
}
// JS — there's no thread-pool equivalent; heavy work still blocks the one JS thread
async function heavyComputation() {
    return expensiveLoop();   // this blocks everything else on the JS thread while it runs
}
// the real fix in RN: move it to a native module, not just "await" it

So Dispatchers.Default for CPU-bound work has no direct JS equivalent — there's no thread to hop to. That's exactly why RN offloads genuinely heavy computation to native modules or Reanimated worklets rather than trying to "async" its way out of blocking the JS thread. Coroutine Job/scope cancellation also has no built-in JS analogue — you reach for AbortController and have to wire cancellation manually.

Q39JNI / native bindings ↔ RN native modules & JSI?Hard
In plain terms. Both are bridges between a managed runtime and native code — JNI connects the JVM to C/C++; JSI connects JS to C++ (and from there to Kotlin/Swift). The engineering lessons transfer almost one-to-one.
  • Minimize boundary crossings — batch data instead of many small calls, in both JNI and JSI, because each crossing has real overhead.
  • Do heavy work on the native side — the managed runtime (JVM or JS) should orchestrate, not compute.
  • Watch object lifetimes across the boundary — JNI global references and JSI host objects both need explicit lifecycle management or you leak native memory the garbage collector can't see.

Having written native modules in Kotlin and Swift for the Speech-to-Text/TTS work, the JSI model felt like solving the exact same class of problem (managed↔native handoff, careful about what crosses the boundary and how often) with a different glue layer on top.

Q40Jetpack Compose ↔ React — declarative UI parallels.Mid
In plain terms. Compose and React solve UI the same way: describe the UI as a function of state, and let the framework figure out what actually needs to change on screen. Once you know one well, the other's core concepts map almost line-for-line.
// Compose
@Composable
fun Counter() {
    var count by remember { mutableStateOf(0) }        // ≈ useState
    LaunchedEffect(count) { logCount(count) }             // ≈ useEffect
    Button(onClick = { count++ }) { Text("$count") }
}
// React — same shape
function Counter() {
    const [count, setCount] = useState(0);          // ≈ remember { mutableStateOf }
    useEffect(() => { logCount(count); }, [count]);   // ≈ LaunchedEffect(count)
    return <Button onClick={() => setCount(count + 1)}>{count}</Button>;
}

recomposition ≈ re-render — both re-run the UI function when observed state changes and diff the result rather than repainting everything. remember ≈ useState/useMemo; LaunchedEffect ≈ useEffect; state hoisting is the identical unidirectional-data-flow pattern in both. Compose's stability/skipping ≈ React.memo plus referential equality (Q22, Q33 on the Android page are the same underlying concept). Coming from Compose, React's render model needs almost no re-learning — which is exactly why Android depth is a genuine React Native accelerant, not a context switch.

07 JavaScript — deeper cuts

The follow-up questions after the basics — scope traps, type quirks, and function mechanics.

Q41Why does setTimeout inside a for loop with var behave differently than with let?Mid
In plain terms. var creates one shared variable for the whole loop. Every callback closes over that same variable, so by the time any callback runs (after the loop finishes), they all read the final value. let creates a fresh, per-iteration binding — each callback gets its own private copy.
// var — one shared binding
for (var i = 0; i < 3; i++) {
    setTimeout(() => console.log(i), 0);
}
// Output: 3 3 3 — loop finished (i=3) before any callback ran

// let — fresh binding per iteration
for (let j = 0; j < 3; j++) {
    setTimeout(() => console.log(j), 0);
}
// Output: 0 1 2 — each callback closes over its own j

The old workaround before let was an IIFE to create a new scope per iteration:

for (var i = 0; i < 3; i++) {
    (function(captured) {
        setTimeout(() => console.log(captured), 0);
    })(i);
}
// Output: 0 1 2 — the IIFE call creates a new scope, capturing the current value

Why it matters: this is the single most common closure interview gotcha, and a real source of bugs in event listeners registered inside loops.

Q42What happens when you try to modify a const object vs a const array?Easy
In plain terms. const locks the binding, not the value. You can't point the name at a different object or array, but you can freely change what's inside the object or array.
const user = { name: 'Sara' };
user.name = 'Ana';          // ✓ mutation is fine — we're not reassigning 'user'
user.role = 'admin';        // ✓ adding properties is fine too
// user = { name: 'Ana' };   // ✗ TypeError — reassigning the binding

const nums = [1, 2, 3];
nums.push(4);                // ✓ [1, 2, 3, 4]
nums[0] = 99;               // ✓ [99, 2, 3, 4]
// nums = [1, 2, 3];         // ✗ TypeError

For genuine deep immutability, use Object.freeze() — but note it's only one level deep:

const config = Object.freeze({ db: { host: 'localhost' } });
config.timeout = 5000;     // silently ignored in strict mode → throws in strict
config.db.host = 'prod';  // ✓ still mutates — freeze is shallow!

For deep immutability in production, use a library like Immer or TypeScript's Readonly<T> to enforce it at the type level.

Q43Why does typeof null return 'object' and how do you handle it?Mid
In plain terms. It's a decades-old bug in JavaScript that was never fixed because fixing it would break the web. The original implementation stored type tags in the low bits of a value's pointer — null's pointer was all zeros, which happened to match the "object" tag.
typeof null        // 'object'  ← historical bug
typeof undefined   // 'undefined'
typeof []          // 'object'  ← arrays are objects too
typeof {}          // 'object'
typeof function(){} // 'function' ← only exception to the "all objects are 'object'" rule

How to handle it — the three common checks and when to reach for each:

// Check for null specifically
x === null

// Check for null or undefined (the loose == shorthand)
x == null    // true for both null and undefined, nothing else

// Check for a real object (not null, not array)
typeof x === 'object' && x !== null && !Array.isArray(x)

// Check for any non-primitive (object or function)
x !== null && (typeof x === 'object' || typeof x === 'function')
Q44How do you reliably check if a variable is an array?Easy
In plain terms. typeof says "object" for arrays, null, and plain objects — not useful. instanceof breaks across iframes. Array.isArray() is the right answer — it works everywhere and handles all edge cases.
const arr = [1, 2, 3];

typeof arr          // 'object' — useless for arrays
arr instanceof Array // true — but breaks across iframes (different Array globals)
Array.isArray(arr)  // true — canonical, works everywhere
Array.isArray(null)  // false — handles null without throwing
Array.isArray({})   // false
Array.isArray('str') // false — strings are iterable but not arrays

// The iframe problem with instanceof:
// An array from a different frame has a different Array constructor,
// so [] instanceof Array is false across frame boundaries.
// Array.isArray checks the internal [[Class]], not the constructor — immune to this.

Rule: always use Array.isArray(). Check for array-like objects (NodeList, arguments) separately — they have .length but Array.isArray returns false.

Q45What's the difference between call, apply, and bind?Mid
In plain terms. All three set what this is inside a function. call and apply invoke immediately; bind returns a new function you call later. call and apply differ only in how you pass extra arguments.
function greet(greeting, punctuation) {
    return `${greeting}, ${this.name}${punctuation}`;
}
const user = { name: 'Sara' };

// call — invoke now, args passed one by one
greet.call(user, 'Hello', '!');    // "Hello, Sara!"

// apply — invoke now, args passed as an array
greet.apply(user, ['Hello', '!']);  // "Hello, Sara!"

// bind — return a NEW function, don't invoke yet
const boundGreet = greet.bind(user, 'Hey');  // partially applied
boundGreet('?');   // "Hey, Sara?"

Memory trick: call uses commas; apply uses an array. Real uses you'll reach for: apply to spread an unknown-length array as arguments (Math.max.apply(null, numbers), though spread does this better now); bind to create a callback with a pre-set context or partial args for event listeners and setTimeout.

// Classic bind use-case: detached method needing its context
class Timer {
    n = 0;
    tick() { this.n++; console.log(this.n); }
}
const t = new Timer();
setInterval(t.tick.bind(t), 1000);   // without bind, 'this' inside tick is undefined

08 React — deeper cuts

The follow-up questions after hooks basics — performance gotchas, memory, StrictMode and logic reuse.

Q46Why would you use useCallback and useMemo, and when do they hurt performance?Hard
In plain terms. They prevent re-doing expensive work or creating new object references on every render. But they themselves have a cost — deps comparison, extra memory — and can make things slower if used blindly on cheap operations.
// ✓ useCallback worth it: stable reference for a memoized child
const MemoRow = React.memo(Row);   // only re-renders when props change

function List({ items }) {
    // Without useCallback, handleDelete is a new function every render
    // → MemoRow's prop changed → it re-renders anyway → memo is useless
    const handleDelete = useCallback((id) => {
        setItems(prev => prev.filter(x => x.id !== id));
    }, []);   // empty deps: function identity is stable forever

    return items.map(item => <MemoRow key={item.id} item={item} onDelete={handleDelete} />);
}

// ✓ useMemo worth it: genuinely expensive computation
const sorted = useMemo(() =>
    items.slice().sort(complexCompareFn), [items]   // re-sorts only when items changes
);

// ✗ useMemo NOT worth it: wrapping cheap operations
const doubled = useMemo(() => count * 2, [count]);   // slower than just writing count * 2

When they hurt:

  • Wrapping cheap operations — the deps comparison costs more than re-computing.
  • Wrong or missing deps — produces stale closures that are worse than no memoization.
  • Over-memoizing a whole tree — the mental overhead of tracking what's stable becomes a maintenance burden without a measurable perf win.

Decision rule: Profile first. If no concrete perf problem is visible with React DevTools Profiler, don't add them. React 19's compiler auto-memoizes most of this, further reducing the need.

Q47How do you handle memory leaks in React applications?Hard
In plain terms. A React memory leak usually means a component got unmounted but something it started (a subscription, timer, fetch) is still running and still holds a reference to the component's state setter — preventing the component tree from being garbage collected.
// ✗ Classic leak: async update after unmount
useEffect(() => {
    fetchUser(id).then(user => setUser(user));   // component may unmount before fetch resolves
}, [id]);   // Warning: Can't perform a React state update on an unmounted component

// ✓ Fix: AbortController cancels the request when the component unmounts
useEffect(() => {
    const controller = new AbortController();
    fetch(`/api/user/${id}`, { signal: controller.signal })
        .then(r => r.json())
        .then(setUser)
        .catch(e => { if (e.name !== 'AbortError') throw e; });
    return () => controller.abort();   // cleanup: cancel on unmount or id change
}, [id]);
// ✗ Leak: event listener never removed
useEffect(() => {
    window.addEventListener('resize', handleResize);
    // no cleanup → listener accumulates every time component mounts
}, []);

// ✓ Fix: return cleanup
useEffect(() => {
    window.addEventListener('resize', handleResize);
    return () => window.removeEventListener('resize', handleResize);
}, []);

Other common sources: WebSocket/SSE connections, setInterval without clearInterval, third-party library subscriptions, and IntersectionObserver. The pattern is always the same — return a cleanup function that tears down whatever the effect set up.

Q48What happens if you don't return anything from a useEffect?Easy
In plain terms. Nothing bad — it's fine. The cleanup function is optional. Not returning anything means "there's nothing to clean up." React only calls the cleanup if you provide one.
// ✓ Fine — no cleanup needed, no return needed
useEffect(() => {
    document.title = `${count} notifications`;
}, [count]);   // setting document.title has nothing to undo

// ✓ Fine — explicit undefined return
useEffect(() => {
    logPageView();
    return undefined;   // same as no return
}, []);

// ✗ WARNING: returning something that isn't a function
useEffect(() => {
    return fetchData();   // fetchData() returns a Promise, not a function — React warns
}, []);
// Fix: mark the effect async inside, or use an IIFE
useEffect(() => {
    (async () => { await fetchData(); })();   // IIFE is synchronous from useEffect's POV
}, []);

The practical gotcha: the warning people actually hit is returning a Promise (by making the effect callback async). React expects either nothing or a synchronous cleanup function — not a Promise. Making the outer callback async violates this.

Q49Explain React.StrictMode and why your useEffect runs twice in development.Mid
In plain terms. StrictMode is a development-only tool that deliberately double-invokes certain functions to help you find bugs. It mounts, then immediately unmounts and re-mounts each component, so if your effect leaks something during unmount (no proper cleanup), you'll see the symptom twice in dev instead of only after the user navigates back and forth.
// This double-mount cycle is what StrictMode does in React 18+ dev mode:
// 1. mount   → effect runs
// 2. unmount → cleanup runs
// 3. re-mount → effect runs again

// ✗ Not cleanup-safe — you'll see two WebSocket connections in dev
useEffect(() => {
    const ws = new WebSocket('wss://...');
    // no cleanup
}, []);

// ✓ Cleanup-safe — StrictMode reveals it works correctly
useEffect(() => {
    const ws = new WebSocket('wss://...');
    return () => ws.close();   // StrictMode unmount calls this → no double connection
}, []);

StrictMode also detects deprecated API usage, warns about side effects in render, and flags unexpected mutations. It's entirely dev-mode only — the double-invoke doesn't happen in production. If your app breaks with StrictMode but works without, the double-invoke is surfacing a real bug (usually a missing cleanup), not causing a new one.

Q50How do you share logic between components without props or context?Mid
In plain terms. Custom hooks are the modern answer — extract shared stateful logic into a function starting with use, and any component that calls it gets its own isolated copy of that state.
// Custom hook — shared logic, isolated state per call site
function useOnlineStatus() {
    const [online, setOnline] = useState(navigator.onLine);
    useEffect(() => {
        const on = () => setOnline(true);
        const off = () => setOnline(false);
        window.addEventListener('online', on);
        window.addEventListener('offline', off);
        return () => { window.removeEventListener('online', on); window.removeEventListener('offline', off); };
    }, []);
    return online;
}

// Used in any component — each gets its own subscription
function Header() {
    const online = useOnlineStatus();
    return <div>{online ? 'Online' : 'Offline'}</div>;
}

The patterns that came before (and when they still apply):

  • Higher-Order Components (HOC) — wrap a component to inject props. Still used for cross-cutting concerns (authentication guards, analytics) but custom hooks handle most cases more clearly with less indirection.
  • Render props — pass a function as a child that receives data. Largely replaced by hooks but occasionally useful when you need to share a render pattern as well as logic.

Decision today: if it's logic only (no UI), custom hook. If it's a shared UI pattern, composition (just render it). HOC/render props for legacy code or when you genuinely need to wrap at the component boundary.

09 CSS & Layout

Layout mechanics, animation performance, and accessibility-aware visibility — the CSS questions that trip up most engineers.

Q51How do you handle z-index conflicts in large applications?Mid
In plain terms. z-index only works within the same stacking context. Every time you apply position + z-index, opacity < 1, transform, or filter to an element, you create a new stacking context — and elements inside it can never appear above elements outside it, no matter how high their z-index is.

Like floors in a building: you can rearrange furniture (z-index) however you want within your floor, but furniture on floor 2 can never appear above the floor 3 ceiling. Adding a transform to a parent drops everything inside to its own "floor."

/* Common trap: parent has transform, child can't escape it */
.card { transform: translateY(0); }    /* creates stacking context */
.card .tooltip { z-index: 9999; }     /* trapped inside .card's context */

/* Fix 1: move tooltip out of the stacking context (portal in React) */
/* Fix 2: avoid creating context on the parent */
.card { will-change: transform; }      /* avoids the stacking context from transform */

Systematic solution for large apps:

/* CSS custom properties for a z-index scale */
:root {
    --z-base: 0;
    --z-dropdown: 100;
    --z-sticky: 200;
    --z-overlay: 300;
    --z-modal: 400;
    --z-toast: 500;
}
.modal { z-index: var(--z-modal); }    /* readable, managed, not 9999 vs 99999

In React, use createPortal to render modals and tooltips directly into document.body, ensuring they're outside any ancestor stacking context:

ReactDOM.createPortal(
    <Tooltip>{content}</Tooltip>,
    document.body   // now in root stacking context, z-index works globally
)
Q52What's the difference between visibility: hidden and display: none for screen readers?Mid
In plain terms. Both hide content visually, but they behave differently in layout and for assistive technology. Neither is a reliable way to hide content from screen readers — aria-hidden is the correct tool for that.

/* Hide from everyone including screen readers */
.hidden { display: none; }
.hidden { visibility: hidden; }
.hidden { aria-hidden: "true"; }   /* explicit and reliable */

/* Visually hidden but READABLE by screen readers (the accessibility pattern) */
.sr-only {
    position: absolute;
    width: 1px;
    height: 1px;
    padding: 0;

    overflow: hidden;
    clip: rect(0, 0, 0, 0);
    white-space: nowrap;
    border-width: 0;
}

/* Visible to screen readers but hidden visually (opposite of .sr-only) */
.aria-hidden { aria-hidden: "true"; }   /* decorative icons, duplicated text */

Why it matters: CSS-based hiding is not a reliable accessibility control — browser and screen-reader combinations vary. Use aria-hidden="true" explicitly when you want something skipped, and the .sr-only pattern when you want something read but not seen (e.g. "Close" label on an icon-only button).

Q53How do you create smooth animations that don't block the main thread?Hard
In plain terms. The browser has a "compositor" thread that can run some animations entirely separately from JavaScript. If your animation only changes transform or opacity, the compositor handles it and it stays smooth even when JS is busy. Any other property — width, height, background, left — forces the browser back to the main thread and can cause jank.
/* ✗ Janky: triggers layout (reflow) every frame */
.animate-bad {
    animation: slide-bad 0.3s ease;
}
@keyframes slide-bad {
    from { left: 0; }
    to { left: 100px; }    /* left changes → layout recalculated every frame */
}

/* ✓ Smooth: only transform — compositor thread, no layout */
.animate-good {
    animation: slide-good 0.3s ease;
}
@keyframes slide-good {
    from { transform: translateX(0); }
    to { transform: translateX(100px); }   /* no layout pass needed */
}

Promote elements to their own compositor layer for complex animations:

/* will-change hints the browser to create a GPU layer ahead of time */
.animated-card {
    will-change: transform, opacity;   /* use sparingly — each layer costs GPU memory */
}
/* Remove it when animation is done: element.style.willChange = 'auto'; */

For JS-driven animations, use requestAnimationFrame rather than setTimeout — it syncs to the display refresh rate and automatically pauses when the tab is hidden:

function animate(timestamp) {
    element.style.transform = `translateX(${progress(timestamp)}px)`;
    if (!done) requestAnimationFrame(animate);   // called at ~60fps, synced to display
}
requestAnimationFrame(animate);

In React, prefer CSS transitions/animations over JS-driven ones. When you need complex gesture-driven animations in React Native, use Reanimated worklets (UI thread) — the same principle applied to mobile.

Q54Why does margin collapse happen and how do you prevent it?Mid
In plain terms. When two vertical margins touch — siblings stacked on top of each other, or a parent and its first/last child — CSS collapses them into a single margin equal to the larger of the two, rather than adding them together. It's a feature designed for readable typography, but it bites you in layouts where you expect padding.
/* Adjacent siblings collapse */
.box-a { margin-bottom: 20px; }
.box-b { margin-top: 30px; }
/* Gap between them: 30px, NOT 50px */

/* Parent-child collapse: if the parent has no border, padding, or BFC */
.parent { /* no padding, no border */ }
.child { margin-top: 40px; }
/* The 40px escapes the parent and applies to the parent itself */

How to prevent it — creating a Block Formatting Context (BFC) stops margins from collapsing across the boundary:

/* Any of these creates a BFC and prevents collapse */
.parent {
    overflow: hidden;     /* classic BFC hack */
    display: flow-root;   /* modern, purpose-built, no side effects */
    display: flex;        /* flex/grid containers are BFCs */
    display: grid;
}

/* Simpler fix: use padding instead of margin for spacing inside a container */
.parent { padding-top: 1px; }   /* even 1px stops the collapse */

Note: margin collapse only happens with vertical (block-axis) margins, never horizontal. And it doesn't happen inside flex or grid containers — another reason to default to those for layout.

Q55How do you make images responsive without breaking aspect ratios?Mid
In plain terms. An image with width: 100% and no height constraint stretches to fill its container horizontally but keeps its intrinsic aspect ratio — that's the baseline. The problems come when you need a specific aspect ratio regardless of the image's natural dimensions, or when you need different image files for different screen sizes.
/* Baseline responsive image */
img {
    max-width: 100%;    /* never wider than container */
    height: auto;       /* let browser calculate height to preserve ratio */
    display: block;     /* removes inline default whitespace gap */
}

/* Fixed aspect ratio regardless of content size */
.image-container {
    aspect-ratio: 16 / 9;    /* modern — browser handles the math */
    overflow: hidden;
}
.image-container img {
    width: 100%;
    height: 100%;
    object-fit: cover;        /* fill the box, crop if needed, preserve ratio */
    object-position: center;  /* which part to show when cropping */
}

For performance — serve different sizes to different screens:

<img
    src="photo-800.jpg"
    srcset="photo-400.jpg 400w, photo-800.jpg 800w, photo-1600.jpg 1600w"
    sizes="(max-width: 600px) 100vw, (max-width: 1200px) 50vw, 800px"
    alt="Description"
    loading="lazy"
    width="800" height="450"    <!-- set width+height to prevent layout shift -->
/>

Why width and height matter: the browser uses them to reserve space before the image loads, preventing Cumulative Layout Shift (CLS) — a Core Web Vital that affects SEO. Even if you set height: auto in CSS, the browser uses the intrinsic ratio from the attributes to hold space.

10 API, Performance & Reliability

Race conditions, slow networks, infinite scroll, and production-grade resilience.

Q56How do you handle race conditions in API calls?Hard
In plain terms. A race condition happens when two requests go out and the older one arrives after the newer one — so you render stale data as if it were current. The fix is ensuring only the latest request's result can update the UI.
// ✗ Race: fast typing causes stale results to overwrite fresh ones
function SearchResults({ query }) {
    const [results, setResults] = useState([]);
    useEffect(() => {
        api.search(query).then(setResults);   // "cat" resolves AFTER "cats" → shows old results
    }, [query]);
}

// ✓ Fix 1: AbortController — cancel the previous request outright
useEffect(() => {
    const controller = new AbortController();
    api.search(query, { signal: controller.signal })
        .then(setResults)
        .catch(e => { if (e.name !== 'AbortError') setError(e); });
    return () => controller.abort();   // cancel on next query or unmount
}, [query]);

// ✓ Fix 2: ignore stale — let request complete but don't apply its result
useEffect(() => {
    let stale = false;
    api.search(query).then(data => { if (!stale) setResults(data); });
    return () => { stale = true; };   // mark as stale on cleanup
}, [query]);

// ✓ Fix 3: use a library (React Query, SWR) — they handle this automatically

In React Native / mobile: the Flow equivalent is flatMapLatest which cancels the in-flight inner flow when a new query arrives — the same pattern, same problem, different syntax (covered in the Android deep dive Q19).

Q57What's your strategy for handling slow internet connections?Hard
In plain terms. Design for the assumption that the network will be slow or intermittent — show cached content immediately, be honest about loading state, and don't let one slow request block the whole page.
  • Show stale-while-revalidate: render cached content instantly, refresh in the background.
  • Skeleton screens over spinners: a skeleton retains the page shape and reduces perceived load time better than a blank spinner.
  • Optimistic UI: apply a mutation locally before the server confirms it, then rollback on failure.
  • Request timeouts: abort requests that take too long and show a retry prompt rather than waiting forever.
  • Progressive loading: render the shell and what's cached, then fill in what requires a network trip.
// Timeout pattern
const withTimeout = (promise, ms) =>
    Promise.race([
        promise,
        new Promise((_, reject) => setTimeout(() => reject(new Error('timeout')), ms))
    ]);

// Retry with exponential backoff
async function fetchWithRetry(url, retries = 3) {
    for (let i = 0; i <= retries; i++) {
        try { return await fetch(url); }
        catch (e) {
            if (i === retries) throw e;
            await new Promise(r => setTimeout(r, 300 * 2 ** i));   // 300ms, 600ms, 1200ms
        }
    }
}

const conn = navigator.connection;
if (conn?.effectiveType === '2g' || conn?.saveData) { loadLowRes(); }
Q58How do you implement infinite scrolling without performance issues?Hard
In plain terms. The naive approach — append new DOM nodes indefinitely — eventually makes the page so slow it freezes. Two fixes: use IntersectionObserver to trigger loads instead of polling scroll events, and virtualize the list so only visible rows are actually in the DOM.
// Step 1: IntersectionObserver to trigger the next page load
function InfiniteFeed({ fetchNextPage, hasNextPage }) {
    const sentinelRef = useRef();

    useEffect(() => {
        const observer = new IntersectionObserver(([entry]) => {
            if (entry.isIntersecting && hasNextPage) fetchNextPage();
        }, { rootMargin: '200px' });   // load 200px before sentinel is visible

        if (sentinelRef.current) observer.observe(sentinelRef.current);
        return () => observer.disconnect();
    }, [fetchNextPage, hasNextPage]);

    return (
        <div>
            {items.map(item => <Row key={item.id} item={item} />)}
            <div ref={sentinelRef} style={{ height: 1 }} />   {/* load trigger */}
        </div>
    );
}
// Step 2: For large lists, virtualize with react-window
import { FixedSizeList } from 'react-window';

const Row = ({ index, style }) => (
    <div style={style}>Item {index}</div>   // style MUST include the virtual position
);

<FixedSizeList height={600} itemCount={10000} itemSize={60} width="100%">
    {Row}
</FixedSizeList>
// Only ~10 DOM nodes regardless of itemCount=10000

Avoid the scroll event: window.addEventListener('scroll', ...) fires 60+ times per second, blocking the main thread. IntersectionObserver fires only when visibility actually changes, and it runs off the main thread.

Q59How do you detect and handle memory leaks in production?Hard
In plain terms. Production memory leaks show up as pages that gradually slow down and eventually crash — especially for long-running SPAs. You need both monitoring (to know it's happening) and tooling (to find what's leaking).

Detection:

  • Monitor performance.memory.usedJSHeapSize in your analytics — a steady upward trend that never decreases on navigation indicates a leak.
  • Use real-user monitoring (Datadog, Sentry Performance) to track memory-related crashes.

Investigation (Chrome DevTools):

// 1. Take a heap snapshot (DevTools → Memory → Heap snapshot)
// 2. Perform the action you suspect leaks (navigate to a route and back)
// 3. Force garbage collection (DevTools → Memory → collect garbage)
// 4. Take another snapshot
// 5. Compare: objects that grew and have no expected survivors are the leak

// Common culprits to look for in the snapshot:
// - Detached DOM nodes (elements removed from the DOM but still referenced)
// - Closures retaining component state (event listeners, observables)
// - Growing arrays/maps in module scope (global caches without eviction)

Fix patterns: WeakMap/WeakRef for caches (allows GC to collect entries); always clean up subscriptions; don't store React component refs in module-level variables; clear module-level caches when they're no longer needed.

Q60How do you make your website work when JavaScript fails to load?Mid
In plain terms. Progressive enhancement — start with working HTML and CSS (which browsers handle before any JS loads), then layer JS on top for a better experience. Graceful degradation works backwards: build the JS-enhanced version, then ensure core functionality doesn't break without it.
<!-- ✗ JS-only: nothing works without it -->
<button onclick="openMenu()">Menu</button>
<div id="menu" style="display:none">...</div>

<!-- ✓ Progressive: works with CSS only, enhanced with JS -->
<a href="#menu">Menu</a>
<div id="menu">...</div>   <!-- linked by anchor, navigable without JS -->

<!-- The checkbox hack: toggle CSS without JS -->
<input type="checkbox" id="nav-toggle" class="sr-only">
<label for="nav-toggle">Menu</label>
<nav class="sidebar">...</nav>
/* CSS handles the open state — no JS required */
#nav-toggle:checked ~ .sidebar { transform: translateX(0); }
.sidebar { transform: translateX(-100%); transition: transform 0.3s; }

For React/Next.js: use Server-Side Rendering (SSR) or Static Site Generation (SSG) so the initial HTML is fully rendered and meaningful. Add <noscript> fallbacks for critical interactive elements. Use native HTML elements (links, forms, buttons) that work without JS rather than <div onClick> everywhere. Critical paths (login, checkout) should work as plain HTML form submissions if needed.

11 Debugging & Tricky Scenarios

Real engineering judgment — how to think through bugs that don't have a simple "check the docs" answer.

Q61User reports a bug that only happens on mobile Safari — how do you debug it?Hard
In plain terms. Mobile Safari bugs are notoriously hard because you can't install other browsers on iOS, iOS Safari has its own WebKit quirks, and remote debugging requires physical hardware. The approach is systematic isolation — is it Safari, is it mobile viewport, is it iOS-specific, or is it a network condition?

Step 1 — reproduce and isolate:

  • Reproduce on a real device first (iOS Simulator is useful but masks some real-world GPU/memory constraints and doesn't perfectly replicate Safari's engine).
  • Test on macOS Safari too — if it breaks there, it's a WebKit/Safari issue generally, not just mobile. If it only breaks on mobile, it's viewport, touch, or performance-related.

Step 2 — remote debugging:

// Enable remote inspection:
// iOS Settings → Safari → Advanced → Web Inspector: ON
// macOS Safari → Develop → [your device name] → [your page]
// Now you have full DevTools on the iOS page from your Mac

Common iOS Safari–specific causes:

  • position: fixed + soft keyboard — the virtual keyboard doesn't resize window.innerHeight in Safari (use CSS svh/dvh viewport units or the visualViewport API).
  • 100vh includes the browser chrome in Safari — height: 100dvh excludes it correctly.
  • Date parsing: new Date('2024-01-15') returns Invalid Date in Safari — use '2024/01/15' or a library.
  • Flexbox gap in older Safari versions — add fallback margins.
  • Missing CSS features — check caniuse.com; Safari often lags by 1–2 years on features Chrome ships.
Q62Your app is slow but only for users with older devices — what do you check?Hard
In plain terms. Older devices have slower CPUs, less RAM, slower GPUs, and older browser engines. What feels instant on a development MacBook can be genuinely unusable on a 4-year-old mid-range Android or iPhone 8. Simulate, profile, then fix the highest-cost items.

Simulate the conditions:

// Chrome DevTools → Performance → CPU throttling: 6x (simulates mid-range Android)
// Chrome DevTools → Network → Slow 3G (simulates real-world connection)
// For Android: use a real low-end device or an emulator with low RAM config
// For React Native: use Flipper + the Performance monitor panel

What to look for (in order of frequency):

  • JS bundle size — a large bundle takes longer to parse and execute on a slow CPU. Profile with Webpack Bundle Analyzer or source maps in Chrome DevTools Coverage tab.
  • Long tasks on the main thread — anything over 50ms blocks input response. Check the Performance panel's main thread track for "Long Tasks" (red corner markers).
  • Render performance — too many React re-renders, non-virtualized long lists, large DOM counts. React DevTools Profiler shows which components render longest.
  • Image decoding — large unoptimized images kill CPU and GPU on old devices. Use WebP, lazy loading, and appropriately-sized images (srcset).
  • Animations triggering layout — animating non-compositor properties (width, height, top, margin) forces layout on every frame. Fix by using only transform and opacity.
Q63How do you handle a situation where your API suddenly starts returning different data?Mid
In plain terms. APIs change — fields renamed, types changed, fields added or removed. The app should degrade gracefully (show what it can, not crash) while you fix the root cause. Defensive data access and runtime validation prevent a backend schema change from taking down the frontend.
// ✗ Crashes on unexpected shape
const name = response.user.profile.name;   // TypeError if profile is null/undefined

// ✓ Optional chaining — returns undefined instead of throwing
const name = response?.user?.profile?.name ?? 'Unknown';

// ✓ Type validation at the boundary with Zod
import { z } from 'zod';
const UserSchema = z.object({
    id: z.string(),
    name: z.string().catch('Unknown'),   // fallback on parse failure
    email: z.string().optional(),
});
const user = UserSchema.safeParse(response.user);
if (!user.success) logError(user.error);   // alert but don't crash

Longer-term defence: agree on API versioning (/v1/, /v2/), use contract testing (Pact) so backend changes that would break the frontend are caught pre-deploy, and monitor field usage in your analytics so you can detect a removed field that's still being read.

Q64A feature works in development but breaks in production — what's your debugging process?Hard
In plain terms. "Works in dev, breaks in prod" almost always means one of a small set of things: different environment variables, minification changing behaviour, a missing API key or CORS config, real network latency instead of localhost, or a timing issue that only surfaces at production speed.

Systematic checklist:

  • Check the browser console first — prod errors often don't surface in monitoring right away, but they're almost always logged. Check the Network tab too for failing API requests or 4xx/5xx responses.
  • Minification — R8/Terser/webpack can rename, inline, or dead-code-eliminate things. Check for code that relies on Function.name, dynamic property access by string, or magic that breaks under minification.
  • Environment variables — a missing REACT_APP_* or process.env.* variable becomes undefined in prod, which silently breaks comparisons and string building.
  • CORS and auth — localhost doesn't hit the real CORS rules or auth flows. Verify prod API credentials and headers are in place.
  • Build the prod bundle locally — npm run build && npx serve build (React) or a release APK — running the prod build locally with local tools is the fastest way to reproduce a prod-only bug without deploying repeatedly.
  • Source maps — generate and upload source maps to your error tracker (Sentry, Datadog) so minified production stack traces resolve to readable code.
Q65How do you prioritize performance fixes when you have limited time?Mid
In plain terms. Measure first, fix the most impactful thing with the least effort, and prioritize user-visible improvements over numbers that look better on a dashboard but users don't feel.

Framework: impact × affected users / effort

  • Find the actual bottleneck first. Optimizing the wrong thing is the most common mistake. Profile with Lighthouse, WebPageTest, or React DevTools Profiler to identify what's actually slow — don't guess.
  • User-perceived metrics first. Fix Largest Contentful Paint (LCP) before Time to Interactive (TTI), because users feel the first paint. Fix what creates the most friction in the actual happy path, not edge cases.
  • Quick wins first. Image optimization (WebP, lazy loading, correct sizing) and code splitting typically yield large gains for low implementation cost. Ship those before tackling complex algorithmic optimizations.
  • Measure before and after. Without a before/after measurement on the same metric, you don't know if you fixed anything — and you can't convince stakeholders the time was worth spending.
