01 AI Engineering fundamentals

How LLMs, retrieval, and agents actually work — the concepts under every "AI feature."

A01What is an LLM actually doing, mechanically?Mid
In plain terms. An LLM reads all the text so far and predicts the single most likely next chunk of text ("token"), appends it, and repeats. There's no separate "thinking" step under the hood — the appearance of reasoning emerges from doing this prediction very well, over and over.

Like an extremely well-read autocomplete: given everything typed so far, what's statistically the best next word — but trained on so much text that "statistically the best next word" ends up looking like reasoning, coding, and conversation.

Text is broken into tokens (roughly word-pieces, not whole words). The model has a fixed context window — the maximum tokens of input + output it can attend to at once. Everything the model "knows" during a conversation must fit in that window; nothing persists between separate API calls unless you resend it. This is why prompt engineering and context management (what you put in that window, and in what order) matters so much.

A02What actually moves the needle in prompt engineering?Mid
In plain terms. Prompting is programming in natural language — you're not "asking nicely," you're specifying a task precisely enough that there's only one reasonable way to do it.

What reliably helps, roughly in order of impact:

  • Be explicit and specific — vague asks get vague answers. State the exact output format, constraints, and edge cases you care about.
  • Show examples (few-shot) — one or two worked examples of input→output often beat paragraphs of instruction, especially for format-sensitive tasks.
  • Let it reason before answering — asking the model to work through a problem step by step ("chain of thought") before giving a final answer measurably improves accuracy on multi-step tasks.
  • Structure with XML/markdown tags — separating instructions, context, and examples with clear tags reduces the model conflating them.
  • Positive AND negative examples — "do this, not that" is more reliable than "don't do that" alone.

Why it matters for interviews: the FDE/AI-eng bar isn't "can you write a prompt," it's "can you systematically iterate a prompt against real failure cases" — treat it like debugging, not like writing a wish.

A03Explain RAG — why not just fine-tune, or paste everything in the prompt?Hard
In plain terms. RAG (Retrieval-Augmented Generation) means: before answering, go fetch the specific pieces of your knowledge base that are actually relevant to this question, and hand only those to the model along with the question. Instead of the model "knowing" everything, it looks things up each time — like an open-book exam.

Why not just dump your whole knowledge base into the prompt? Context windows are finite and costly, and stuffing in irrelevant text dilutes the model's attention and hurts accuracy. Why not fine-tune the facts in? Fine-tuning is expensive to keep updated, doesn't reliably teach new facts (it's better for style/behaviour), and gives no easy way to cite sources or update just one document.

The pipeline: chunk your documents → embed each chunk into a vector (a list of numbers capturing its meaning) → store in a vector database → at query time, embed the user's question the same way → retrieve the chunks whose vectors are closest ("semantic search") → pass those chunks + the question to the model.

// conceptual RAG query
const queryVec = await embed(userQuestion);
const chunks = await vectorDb.search(queryVec, { topK: 5 });
const answer = await llm.complete({
    system: "Answer using only the provided context.",
    context: chunks.map(c => c.text).join('\n\n'),
    question: userQuestion
});

Where it breaks: retrieval quality is the real bottleneck, not the LLM — bad chunking (splitting mid-sentence), too few/many retrieved chunks, or a query that doesn't semantically match the phrasing in your docs all silently degrade answers. Debug retrieval before blaming the model.

A04What makes something an "agent" instead of just a chatbot?Mid
In plain terms. A chatbot answers with text. An agent can act — it decides to call a tool (search the web, run code, hit an API, edit a file), looks at the result, and decides what to do next, in a loop, until the task is done.

The mechanism is tool use / function calling: you describe available tools (name, description, parameters) to the model; the model's response can be either a normal text reply or a request to call one of those tools with specific arguments; your code executes the tool and feeds the result back in; the model continues. That loop — think, act, observe, repeat — is the agent.

// one iteration of an agent loop
const response = await llm.complete({ messages, tools: [searchTool, runCodeTool] });
if (response.type === 'tool_use') {
    const result = await executeTool(response.tool, response.args);
    messages.push(response, { role: 'tool_result', content: result });
    // loop continues — model sees the result and decides the next step
}

What makes agents hard in practice: error recovery (a failed tool call needs graceful handling, not a crash), knowing when to stop (avoiding infinite loops or runaway tool calls), and giving the agent just enough autonomy without letting it take unsafe or expensive actions unsupervised. This is exactly the shape of the agentic PR-review and test-generation pipelines I built at Dream11 — the agent reads a diff, decides what to check, runs static analysis or test tools, and iterates.

A05When does fine-tuning actually make sense over prompting?Mid
In plain terms. Prompting changes what you ask; fine-tuning changes the model itself by training it further on examples. Fine-tuning is a bigger, slower, more expensive lever — reach for prompting first, almost always.

Prompting (with good examples and RAG for facts) solves the vast majority of real product needs and iterates in seconds. Fine-tuning earns its cost when you need: a very specific output style or format consistently reproduced at scale where prompting is unreliable, much lower latency/cost by using a smaller fine-tuned model instead of a large general one, or a narrow, repetitive task where you have thousands of labeled examples.

Common mistake: fine-tuning to "teach the model facts." Fine-tuning is poor at reliably injecting new factual knowledge — it tends to shift style and behaviour more than ground-truth recall. For facts, RAG almost always wins over fine-tuning.
A06How does a vector database find "similar" text?Mid
In plain terms. An embedding model turns text into a list of numbers (a vector) such that texts with similar meaning end up as nearby points in that number-space — even if they don't share any words. "Similarity search" is just finding the nearest points.

Like plotting every sentence as a dot on a giant map, where dots close together mean similar meaning. "How do I reset my password" and "I forgot my login" land near each other even though they share almost no words, because an embedding model captures meaning, not just vocabulary.

Similarity is typically measured with cosine similarity (the angle between two vectors) or dot product. Searching this space naively (compare against every vector) doesn't scale, so vector DBs (Pinecone, Weaviate, pgvector, FAISS) build approximate-nearest-neighbour indexes (like HNSW) that trade a little accuracy for huge speed at scale. Why it matters: this is the retrieval half of RAG — embedding quality and index tuning directly determine answer quality, independent of which LLM you use downstream.

02 Claude Skills

What Skills are, how they're structured, how to build one well, and how to evaluate it.

A07What is a Claude Skill, conceptually?Easy
In plain terms. A Skill is a packaged folder of instructions and reference material that teaches an AI agent how to do a specific kind of task well — the same way a new hire gets a runbook instead of having to rediscover best practices from scratch every time.

Think of a Skill as a specialist's playbook handed to a generalist. The model already "knows" how to write code or edit a document in a generic sense, but a Skill encodes the hard-won, environment-specific details — which library to use, which pitfalls to avoid, what the output format needs to look like — so the agent doesn't have to rediscover them by trial and error on every task.

Practically, a Skill is a directory containing a primary instructions file (commonly named SKILL.md) plus any supporting reference files, and the agent reads it before attempting a matching task. The point isn't to teach the model something it's fundamentally incapable of — it's to close the gap between "generically capable" and "reliably good at this exact recurring task in this exact environment."

A08What does a well-structured Skill actually contain?Mid
In plain terms. A good Skill has two jobs: help the agent decide when to use it, and tell it precisely how once it's decided. Weak Skills conflate these and end up either never triggering or triggering on the wrong tasks.

The recurring structure that works well:

  • A sharp trigger description — the metadata the agent scans before even opening the file needs to unambiguously say what kinds of requests this Skill covers, using the actual words a user would use, not just an abstract category name.
  • Concrete constraints, not vague advice — "use library X, not Y, because Z breaks in this environment" beats "write good code." Specificity is what makes a Skill better than the model's generic knowledge.
  • Worked examples — a short example of the exact input→output shape removes ambiguity faster than another paragraph of prose.
  • Explicit failure modes — naming the mistakes people commonly make ("don't use localStorage here, it silently fails") saves the agent from repeating them.
  • Reference files kept separate — long lookup tables or exhaustive references live in their own files the agent reads on demand, so the main instructions file stays scannable rather than becoming an unreadable wall of text.
Anti-pattern: a Skill that just restates general best practices the model already knows. That adds token cost with no behaviour change. A Skill earns its place by encoding something the model would get wrong or leave to chance without it.

A minimal but well-formed example — a Skill for generating chart images in a specific internal format:

---
name: chart-generator
description: Use when the user asks for a chart, graph, or data
  visualization that needs to match our internal dashboard style.
  Covers bar, line, and pie charts from CSV or JSON input.
---

# Chart Generator

## When NOT to use this
Ad-hoc one-off plots for exploratory analysis — use plain matplotlib
for those. This Skill is only for charts matching our dashboard style.

## Required style
- Font: Inter, 12px axis labels
- Color palette: use palette.json in this folder, NOT default colors
- Always include a title and a source-data timestamp footer

## Common mistake to avoid
Do not use library auto-scaling for the y-axis on revenue charts —
it makes small monthly changes look dramatic. Always start y-axis at 0
for revenue/count metrics; auto-scale is fine for percentage metrics.

## Reference
See palette.json for exact hex values and chart-examples.md for
three worked input→output examples.

Notice the shape: the description up top is written the way a real request would be phrased ("chart, graph, visualization"), the body is almost entirely constraints the model wouldn't otherwise know (exact palette, the y-axis mistake), and the exhaustive example set lives in a separate reference file rather than bloating this one.

A09Walk through building a Skill from scratch.Mid
In plain terms. You don't write a Skill by imagining what might go wrong — you write it by watching the agent actually fail at the task, then encoding the fix. It's closer to debugging than authoring documentation from a blank page.
  1. Run the task without a Skill first. Give the agent the raw task and see where it goes wrong or makes inconsistent choices — those gaps are exactly what the Skill needs to close.
  2. Write the trigger description so the agent recognizes matching future requests, phrased the way real requests are phrased, not how a category label would be phrased.
  3. Encode the specific fixes for the failures you observed — the missing constraint, the wrong default, the undocumented environment quirk.
  4. Re-run the same task with the Skill loaded and confirm the failure is actually gone, not just superficially addressed.
  5. Test adjacent, related tasks to check the Skill generalizes and doesn't overfit to your one example.
  6. Trim aggressively. Every extra paragraph costs context budget on every future invocation — keep only what changed the outcome in testing.

Why it matters for the interview: the mindset transfers directly from mobile engineering — a Skill is basically a "runbook as code," and building one well is an iteration loop, not a one-shot writing exercise, exactly like debugging a flaky test or fixing a build-time regression.

A10How do you evaluate whether a Skill is actually working?Hard
In plain terms. The same way you'd evaluate any change to a system: build a fixed set of representative test tasks, run them with and without the Skill, and measure the difference — not vibes, a before/after comparison on the same inputs.

A practical eval loop:

  1. Collect a task set that represents the real range of requests the Skill should cover — including edge cases and near-misses that shouldn't trigger it.
  2. Define pass/fail criteria per task up front — a rubric, an exact-match check, or a grading model, decided before you look at outputs (otherwise you unconsciously grade generously).
  3. Run baseline (no Skill) vs. treatment (with Skill) on the identical task set.
  4. Look at the diffs, not just the aggregate score. A Skill that fixes 3 failures but introduces 1 new one in a previously-passing case is a net win numerically but might indicate the Skill overcorrects — worth reading actual transcripts, not just a pass rate.
  5. Track variance across runs — LLM outputs aren't fully deterministic, so run each task multiple times before concluding a change helped or hurt.

What "good" looks like: a measurable jump in the pass rate on the tasks the Skill targets, with no regression on tasks it shouldn't touch (checked via the trigger-description precision — does it fire only when it should?). This is precisely the discipline of the eval track below (A15–A20) applied to one specific artifact instead of a whole system.

A concrete, minimal harness for this comparison:

const tasks = [
    { input: 'chart of monthly revenue for Q1', mustInclude: ['y-axis starts at 0', 'Inter font'] },
    { input: 'quick scatter plot for my analysis', mustNotTrigger: 'chart-generator' },   // negative case
    // … representative sample of real request phrasings
];

async function runEval(withSkill) {
    let passed = 0;
    for (const t of tasks) {
        const output = await agent.run(t.input, { skills: withSkill ? ['chart-generator'] : [] });
        if (gradeAgainstCriteria(output, t)) passed++;
    }
    return passed / tasks.length;
}

const baseline = await runEval(false);   // score without the Skill
const treatment = await runEval(true);   // score with the Skill loaded
console.log({ baseline, treatment, delta: treatment - baseline });

The negative case in the task list matters as much as the positive ones — it's how you catch a Skill that's too eager to trigger and starts hijacking unrelated requests.

A11Skill vs. a longer system prompt vs. fine-tuning — how do you choose?Mid
In plain terms. All three encode "how to behave," but they differ in when the knowledge loads and how narrow it is. A system prompt is always present and general; a Skill is loaded only when relevant and can be as long and detailed as the task needs; fine-tuning bakes behaviour into the model weights themselves.
System promptSkillFine-tuning
Loadedalways, every callonly when task matchesbaked into the model
Best foridentity, tone, global rulestask-specific, detailed know-howconsistent style/format at scale
Cost per callalways paidonly paid when usednone (in weights)
Iteration speedfastfastslow, needs retraining

Rule of thumb: global behaviour that should apply to every interaction → system prompt. Deep, specific know-how needed only for certain recurring task types → a Skill, so you're not paying its token cost on unrelated requests. Extremely high-volume, narrow, stylistically-consistent output where prompting proves unreliable → consider fine-tuning, but only after a Skill has been tried and measured.

A12Why do good Skills use "progressive disclosure" instead of one giant file?Mid
In plain terms. Context window space is a real, limited budget — every token spent on instructions the agent didn't need is a token not spent on the actual task. So good Skills keep the main file short and push exhaustive detail into separate reference files the agent only opens if the task actually needs that specific detail.

Like a table of contents versus reading an entire textbook cover to cover for every question. A short index gets you to the right chapter fast; you don't reread the whole book to answer one question.

Concretely: the primary instructions file stays scannable — core guidance, decision rules, common cases — while deep reference material (exhaustive API listings, edge-case tables, long examples) lives in linked files the agent fetches only when the task at hand actually touches that area. This keeps the "always paid" cost of having the Skill loaded low, while still supporting deep, thorough coverage when it's genuinely needed.

03 Evals & reliability

How AI engineers actually know a system is good, and keep it good as it changes.

A13What is an "eval" and why is it the core skill of AI engineering?Mid
In plain terms. An eval is a test suite for an AI system — a fixed set of inputs with a way to judge whether the output was good. It's the equivalent of unit tests, except the thing under test is non-deterministic and "correct" is often fuzzy.

Without evals, "improving the prompt" is guessing — you tweak something, try it a couple of times, it feels better, and ship it. That's exactly how regressions sneak in on the cases you didn't happen to try. With an eval set, every change gets a before/after number on a fixed, representative sample, so you can actually tell improvement from noise.

Why it's the central skill: in traditional software, correctness is largely binary and testable with assertions. In AI systems, output quality is graded, not binary, and inputs are effectively infinite — so the entire discipline shifts from "did the test pass" to "how do we build a trustworthy, representative sample and a fair way to grade it." That's evals.

A14What are the main ways to grade an LLM's output?Mid
In plain terms. Grading gets harder as the task gets more open-ended. For tasks with one clearly right answer, checking is cheap and exact. For open-ended writing or reasoning, you often need another model — or a human — to judge quality.
  • Exact match / regex / schema validation — fastest and most reliable, when the task has a precisely correct answer (a JSON structure, a specific extracted value, a classification label).
  • Rubric-based grading — define specific criteria ("does it cite a source," "is the tone appropriate," "does it avoid X") and check each independently, either programmatically or with a grading model.
  • LLM-as-judge — use a strong model to score an output against a rubric or compare two outputs. Fast to scale, but has its own biases (favors longer or more confident-sounding answers) that you have to actively guard against.
  • Human review — the ground truth for genuinely subjective or high-stakes quality, but slow and expensive, so usually reserved for calibrating the automated methods above or for a small held-out sample.

Best practice: use the cheapest method that's actually valid for the task, and periodically spot-check LLM-judge scores against human judgment to make sure the judge hasn't drifted from what actually matters.

A15How do you build a good eval dataset?Hard
In plain terms. A test set is only useful if it actually looks like the real traffic your system will see. A handful of easy examples you made up yourself will tell you almost nothing about how it behaves in production.
  • Sample real usage where possible — actual user queries or, early on, queries from the people closest to the problem (support tickets, sales calls, internal dogfooding) rather than invented ones.
  • Deliberately include hard cases — ambiguous phrasing, edge cases, adversarial inputs, and near-misses that shouldn't trigger a feature. Easy cases don't discriminate between a good system and a mediocre one.
  • Keep a held-out set you don't optimize against directly, so you can catch overfitting to your visible eval set — the same discipline as a train/validation/test split.
  • Version and grow it over time — every real production failure you find becomes a new eval case, so regressions can never silently reappear.

Size: starting with even 20–50 well-chosen, diverse cases beats 500 near-duplicates. Grow it as you discover failure modes, rather than trying to enumerate everything up front.

A16How do you stop a prompt change from silently breaking something?Hard
In plain terms. Run your eval set automatically every time the prompt, model, or pipeline changes — the same discipline as running CI before merging code — instead of eyeballing a couple of manual test runs and shipping.

Treat prompts like code: version them, and gate changes on the eval suite passing at or above the current baseline score. A drop on any tracked metric is a signal to look closer before merging, exactly like a failing CI job. This is the transferable instinct from mobile: I built CI pipelines that caught regressions before release; an eval suite is the same principle applied to a prompt or an agent instead of a binary.

Extra wrinkle vs. normal CI: LLM outputs have run-to-run variance even at the same input, so a single failing run isn't necessarily a real regression — track pass rate across multiple runs per case, not one binary pass/fail per case, before concluding something broke.
A17How do you reduce hallucinations in a production AI feature?Hard
In plain terms. A model will confidently state things that aren't true when it's forced to answer from memory rather than from evidence you actually gave it. Every real lever for reducing this comes down to grounding the answer in real, retrievable evidence and making it easy for the model to say "I don't know."
  • Ground with retrieval (RAG) — give the model the actual source text and instruct it to answer only from what's provided, rather than relying on memorized training data.
  • Explicitly permit "I don't know" — a model without an escape hatch will guess; instructing it that an uncertain or unsupported answer is an acceptable outcome measurably reduces confident fabrication.
  • Ask for citations — requiring the model to point to which retrieved chunk supports each claim both improves grounding and gives users (and your evals) a way to verify the answer.
  • Lower temperature for factual tasks — reduces the randomness that can push a model toward a plausible-sounding but wrong completion.
  • Verify high-stakes outputs — for anything consequential, add a second pass (rule-based check, or a separate model call) that verifies claims against the source before the answer reaches the user.

Why it matters: this is the difference between a demo and a trustworthy product — hallucination rate under real, hard inputs is usually the single most-tracked metric in a production AI eval suite.

A18How do you think about cost and latency trade-offs in an AI feature?Mid
In plain terms. Bigger, smarter models cost more per call and respond slower. Most production systems don't need the biggest model for every step — they route different parts of the task to differently-sized models based on how hard that specific part actually is.
  • Model tiering — use a fast, cheap model for simple classification/routing steps, and reserve the most capable model for the step that genuinely needs deep reasoning.
  • Caching — repeated or near-identical queries (common in support/FAQ-style features) can skip the model call entirely.
  • Streaming — start showing tokens as they're generated instead of waiting for the full response, which improves perceived latency even when total generation time is unchanged.
  • Prompt length discipline — trimming unnecessary context (this is exactly what Skill progressive-disclosure design is for) directly reduces both cost and latency, since you pay per token.

Mobile-engineer instinct that transfers directly: this is the same trade-off shape as choosing Dispatchers.IO vs Default, or deciding what goes in a Baseline Profile — spend the expensive resource only where it actually changes the outcome.

04 Forward Deployed Engineer

The client-facing, rapid-integration skillset FDE roles actually test for — and how mobile/full-stack experience maps onto it.

A19What does a Forward Deployed Engineer actually do day to day?Easy
In plain terms. An FDE embeds directly with a customer, understands their specific messy real-world problem, and builds and ships a working solution against their actual systems and data — fast, and often on-site or in close collaboration with the client's team, rather than building a generic product feature from a distance.

The role sits at the intersection of three things: engineering (you actually build and ship working software, often integrating with a client's existing systems, APIs, and data), product judgment (you're often the one figuring out what to build, not just executing a spec someone else wrote), and relationship/communication skill (you're translating between what the client says they need and what's technically feasible, often in real time, in the room).

How it differs from a typical product engineering role: less time on long-term architecture for a single codebase serving millions of generic users, more time on rapid, correct, pragmatic solutions tailored to one customer's specific constraints and systems — and a much shorter feedback loop between "customer describes a problem" and "customer sees it solved."

A20What technical breadth do FDE interviews actually probe?Mid
In plain terms. Depth in one stack matters less than the ability to quickly get productive in whatever stack the client already has. Interviews probe for "can you learn and integrate fast," not just "do you already know this exact framework."
  • Systems integration — reading unfamiliar APIs and data formats quickly, understanding auth flows, working with data you don't control the schema of.
  • Full-stack comfort — being unblocked whether the gap is frontend, backend, data pipeline, or infra, because in a small embedded team there's often no one else to hand it to.
  • Rapid prototyping under ambiguity — building something functional from a loosely specified problem fast, then iterating with the client's feedback rather than a written spec.
  • AI/LLM integration specifically — increasingly core for AI-focused FDE roles: connecting an LLM to a client's actual data and systems (which is exactly the RAG/agent/tool-use material above), not just calling a chat API.
  • Debugging in someone else's environment — client systems are messier and less documented than your own codebase; comfort with ambiguity and incomplete information is a real, tested skill.

Directly transferable from my background: building native modules that bridge Kotlin/Swift into React Native, wiring Speech-to-Text/TTS across platforms, and standing up CriQ's full pipeline (Android + iOS + React Native + backend + CI/CD) end-to-end are all "learn an unfamiliar surface fast and make it work correctly" — the exact muscle FDE work exercises, just aimed at a client's systems instead of my own company's.

A21FDE vs. Solutions Engineer vs. product Software Engineer — how do they actually differ?Mid
In plain terms. The three roles sit on a spectrum from "mostly configuring an existing product" to "mostly writing new code," with FDE landing in the middle — closer to the customer than a typical product engineer, but writing more real code than a typical solutions/sales engineer.

Why this distinction matters in an interview: the honest answer to "why FDE and not a regular SWE role" should point at genuinely wanting the tight customer feedback loop and the breadth of ambiguous, cross-stack problem solving — not just "I like talking to people." Interviewers are checking whether you understand what you're actually signing up for.

A22How do you turn a vague client ask into a concrete technical plan?Hard
In plain terms. A client describes the symptom ("our support team is drowning in tickets"), not the technical solution. The FDE's job is to dig one or two levels deeper to find the actual bottleneck before writing any code — otherwise you build a technically impressive solution to the wrong problem.
  1. Ask for a concrete example, not an abstract description — "walk me through the last three tickets that took too long" surfaces the real pattern far faster than "what's slow about your process."
  2. Identify what's actually being asked for vs. what would actually help — clients often propose a specific solution ("build us a chatbot") when the underlying need is narrower ("stop answering the same 5 questions manually"); a smaller, faster win often beats the originally-requested scope.
  3. Propose the smallest version that proves value fast — a scoped prototype the client can react to beats a long requirements document, because their feedback on a real thing is more accurate than their feedback on a spec.
  4. Make the constraints explicit and shared — data access, security, timeline — so the client is deciding trade-offs with you, not discovering them after the fact.

Resume-relevant framing: this mirrors leading architecture and design reviews across cross-functional teams — the skill of translating a fuzzy ask ("this screen feels janky") into a concrete, scoped technical plan (diagnose the actual bottleneck, then fix it) is the same muscle, just pointed at an external client instead of an internal PM.

A23How would you frame CriQ as an FDE-style story in an interview?Mid
In plain terms. CriQ is a genuine "0 to 1, ambiguous problem, full ownership, ship fast, integrate everything" story — which is exactly the shape FDE interviewers are listening for, even though it wasn't literally a client-embedded role.

The narrative beats that map directly onto FDE evaluation criteria:

  • Ambiguous starting point, end-to-end ownership — led the 0→1 build across Android, iOS (via React Native), backend services, and release pipelines, for two different markets with different sign-in requirements — no existing playbook to follow.
  • Fast iteration under real constraints — CodePush OTA updates and GitHub Actions pipelines built specifically to let the team ship fixes without waiting on app-store review cycles, because speed of iteration mattered.
  • Deep systems integration — native modules in Kotlin and Swift bridging on-device ML inference and a dual-engine Speech-to-Text/Text-to-Speech pipeline into a cross-platform app, i.e., getting genuinely different systems (native ML runtimes, RN's JS layer, platform audio APIs) to work together correctly.
  • Built the AI tooling too, not just the product — the agentic PR-review and automated test-generation pipelines are a second, self-contained "solve an ambiguous internal problem with an AI system, ship it, and measure whether it actually helped" story, directly relevant to AI-engineering-flavoured FDE roles.

How to actually tell it: lead with the ambiguity and the constraint (two markets, tight timeline, no existing pattern), then the integration complexity, then the measurable outcome — that structure is what separates a strong FDE story from a generic "I built a feature" answer.

A24What's different about debugging and support in a client-facing role?Mid
In plain terms. When something breaks in a client-facing deployment, you're usually debugging a system you don't fully control, with a client watching, and often without the full observability tooling you'd have in your own production environment. Composure and communication under that pressure is a distinct, testable skill from pure debugging ability.

What actually matters in this setting:

  • Communicate status before you have the fix — "here's what I know, here's what I'm checking next, here's when I'll update you" keeps trust intact even mid-investigation; silence during an incident erodes confidence faster than the bug itself.
  • Reproduce with minimum access — client environments often restrict what logs/data you can see, so isolating a bug with partial visibility (targeted logging, asking the client for a specific screenshot or trace) is a real, practiced skill, not an afterthought.
  • Fix the immediate issue, then flag the systemic one separately — a client wants their specific problem solved now; whether it reveals a deeper architectural issue is a second, calmer conversation, not something to conflate with the urgent fix.

Directly relevant experience: systematic ANR and crash triage, memory profiling, and targeted fixes for platform stability at ShareChat's scale is the same discipline — diagnose under real production constraints, communicate clearly, fix without guessing — just aimed at internal stakeholders instead of an external client.
