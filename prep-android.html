01 Kotlin language

Language internals we tend to forget — explained from the ground up.

Q01Does val guarantee immutability?Easy
In plain terms. val locks the label, not the box. You can't point the name at a different object, but if the object itself has moving parts, those can still move.

Think of val as writing a friend's name in permanent marker in your address book. You can't erase the name and write a different friend — but that friend can still redecorate their house. The reference is fixed; the contents may not be.

So val gives you a read-only reference: reassignment is a compile error. It does not make the referenced object deeply immutable.

val list = mutableListOf("a")
list.add("b")   // fine — we're mutating the object, not the reference
// list = mutableListOf()  // compile error — reassigning the reference

Two subtleties worth knowing: a val backed by a custom getter can return different values on each read (it's read-only, not constant), and true immutability needs an immutable type — declare List instead of MutableList, or use a read-only val with no custom getter. Why it matters: "make it a val" is not the same as "make it thread-safe or unchangeable" — a very common misconception.

Q02lateinit vs by lazy — when each?Easy
In plain terms. Both let you declare a property now and fill it in later. lateinit means "I promise I'll set this myself before anyone reads it." lazy means "compute it automatically the first time someone actually asks for it, then remember the answer."
lateinitby lazy
Mutabilityvar onlyval only

Thread-safenoyes by default (SYNCHRONIZED)

Reach for lateinit when something else supplies the value shortly after construction — a dependency-injection field, a view bound in onCreate, a test's @Before setup. Reach for lazy for an expensive value you might never need (a heavy parser, a formatter, a database handle): you skip the cost until first use, and it's cached after.

lateinit var binding: ActivityMainBinding   // set in onCreate
val gson: Gson by lazy { Gson() }             // built on first use, then reused

Gotcha: reading a lateinit before you set it throws UninitializedPropertyAccessException. Guard with if (::binding.isInitialized) when unsure. For lazy, if the property is only ever touched from one thread you can pass LazyThreadSafetyMode.NONE to drop the locking overhead.

Q03What does the compiler generate for a data class?Easy
In plain terms. A data class is Kotlin writing all the tedious "value object" boilerplate for you — how to compare two of them, how to print one, how to copy one with a tweak.

For the properties in the primary constructor the compiler generates:

  • equals() / hashCode() — so two instances with the same field values are considered equal (structural equality), and work correctly as Map keys or Set members.
  • toString() — a readable User(id=1, name=Sara) instead of a memory address.
  • copy() — make a near-identical instance changing only what you name: user.copy(name = "Ann"). Essential for immutable state updates.
  • componentN() — enables destructuring: val (id, name) = user.
Gotcha that bites people: only properties declared in the primary constructor count toward equals/hashCode/copy. A property added in the class body is silently ignored by equality — a real source of "why do these two objects look equal when they're not?" bugs. Also, don't use a data class for entities with real identity (a DB row) where two different rows might share all current field values; use a stable id and compare on that.
Q04sealed class vs enum — and what's a sealed interface?Mid
In plain terms. An enum is a fixed list of identical-shaped options (like the four suits in a deck). A sealed class is a fixed list of options that can each carry different data (like "Loading" carries nothing, but "Error" carries a message).

An enum's members are all the same type and each is a single shared instance. A sealed hierarchy is a closed set of subtypes, each free to hold its own fields, and you can have many instances of each.

sealed interface UiState
data object Loading : UiState
data class Success(val data: List<Item>) : UiState
data class Error(val msg: String) : UiState

Because the compiler knows the full set of subtypes, a when over a sealed type is exhaustive — you don't need an else, and the day you add a new subtype, every when that forgot to handle it becomes a compile error instead of a silent runtime bug. That safety net is the whole point.

A sealed interface (Kotlin 1.5+) is the same idea but as an interface, so a type can belong to more than one sealed family and you avoid forcing an abstract class. Why it matters: modeling screen state as a sealed type (Loading / Success / Error) is the idiomatic Android pattern and makes impossible states unrepresentable.

Q05What does inline do, and why crossinline / noinline?Mid
In plain terms. Normally, passing a lambda to a function secretly creates a little object every time. inline tells the compiler to copy-paste the function's body (and the lambda's body) straight into the call site, so no object is created and there's no call overhead.

It's the difference between phoning a recipe helpline every time you cook (allocate an object, make a call) versus having the recipe steps printed right into your own cookbook page (inlined — no call at all).

That copy-paste unlocks two extra powers: non-local return (you can return from the enclosing function inside the lambda) and reified type parameters. It's exactly why map, filter, let, forEach are cheap despite taking lambdas.

The two modifiers handle edge cases:

  • noinline — "don't inline this particular lambda", needed when you want to store it in a variable or pass it onward (an inlined lambda has no object to pass).
  • crossinline — "still inline it, but forbid non-local returns", required when the lambda will be called from a different context (e.g. inside a Runnable or another object), where a non-local return wouldn't make sense.

Gotcha: don't inline large functions or ones called from many places — you're duplicating that bytecode at every call site, which bloats the binary. Inlining pays off for small, lambda-taking helpers.

Q06What problem do reified type parameters solve?Mid
In plain terms. At runtime the JVM "forgets" generic types — inside a normal generic function you can't ask "what is T?". reified makes the compiler remember the real type for you, so you can use it like a normal class.

This forgetting is called type erasure: List<String> and List<Int> are the same class at runtime. So T::class or x is T normally won't compile. Because inline functions get copied into the call site, the compiler does know the concrete type there, and reified exposes it:

inline fun <reified T> Gson.fromJson(json: String): T =
    fromJson(json, T::class.java)

val user: User = gson.fromJson(body)   // no need to pass User::class

You'll see it in DI lookups (get<Repo>()), JSON parsing, and Intent extras helpers. Constraint: reified only works inside inline functions.

Q07How are extension functions resolved — statically or dynamically?Mid
In plain terms. An extension function isn't really added to the class — it's a normal function in disguise that takes the object as a hidden first argument. So which one runs is decided by the declared type of your variable, not the actual object inside it.

They compile to plain static methods, so they don't participate in polymorphism (virtual dispatch). Resolution happens at compile time, by the type you wrote:

open class A; class B : A()
fun A.name() = "A"
fun B.name() = "B"
val a: A = B()
a.name()   // "A" — chosen by declared type A, even though the object is a B

Two consequences worth stating: extensions can't override real members, and if a member function and an extension have the same signature, the member always wins. Why it matters: extensions are great for readability and utility APIs, but if you expect overriding behaviour, you need real inheritance instead.

Q08Differentiate let, run, with, apply, also.Easy
In plain terms. These five all run a block of code "on" an object. They differ on just two questions: inside the block, is the object called this or it? And does the whole expression give you back the object, or the block's result?

Rule of thumb: want the object back to keep chaining? apply/also. Want a computed result? let/run/with. Prefer it when it reads clearer or you need to rename it.

Q09Explain property delegation (by) with a real use.Mid
In plain terms. Delegation lets a property hand off its "get" and "set" work to a helper object. Instead of writing the getter/setter yourself, you say "by someone-else" and that helper decides what happens on read and write.

The helper just needs getValue/setValue operators. Kotlin ships several: lazy (compute once), Delegates.observable (fire a callback on change), Delegates.notNull, and map-backed properties. Android and its libraries lean on this heavily.

var name: String by Delegates.observable("") { _, old, new ->
    log("$old -> $new")     // runs every time name changes
}

// You already use delegates constantly:
private val viewModel: HomeViewModel by viewModels()
private val args: DetailArgs by navArgs()

Why it matters: by viewModels(), by lazy, Fragment by navArgs(), and preference delegates are all this mechanism. Recognizing it demystifies a lot of "magic" Android syntax.

Q10companion object vs Java static?Easy
In plain terms. Kotlin has no static keyword. A companion object is one shared instance attached to the class, and you call its members through the class name — so it looks static, but it's actually a real object.

Because it's a genuine object, it can do things Java statics can't: implement an interface, be passed as an argument, hold state, be extended. That flexibility is the trade-off for a tiny bit more ceremony.

class User {
    companion object {
        fun fromJson(s: String): User = ...   // User.fromJson("…")
    }
}

For Java callers to see a member as a true static, annotate it @JvmStatic. And when you want a genuine free-standing utility (no class needed at all), a top-level function is the idiomatic Kotlin choice rather than a companion.

02 Coroutines & Flow

Structured concurrency and reactive streams — the highest-signal Android topic, built up from intuition.

Q11What is structured concurrency and why does it matter?Mid
In plain terms. Every background job you start belongs to a "parent" scope. The parent isn't finished until all its children finish, and if you cancel the parent, all its children are cancelled too. Nothing escapes and runs off on its own.

Like a parent taking kids to a park: they don't leave until every kid is accounted for, and if the parent calls it a day, all the kids come home. No child wanders off to keep playing forever — which, in code, is a leaked coroutine.

Concretely a CoroutineScope owns a Job; anything you launch inside becomes its child. This gives three guarantees: the scope waits for all children, cancelling the scope cancels all children, and a child's failure propagates to the parent. The payoff is automatic cleanup.

viewModelScope.launch {     // child of the ViewModel's scope
    val data = repo.load()
    _state.value = data
}
// When the ViewModel clears, viewModelScope cancels this automatically —
// no manual teardown, no leak, no update to a dead screen.

Why it matters: before structured concurrency (raw threads, callbacks), forgetting to cancel work was the #1 source of leaks and crashes on rotation. Now it's handled by construction.

Q12When do you use Main, IO, Default and Unconfined dispatchers?Easy
In plain terms. A dispatcher decides which thread pool your coroutine runs on. You pick based on the kind of work: touching the screen, waiting on the network/disk, or crunching numbers.
  • Main — UI work and anything that touches views. The only place you're allowed to update the screen.
  • IO — work that waits: network calls, database queries, reading files. A big elastic pool (up to 64 threads) because these threads mostly sit idle waiting.
  • Default — work that burns CPU: parsing, sorting, image processing. Pool sized to the number of cores, since more threads than cores wouldn't help.
  • Unconfined — starts on the calling thread and resumes wherever the suspending call happens to resume. Rarely used outside tests; easy to misuse.
suspend fun loadUser(): User = withContext(Dispatchers.IO) {
    api.fetchUser()          // waits on network → IO
}
Best practice — "main-safe" functions: switch threads inside the suspend function with withContext, so callers can call it from Main without thinking. The function owns its threading, not the caller.
Q12blaunch vs async?Easy
In plain terms. Use launch for "go do this, I don't need a value back." Use async for "go compute this value, I'll collect it later" — especially to run several computations at the same time.

launch returns a Job (a handle to cancel/join). async returns a Deferred<T>, a promise of a result you unwrap with await(). The classic use of async is parallel decomposition:

coroutineScope {
    val user = async { api.user() }    // both start immediately,
    val feed = async { api.feed() }    // running in parallel
    render(user.await(), feed.await())    // wait for both
}

If you called them sequentially with two withContext blocks, you'd wait for user then feed. With two async they overlap, so total time is the slower of the two, not the sum. Gotcha: a launch exception is thrown into the scope right away, but an async exception surfaces only when you call await().

Q13coroutineScope vs supervisorScope?Mid
In plain terms. Both group several child jobs. The difference is what happens when one child fails: in coroutineScope, one failure cancels all the siblings; in supervisorScope, each child can fail on its own without taking the others down.

A relay race (coroutineScope): if one runner drops out, the whole team is done. Independent solo runners (supervisorScope): one tripping doesn't stop the rest.

Use plain coroutineScope when the results belong together and a partial result is useless (fetch user and their settings — if one fails, abort). Use supervisorScope when you launch independent pieces and want partial success:

supervisorScope {
    launch { loadHeaderWidget() }    // if this crashes,
    launch { loadFeedWidget() }      // these two still load
    launch { loadAdsWidget() }
}

Why it matters: a dashboard where one failing widget shouldn't blank the whole screen is the textbook case for supervisor scoping.

Q14How does coroutine cancellation actually work?Hard
In plain terms. Cancellation is a polite request, not a force-kill. When you cancel, the coroutine is asked to stop, and it only actually stops at points where it checks the request. A busy loop that never checks will keep running.

Cancelling flips the Job to a "cancelling" state. Every suspension point from the coroutines library (like delay, withContext, network calls) checks this and throws CancellationException to unwind. But pure CPU work with no suspension points won't notice — you must cooperate:

while (isActive) {          // checks cancellation each iteration
    stepHeavyWork()
}
// or call ensureActive() / yield() inside the loop

Two important rules. First, don't accidentally swallow cancellation: a broad catch (e: Exception) will catch CancellationException too — rethrow it, or catch more specifically. Second, cleanup that must run even during cancellation (closing a file, a final DB write) goes in withContext(NonCancellable) { … }, otherwise the cleanup's own suspension points get cancelled.

Q15How do you handle exceptions in coroutines?Hard
In plain terms. For a specific risky call, wrap it in try/catch like normal. For "catch anything that slips through this whole scope," install a handler at the top. And remember launch and async report errors differently.
  • try/catch around a suspending call handles it locally — the simplest, most common approach.
  • launch throws immediately up the job tree; a child failure cancels the parent (unless it's a supervisor).
  • async stores the exception and only re-throws it when you call await() — so always wrap the await(), not the async.
  • CoroutineExceptionHandler is a last-resort net for uncaught exceptions on a root launch. It does nothing for async results and nothing mid-tree.
  • SupervisorJob stops one child's failure from cancelling its siblings.
val handler = CoroutineExceptionHandler { _, e -> log(e) }
scope.launch(handler) {
    try { risky() } catch (e: IOException) { showRetry() }
}

Why it matters: most "my error handler didn't fire" confusion comes from putting a handler on an async or somewhere other than the root coroutine.

Q16What is a Flow and what makes it "cold"?Mid
In plain terms. A Flow is a stream that produces values over time — like a list you receive one item at a time, asynchronously. "Cold" means the stream does nothing until someone actually starts listening, and it starts fresh for each listener.

A cold flow is like a song on a streaming service: it only plays when you press play, and each listener gets their own playback from the start. (A "hot" stream, by contrast, is like live radio — it's broadcasting whether or not anyone's tuned in.)

fun ticker() = flow {
    var i = 0
    while (true) { emit(i++); delay(1000) }
}
// The while-loop hasn't run yet. It starts only on collect:
ticker().collect { println(it) }   // now it emits 0, 1, 2, …

Why it matters: because collection re-runs the producer, a flow that queries the database gives every collector fresh data, and you can freely build, transform, and combine flows without triggering any work until the UI subscribes.

Q17StateFlow vs SharedFlow vs LiveData?Hard
In plain terms. All three are "hot" — they broadcast to whoever's listening. Use StateFlow for a current value that the screen mirrors (the UI state). Use SharedFlow for one-off events that should fire once (show a toast, navigate). LiveData is the older, lifecycle-aware version of the same idea.
  • StateFlow — always holds one current value, and only emits when that value actually changes (it de-duplicates). Perfect for screen state, which always has a "current" snapshot.
  • SharedFlow — a broadcaster you can tune: how many past values to replay to new subscribers (replay), buffer size, overflow policy. It emits every value, including repeats. Perfect for events with replay = 0.
  • LiveData — lifecycle-aware automatically and main-thread only; the pre-Flow standard. Flows need repeatOnLifecycle to collect safely.
val uiState = MutableStateFlow(UiState.Loading)   // state
val events = MutableSharedFlow<Event>(replay = 0)  // one-off events
Classic bug: using StateFlow for a navigation event. Because it holds and replays its latest value, rotating the screen re-delivers the event and you navigate twice. Events want SharedFlow(replay = 0).
Q18What does flowOn do and why not just withContext?Mid
In plain terms. flowOn sets which thread the flow's producer and upstream operators run on, while your collector stays on its own thread (usually Main). You can't just wrap emissions in withContext because a flow isn't allowed to change threads mid-emission.

Operators before flowOn run on the dispatcher you give it; the collector is untouched. This cleanly separates "where the work happens" from "where the UI updates":

repo.dataFlow()
    .map { heavyTransform(it) }
    .flowOn(Dispatchers.Default)   // producer + map run on Default
    .collect { render(it) }        // collected on Main (caller)

Trying to emit from inside a withContext(Default) block throws IllegalStateException — flows enforce that emissions preserve context, precisely so this stays predictable. flowOn is the sanctioned way to move upstream work.

Q19Implement search-as-you-type with Flow operators.Hard
In plain terms. As the user types, you want to wait until they pause, ignore repeat text, and — crucially — throw away a search request the moment they type another letter, so results never show for stale input.
queryFlow
    .debounce(300)                 // wait 300ms after typing stops
    .filter { it.length >= 2 }      // ignore 1-char noise
    .distinctUntilChanged()        // skip if text didn't change
    .flatMapLatest { q ->          // cancel the previous search
        repo.search(q).catch { emit(emptyList()) }
    }
    .flowOn(Dispatchers.IO)
    .collect { results -> _state.value = results }

The star is flatMapLatest: each new query cancels the in-flight request for the old one, so if "ca" is still loading when you type "cat", the "ca" request is dropped and you never see it flicker in. Two neighbours worth knowing: conflate() drops intermediate values when the collector is slow (keep only the newest), and buffer() lets a fast producer run ahead of a slow consumer instead of waiting for it.

Q20How do you turn a callback API into a Flow?Mid
In plain terms. Many Android APIs give you results by calling a listener you register (location updates, Firebase, sensors). callbackFlow wraps that listener as a flow and — importantly — automatically unregisters it when nobody's listening anymore.
fun locationFlow(client: LocationClient) = callbackFlow {
    val cb = LocationCallback { loc -> trySend(loc) }   // push into the flow
    client.register(cb)
    awaitClose { client.unregister(cb) }   // cleanup when collection stops
}

trySend pushes each callback value into the stream; awaitClose suspends until the collector cancels, then runs your teardown. Why it matters: forgetting to unregister a listener is a classic leak — callbackFlow makes correct cleanup the default, and it's the canonical bridge from the old callback world to flows.

Q21Why collect with repeatOnLifecycle instead of lifecycleScope.launch?Hard
In plain terms. If you just collect a flow in lifecycleScope, it keeps running while the screen is in the background — wasting work and risking updates to a screen the user can't see. repeatOnLifecycle automatically pauses collection when the screen isn't visible and resumes it when it comes back.

It starts collecting when the lifecycle reaches STARTED and cancels the collection at STOPPED, restarting it on the next STARTED. So a backgrounded screen consumes nothing.

lifecycleScope.launch {
    repeatOnLifecycle(Lifecycle.State.STARTED) {
        viewModel.uiState.collect { render(it) }
    }
}
// For a single flow, the shorthand:
// viewModel.uiState.flowWithLifecycle(lifecycle).collect { … }

Why it matters: before this API, people manually started/stopped collection in lifecycle callbacks and got it wrong, causing background work and leaks. This is the recommended pattern for collecting UI flows in Views.

03 Core & Lifecycle

The framework mechanics behind every screen — with the intuition first.

Q22Walk the Activity lifecycle and where you'd do what.Easy
In plain terms. Android tells your screen when it's being created, becoming visible, coming to the foreground, leaving the foreground, being hidden, and being destroyed. Each callback is your cue to set up or tear down the right things.
onCreate   // build UI, bind views, restore state — runs once per instance
onStart    // screen is now visible
onResume   // screen is in front and interactive — start camera/sensors
onPause    // losing focus — release exclusive resources, keep this FAST
onStop     // hidden — persist data, stop heavy updates
onDestroy  // final cleanup

Key mental model: onResume/onPause pair for "actively in front" work (camera, GPS, animations); onStart/onStop pair for "visible at all" work. onPause must be quick because the next screen can't fully resume until it returns.

The rotation trap: rotating the device destroys and recreates the Activity, wiping member fields. Keep transient UI data in a ViewModel/SavedStateHandle, not in Activity fields, or it vanishes on every rotation.

Q23Why does a Fragment have two lifecycles?Hard
In plain terms. A Fragment object can outlive its own UI. When you navigate forward, the fragment stays alive on the back stack but its view is destroyed and later rebuilt. So there are two clocks: one for the fragment, one for its view.

The fragment lifecycle spans onAttach…onDestroy; the view lifecycle is the shorter onCreateView…onDestroyView that can happen multiple times within one fragment's life.

The bug this causes: if you observe LiveData or collect a flow using the fragment (this) as the lifecycle owner, when the view is recreated you end up with the old observer still pointing at a destroyed view plus a new one — duplicate callbacks and a leak. Always use viewLifecycleOwner for anything tied to the UI, and null out your view binding in onDestroyView.
viewModel.state.observe(viewLifecycleOwner) { render(it) }   // ✓ view's clock
Q24Explain Looper, Handler and MessageQueue.Hard
In plain terms. This trio is how a thread stays alive doing "wait for the next task, run it, repeat" forever. It's the engine that lets the main thread keep processing taps, draws, and lifecycle events one after another without exiting.

Picture a restaurant kitchen. The MessageQueue is the rail of order tickets, sorted by when they're due. The Looper is the chef who keeps grabbing the next ticket and cooking it, endlessly. A Handler is any waiter who clips new tickets onto the rail — and those tickets get cooked on the chef's station (that Looper's thread).

  • MessageQueue — a time-ordered queue of Message/Runnable items.
  • Looper — an infinite loop pulling the next due item and dispatching it. The main thread's Looper is what keeps your app running and processes input, drawing, and lifecycle callbacks.
  • Handler — bound to a specific Looper; you post work through it and that work runs on the Looper's thread.
Handler(Looper.getMainLooper()).post { updateUi() }   // hop back to the UI thread

To give a worker thread its own loop: Looper.prepare() → create Handlers → Looper.loop() (or just use HandlerThread). Why it matters: coroutines and executors have replaced hand-written handlers for most work, but Dispatchers.Main is literally posting to the main Looper — understanding this explains how "run on the UI thread" works under the hood.

Q25Application Context vs Activity Context — which to pass where?Mid
In plain terms. A Context is a handle to "the app environment" — resources, system services, the ability to start screens. There are two flavours: one tied to a single screen (Activity) and one that lives as long as the whole app (Application). Pick by how long the thing you're giving it to will live.
  • Use the Activity context for UI that belongs to that screen — inflating views, showing themed dialogs, starting activities. It knows the current theme and window.
  • Use the Application context for things that outlive any single screen — singletons, the database, long-lived caches.
The leak: store an Activity context in a singleton or static field and you pin that whole Activity (and its entire view tree) in memory forever, even after it's closed. Rule of thumb: needs theming or a window → Activity context; otherwise → Application context.
Q26How do you survive process death (not just rotation)?Hard
In plain terms. When your app is in the background, Android can quietly kill it to reclaim memory, then rebuild it later when the user returns. A ViewModel survives rotation but not this. To restore key inputs, you save them to a small bundle that outlives the kill.

That bundle is exposed through SavedStateHandle, which persists across process death:

class SearchViewModel(val handle: SavedStateHandle) : ViewModel() {
    val query = handle.getStateFlow("q", "")
    fun setQuery(q: String) { handle["q"] = q }
}

Keep it small and serializable — this is for a search string, a selected tab, a scroll position, not a full data set. Anything large should be re-fetchable from the repository. How to test it: background the app and use "Don't keep activities" or terminate the process from Android Studio, then return — a lot of bugs only show up here, never during simple rotation.

Q27What causes an ANR and how do you debug one?Hard
In plain terms. ANR = "Application Not Responding." It's Android's dialog that pops up when the main thread is stuck too long to handle a tap or draw. The main thread is the only one that can update the UI, so if it's busy, the app appears frozen.

Triggers: roughly 5 seconds of blocked input dispatch, ~10s for a foreground broadcast, or a slow service start. The usual culprit is doing heavy work on the main thread — a database query, a network call, big JSON parsing — or a deadlock / lock contention there.

How to diagnose:

  • Read the stack dump at /data/anr/traces.txt — it shows exactly what the main thread was doing when it froze.
  • Use Play Console's ANR vitals for real-world field data.
  • Turn on StrictMode in debug to flag accidental main-thread disk/network I/O early.
  • Capture a Perfetto/Macrobenchmark trace to see the blocking work on the timeline.

Fix: move the work off-main with coroutines (withContext(IO)) and keep onCreate/onResume lean.

Q28WorkManager vs Service vs coroutine — how do you choose?Mid
In plain terms. Three questions decide it: does the work only matter while this screen is open, must it reliably finish even if the app closes, or must it run right now and keep going with the user aware of it?
  • Coroutine in viewModelScope — work tied to a screen, nothing to persist. Cancels when the screen goes away. (Loading a list, submitting a form.)
  • WorkManager — work that's deferrable but guaranteed: must survive app death and reboots, can wait for conditions (on Wi-Fi, while charging), and retries with backoff. (Sync, upload, periodic cleanup.)
  • Foreground Service — user-visible work that must run now and continuously, with a persistent notification. (Music playback, active navigation, live location.)

Why it matters: since Android 12+, the OS aggressively restricts plain background work, so "do it later, reliably" almost always means WorkManager now — reaching for a bare Service or a naked thread is usually the wrong answer.

04 Views & Compose

From RecyclerView recycling to recomposition — the performance heart of the UI.

Q29How does RecyclerView recycle, and why DiffUtil?Mid
In plain terms. A list might have 10,000 rows, but the screen only shows ~10 at a time. RecyclerView keeps just enough row views to fill the screen (plus a few) and reuses them as you scroll, refilling each with new data instead of building fresh views. DiffUtil then figures out the minimum that changed so only those rows update.

Like a small set of whiteboards on a conveyor: as one scrolls off the top, it's wiped and rewritten with the content that's about to appear at the bottom. You never buy 10,000 whiteboards.

Creating a view (inflation) is expensive; recycling pays that cost once per on-screen slot. When your data changes, DiffUtil (via ListAdapter) compares old and new lists and dispatches only the real changes — far better than notifyDataSetChanged(), which redraws everything and kills animations.

class Diff : DiffUtil.ItemCallback<Item>() {
    override fun areItemsTheSame(a: Item, b: Item) = a.id == b.id      // same row?
    override fun areContentsTheSame(a: Item, b: Item) = a == b          // changed?
}
Q30What is overdraw and how did you reduce jank on busy screens?Mid
In plain terms. Every frame, Android measures where things go, lays them out, and draws them — all within ~16ms to hit 60fps. "Overdraw" is painting the same pixel several times (stacked opaque backgrounds), and deeply nested layouts multiply the measure/layout cost. Both eat into that budget and cause stutter (jank).

Levers that reclaim frame time, roughly in impact order:

  • Flatten the view hierarchy — ConstraintLayout instead of nested LinearLayouts, so fewer measure/layout passes.
  • Remove redundant backgrounds so the GPU isn't repainting hidden pixels.
  • Use DiffUtil so scrolling rebinds only changed rows, not everything.
  • Profile with the GPU Overdraw debug overlay, Layout Inspector, and frame timing in Perfetto.

Real example: on a high-traffic chatroom feed at ShareChat, flattening layouts, fixing RecyclerView diffing, and auditing Compose recomposition together cut UI jank and unnecessary re-rendering by about 80%.

Q31What is recomposition and when does it trigger?Mid
In plain terms. In Compose you describe the UI as functions of your data. When a piece of data changes, Compose re-runs the functions that read that data to update the screen. That re-running is "recomposition."

It's smart about it: only composables that actually read the changed State re-run, and Compose can skip a composable entirely if its inputs haven't changed. Because a composable may run often, in any order, or be skipped, it must be side-effect-free — no network calls or mutations in the composable body itself (those go in effect APIs).

@Composable
fun Greeting(name: String) {   // re-runs only when 'name' changes
    Text("Hello $name")
}

Why it matters: the whole mental shift from Views is "don't imperatively update widgets — change the state and let Compose re-render." Understanding what triggers (and skips) recomposition is the key to both correctness and performance.

Q32remember vs rememberSaveable, and state hoisting?Mid
In plain terms. Since composables re-run constantly, remember is how you tell Compose "hang on to this value between re-runs instead of recreating it." rememberSaveable does the same but also survives rotation and process death. State hoisting means moving state up to the caller so a component becomes a simple, reusable "given values, show UI" function.

remember caches across recompositions but is lost on config change. rememberSaveable additionally saves to the state bundle (needs a Saver for custom types).

@Composable
fun Counter(count: Int, onInc: () -> Unit) {   // stateless: state lives in the parent
    Button(onClick = onInc) { Text("$count") }
}

Hoisting (pass value down, send onValueChange up) gives you unidirectional data flow: state has one owner, children are dumb and testable, and you avoid two sources of truth. Why it matters: it's the Compose equivalent of "lift state up" in React and the foundation of predictable UI.

Q33Why do unstable parameters hurt Compose performance?Hard
In plain terms. Compose saves work by skipping a component when its inputs haven't changed. But it can only skip if it can trust those inputs to reliably report equality. A plain List can't be trusted (it might be secretly mutable), so Compose gives up skipping and re-runs the component unnecessarily.

A type is "stable" if Compose can rely on its equals and be notified when its public properties change. List (the interface) is treated as unstable because nothing guarantees it won't mutate underneath — so passing one can defeat skipping and cause avoidable recompositions.

Fixes:

  • Use ImmutableList from kotlinx-collections-immutable, or wrap data in a stable holder.
  • Annotate types you can vouch for with @Immutable / @Stable.
  • Hoist lambdas with remember so they keep a stable identity.
  • Run the Compose compiler's stability/recomposition reports to find the actual offenders instead of guessing.

Why it matters: "my list screen recomposes too much" is one of the most common real Compose performance problems, and it almost always traces back to stability.

Q34When do you use LaunchedEffect vs DisposableEffect vs rememberCoroutineScope?Hard
In plain terms. Composables re-run constantly, so you can't just start a coroutine or register a listener in the body — it'd fire on every recomposition. The effect APIs are the safe, lifecycle-aware places to do "side effects," each suited to a different shape of work.
  • LaunchedEffect(key) — run a suspend block when the composable first appears or when key changes; auto-cancelled when it leaves. For one-shot loads, showing a snackbar, kicking off an animation.
  • DisposableEffect(key) — register something non-suspending (a listener, an observer) and provide an onDispose to unregister it. For cleanup-required resources.
  • rememberCoroutineScope — a scope you launch from event callbacks like onClick, not during composition.
  • SideEffect — push Compose state out to non-Compose code after each successful recomposition.
LaunchedEffect(userId) {           // re-runs if userId changes
    viewModel.load(userId)
}
DisposableEffect(Unit) {
    val cb = register()
    onDispose { unregister(cb) }   // cleanup on leave
}
Q35Two Compose gotchas: LazyColumn keys and Modifier order.Mid
In plain terms. Two things trip everyone up. First, lists need a stable "id" per item or Compose loses track of them on reorder. Second, the order you chain modifiers changes the result — they apply from the outside in.

Keys: give each item a stable identity so Compose preserves its state and animates correctly when the list changes. Without keys it falls back to position and mis-associates state.

LazyColumn {
    items(list, key = { it.id }) { item -> Row(item) }
}

Modifier order: Modifier.padding(16.dp).background(Blue) paints the background inside the padding (blue box with a gap around it), while Modifier.background(Blue).padding(16.dp) paints it around the padding (bigger blue box). Same two modifiers, different visual — because each wraps the next.

Bonus: wrap derived UI flags in derivedStateOf so they only recompute when the result changes, e.g. val showFab by remember { derivedStateOf { listState.firstVisibleItemIndex > 0 } } — this avoids recomposing on every scroll pixel.

05 Architecture

Patterns that keep a large codebase testable and maintainable.

Q36MVVM vs MVI vs MVP — trade-offs?Mid
In plain terms. All three separate "what the screen shows" from "the logic behind it," so the UI stays dumb and the logic stays testable. They differ in how the UI and logic talk to each other.
  • MVP — a Presenter holds the logic and calls methods on a View interface directly. Testable, but chatty and the two-way link is fragile on config changes. Older style.
  • MVVM — the ViewModel exposes observable state and the View just reacts to it. Less boilerplate, survives rotation, and is the Jetpack default.
  • MVI — one immutable state object, updated only by reducing intents (user actions), in a strict one-way loop. Very predictable and debuggable, at the cost of more ceremony.

What I actually do: MVVM with a single immutable UiState exposed as a StateFlow — it borrows MVI's predictability (one source of truth, unidirectional) without the full reducer machinery. That's the pattern I used for the ShareChat chatroom and intervention surfaces.

Q37Explain Clean Architecture layers and the dependency rule.Mid
In plain terms. Split the app into rings: UI on the outside, pure business rules in the centre, and the data sources (network/DB) on another outer ring. The rule is that inner rings never know about outer ones — the business logic has no idea Android, Retrofit, or Room even exist.

Three layers: presentation (ViewModel/UI) → domain (use cases + entities, pure Kotlin) → data (repositories, network, DB). The dependency rule says arrows point inward. Repositories are declared as interfaces in the domain and implemented in the data layer, so you can swap a data source or unit-test the domain with zero framework.

// domain layer — knows nothing about Android
class GetFeedUseCase(val repo: FeedRepository) {
    suspend operator fun invoke() = repo.load()
}

Use cases capture one business action and keep ViewModels thin. Caveat: don't apply the full ceremony to trivial screens — the extra layers cost indirection, so scale the rigor to the complexity.

Q38What does "single source of truth" mean in a repository?Mid
In plain terms. The UI trusts exactly one place for its data — usually the local database. The network never talks to the UI directly; it only updates the database, and the UI, which is watching the database, refreshes automatically.

The repository exposes a Flow from the DB; a refresh fetches from the network and writes into the DB, which then emits to everyone observing:

fun articles(): Flow<List<Article>> = dao.observeAll()   // UI reads only this
suspend fun refresh() { dao.upsert(api.fetch()) }        // network writes to DB

The payoff: offline support for free (DB works with no network), consistent data after writes (no manual UI patching), and one clean reactive contract instead of the UI juggling two streams and trying to merge them. Why it matters: this is the backbone of offline-first apps and pairs directly with Room's flow queries and Paging's RemoteMediator.

06 Dependency Injection

Dagger/Hilt mechanics — frequently the make-or-break Android round.

Q39Why DI, and what does Hilt add over Dagger?Easy
In plain terms. Instead of a class building its own helpers with new, someone hands the helpers to it. That "someone" is the DI framework. The win: you can swap a real network client for a fake one in tests, and you control how long each object lives from one place.

Like a chef who's handed prepped ingredients rather than growing their own vegetables. The chef (your class) focuses on cooking (its job), and in a test kitchen you can hand over pretend ingredients (mocks) without changing the recipe.

Dagger generates all this wiring at compile time — no reflection, so it's fast and type-safe, and mistakes are caught as build errors. But raw Dagger has a lot of boilerplate. Hilt sits on top and provides ready-made Android components and scopes via annotations (@HiltAndroidApp, @AndroidEntryPoint, @HiltViewModel), so you write far less setup and get a consistent structure automatically.

Q40@Binds vs @Provides — when each?Mid
In plain terms. Both tell the DI framework how to supply a type. Use @Provides when you need to build the object yourself (there's construction logic). Use @Binds when you just need to say "when someone asks for this interface, give them that implementation" — no building required.
@Module @InstallIn(SingletonComponent::class)
abstract class RepoModule {
    @Binds abstract fun bindRepo(impl: UserRepoImpl): UserRepo   // interface → impl
}

@Module @InstallIn(SingletonComponent::class)
object NetModule {
    @Provides @Singleton
    fun retrofit(): Retrofit = Retrofit.Builder()...build()      // needs a builder
}

@Binds is an abstract method — Dagger generates less code and it's the preferred way to map an interface to an implementation Dagger can already construct. @Provides is a real method body, needed for third-party types, builders, or any construction logic. Decision rule: can the implementation be constructor-injected and you only need to expose its interface? → @Binds. Do you have to assemble it yourself? → @Provides.

Q41Walk through Hilt components and scopes.Hard
In plain terms. A "scope" answers "how long should there be exactly one of this?" A @Singleton lives for the whole app; an @ActivityScoped object lives as long as one Activity. Hilt has a fixed set of these tied to Android's building blocks.
ComponentScopeLives as long as
SingletonComponent@Singletonthe Application
ActivityRetainedComponent@ActivityRetainedacross config changes
ViewModelComponent@ViewModelScopedthe ViewModel
ActivityComponent@ActivityScopedthe Activity
FragmentComponent@FragmentScopedthe Fragment

Scoping means "reuse one instance for this lifetime"; an unscoped binding hands out a fresh instance on every injection. The key rule: a binding can only depend on things from its own component or an ancestor — you can't inject an @ActivityScoped object into a @Singleton, because the singleton outlives the activity and would hold a stale reference (and leak it).

Gotcha: don't over-scope. Marking a cheap, stateless object @Singleton just pins memory for no benefit. Scope things that are expensive to build or that must be shared, and leave the rest unscoped.

Q42Two impls of the same type — qualifiers and multibinding?Mid
In plain terms. Sometimes you have two of the same type — say, two OkHttpClients, one authenticated and one not. A qualifier is a label that tells DI which one you want. Multibinding is the opposite: collect many implementations into one Set or Map.

Qualifiers disambiguate identical types:

@Qualifier annotation class Authed
@Provides @Authed fun authedClient(): OkHttpClient = ...
// inject with the label:  @Authed client: OkHttpClient

Multibinding (@IntoSet, @IntoMap) lets many modules each contribute one item into a shared collection — perfect for plugin-style registries. For example, each feature registers its own analytics tracker with @IntoSet, and the dispatcher injects the whole Set<Tracker> without knowing any of them by name. Why it matters: this is how you build extensible systems where features add capabilities without the core code changing.

07 Memory & Performance

Leaks, caching, startup and app-size — where senior engineers earn their title.

Q43Most common Android memory leaks and how you find them?Hard
In plain terms. A memory leak here means something short-lived (a screen) can't be cleaned up because something long-lived is still holding a reference to it. The garbage collector sees it's still "reachable" and keeps it — so closed screens pile up in memory until the app slows or crashes.

The usual suspects, all variations of "long-lived thing grips a short-lived thing":

  • A static field or singleton holding an Activity or View context.
  • Non-static inner classes or anonymous listeners (they secretly reference their outer Activity) that outlive it.
  • A Handler with delayed messages keeping the Activity alive until they run.
  • Listeners/observers you registered but never unregistered; unclosed disposables.
  • Observing LiveData/flows with the fragment instead of viewLifecycleOwner.

How to find them: LeakCanary watches destroyed objects, forces a GC, and if they're still reachable it computes the shortest reference chain from a GC root to the leak — literally showing you the field that's holding on. The Android Studio Memory Profiler (heap dumps, retained set) and StrictMode round out the toolkit.

Q44How does an LRU cache work, and where would you use one?Mid
In plain terms. LRU = "Least Recently Used." It's a fixed-size cache that, when full, throws out whatever you haven't touched in the longest time — betting that what you used recently, you'll use again soon.

Like a small desk: when it's full and you need space, you file away the paper you haven't looked at in ages, keeping the ones you keep reaching for within arm's reach.

The classic implementation is a hash map plus a doubly-linked list: the map gives O(1) lookup, and the list tracks recency (move an item to the front on each access, evict from the back). Android's LruCache caps by a size you define — for bitmaps, that's memory in KB via sizeOf:

val maxKb = (Runtime.getRuntime().maxMemory() / 1024 / 8).toInt()
val cache = object : LruCache<String, Bitmap>(maxKb) {
    override fun sizeOf(k: String, b: Bitmap) = b.byteCount / 1024
}

Where: the in-memory tier of an image loader (recently shown images stay hot), backed by a disk LRU for the next tier down.

Q45What are Baseline Profiles and how do they help startup?Hard
In plain terms. Normally Android compiles your code to fast machine code gradually, while the app runs, so the first launch is slower. A Baseline Profile is a list of the important startup code paths that gets compiled ahead of time at install, so the app is fast from the very first open.

The device's runtime (ART) uses the profile to AOT-compile those hot paths at install time instead of relying on the JIT warming up during use — commonly a 20–30% cold-start improvement and fewer janky early frames.

Workflow: use the Macrobenchmark library to run a representative journey (launch + a key screen), which generates the profile; ship it inside the APK/AAB; verify with cold-start traces. Pair it with the App Startup library to order your initializers and avoid a bloated Application.onCreate that delays everything.

Q46What does R8 do, and what breaks without keep rules?Mid
In plain terms. R8 is the tool that makes your release build smaller and faster: it deletes code you don't use, renames things to short names, and optimizes. The catch: it can't see code that's only reached "by name" at runtime (reflection), so it may delete or rename it and break things.

R8 does three jobs in one pass: shrinking (remove unused code/resources), obfuscation (rename to short names), and optimization (inlining, etc.). It breaks when code is accessed reflectively or by string — JSON model classes (Gson/Moshi), JNI-called methods, classes named in XML — because R8 can't tell they're used.

# keep model classes from being renamed/removed
-keep class com.app.model.** { *; }

Libraries usually ship their own "consumer" keep rules, so you mostly add rules for your own reflected classes. Golden rule: always test the minified release build, not just debug — R8 only runs on release, so these bugs are invisible in debug.

Q47How do you load a huge image without OOM?Mid
In plain terms. A photo's memory cost depends on its pixel dimensions, not its file size. A 12-megapixel image is tens of megabytes once decoded — even if the JPEG on disk is small. Loading several at full size blows the memory budget (OutOfMemory). So you shrink it to the size you'll actually display before decoding.

The math: memory = width × height × bytes-per-pixel. If your ImageView is 400px wide, decoding a 4000px image wastes 100× the memory. The fix is to read just the bounds first, compute a downscale factor, then decode at the smaller size:

val opts = BitmapFactory.Options().apply { inJustDecodeBounds = true }
BitmapFactory.decodeFile(path, opts)          // read size only, no pixels
opts.inSampleSize = calculateInSampleSize(opts, reqW, reqH)  // e.g. 4
opts.inJustDecodeBounds = false
val bitmap = BitmapFactory.decodeFile(path, opts)  // decode downsampled

In practice: use Coil or Glide, which handle sampling, memory + disk caching, bitmap pooling, and lifecycle for you — and that's the right answer too.

08 Data, Build & Tooling

Persistence, modularization and the build pipeline I love optimizing.

Q48Room essentials: relations, @Transaction, reactive queries.Mid
In plain terms. Room is a friendly layer over SQLite: you write typed methods instead of raw SQL strings, and it checks your queries at compile time. Its superpower is that a query returning a Flow auto-updates whenever the underlying data changes.

A DAO method returning Flow<T> re-emits every time its tables change — that's what makes the database a live single source of truth. Use @Transaction for multi-step reads/writes that must be all-or-nothing and consistent (e.g. a @Relation query that runs several statements, or updating parent + children together).

@Query("SELECT * FROM article")
fun observeAll(): Flow<List<Article>>   // live-updating

@Transaction @Query("SELECT * FROM user")
fun usersWithPosts(): List<UserWithPosts>   // atomic multi-table read

Provide explicit Migrations when the schema changes (and test them); fallbackToDestructiveMigration is dev-only since it wipes data. And prefer DataStore over SharedPreferences for key-value: it's async, transactional, and flow-based, so no accidental disk I/O on the main thread.

Q49How does Paging 3 work end to end?Hard
In plain terms. For a feed with thousands of items, you don't load everything at once — you load a page, and fetch the next page as the user scrolls near the bottom. Paging 3 automates all of that: loading, prefetching, showing spinners, retrying.

A PagingSource loads pages on demand; the Pager exposes a Flow<PagingData> the UI collects (RecyclerView's PagingDataAdapter or Compose's LazyPagingItems). It handles prefetch distance, placeholders, retry, and load-state UI for you.

For offline-first, add a RemoteMediator: it fetches from the network and writes into Room, while the PagingSource reads from Room. So the database is the source of truth and the network just tops it up — the same single-source-of-truth idea as the repository pattern, applied to pagination. Page keys can be simple indices or cursors.

Q50How did you cut build time 53%? (Gradle & modularization)Hard
In plain terms. Slow builds usually mean Gradle rebuilds more than it needs to. The fixes come down to: split the app so only the changed part recompiles, reuse previous build results via caching, and cut wasteful work in the build itself. At Dream11 this took builds from ~30 min to ~14 min (53%).

The big levers, roughly in impact order:

  • Modularization — split the monolith into feature/data modules so Gradle recompiles only what changed and builds the rest in parallel. Prefer implementation over api to keep the recompilation graph small (a change doesn't ripple to unrelated modules).
  • Build & configuration cache — reuse task outputs and the configuration phase across builds, both locally and as a shared remote cache in CI so teammates/CI reuse each other's work.
  • Dependency hygiene — remove unused dependencies, avoid dynamic versions (they defeat caching), enable parallel builds, and give Gradle enough JVM heap.
  • KSP over KAPT — KSP skips the Java stub generation that makes KAPT annotation processing slow.
  • CI specifics — warm the remote cache, shard tests, and avoid unnecessary clean builds.

Related win: Dynamic Feature Modules ship big rarely-used code (FFMPEG, Agora) on demand rather than in the base APK, which is how we also cut 15MB+ off install size.
