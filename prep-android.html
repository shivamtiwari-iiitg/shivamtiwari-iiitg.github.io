01 Kotlin language

Language internals interviewers actually probe.

Q01Does val guarantee immutability?Easy

No. val only makes the reference read-only — you can't reassign it. The object it points to can still mutate.

val list = mutableListOf("a")
list.add("b")   // fine — the list mutates
// list = mutableListOf()  // compile error — reassignment

A val can even return different values if it's backed by a custom getter. True immutability needs an immutable type (e.g. List instead of MutableList, or a read-only val with no custom getter).

Use lateinit for dependencies injected after construction (DI, onCreate). Use lazy for expensive values you may never need. Reading an uninitialized lateinit throws UninitializedPropertyAccessException; guard with ::prop.isInitialized.

Q03What does the compiler generate for a data class?Easy

For properties in the primary constructor it generates equals(), hashCode(), toString(), copy() and componentN() (for destructuring).

Gotcha: only primary-constructor properties are considered. A property declared in the body is excluded from equals/hashCode/copy — a common source of "why aren't these equal?" bugs. Also avoid data class for entities with identity (use a real id) since structural equality may not be what you want.

Q04sealed class vs enum — and what's a sealed interface?Mid

An enum is a fixed set of singleton instances, all the same shape. A sealed class is a closed set of subtypes that can each carry different data and have multiple instances.

sealed interface UiState
data object Loading : UiState
data class Success(val data: List<Item>) : UiState
data class Error(val msg: String) : UiState

Because the hierarchy is closed, when over a sealed type is exhaustive — no else needed, and adding a new subtype turns missed branches into compile errors. sealed interface (Kotlin 1.5+) lets a type participate in several sealed hierarchies and avoids forcing a class. Ideal for modeling UI state.

Q05What does inline do, and why crossinline / noinline?Mid

inline copies the function body — and its lambda bodies — into the call site at compile time. This removes the Function object allocation per lambda and enables non-local returns and reified generics. It's why map, filter, let are cheap.

  • noinline — keep a specific lambda not inlined (so you can store or pass it on).
  • crossinline — still inlined, but forbids non-local returns (needed when the lambda is invoked from another execution context, e.g. inside a Runnable).

Don't inline large functions — it bloats bytecode at every call site.

Q06What problem do reified type parameters solve?Mid

The JVM erases generics at runtime, so normally you can't do T::class or is T. In an inline function, marking the type reified makes the concrete type available at the call site.

inline fun <reified T> Gson.fromJson(json: String): T =
    fromJson(json, T::class.java)

val user: User = gson.fromJson(body)  // no .class needed
Q07How are extension functions resolved — statically or dynamically?Mid

Statically, by the declared type, not the runtime type. They compile to static methods taking the receiver as the first argument — they don't actually modify the class or participate in virtual dispatch.

open class A; class B : A()
fun A.name() = "A"
fun B.name() = "B"
val a: A = B()
a.name()  // "A" — resolved by declared type A

If a member function and an extension have the same signature, the member wins.

Q09Explain property delegation (by) with a real use.Mid

Delegation routes a property's get/set to a delegate object that implements getValue/setValue. Built-ins: lazy, Delegates.observable, Delegates.notNull, and map-backed properties.

var name: String by Delegates.observable("") { _, old, new ->
    log("$old -> $new")
}
// Fragment args, viewModels(), preferences are all delegates
private val viewModel: HomeViewModel by viewModels()
Q10companion object vs Java static?Easy

Kotlin has no static. A companion object is a single instance tied to the class; its members are accessed via the class name. Unlike static, it's a real object — it can implement interfaces and be passed around. Annotate members with @JvmStatic to expose true statics for Java interop. Top-level functions are the idiomatic replacement for genuine utility statics.

02 Coroutines & Flow

Structured concurrency and reactive streams — the highest-signal Android topic.

Q11What is structured concurrency and why does it matter?Mid

Every coroutine runs inside a CoroutineScope and becomes a child of a parent Job. That gives three guarantees: a parent doesn't complete until its children do; cancelling the parent cancels all children; and a child failure propagates up. The practical payoff is no leaked coroutines — when a ViewModel clears, viewModelScope cancels everything it launched.

Q12When do you use Main, IO, Default and Unconfined dispatchers?Easy
  • Main — UI work and touching views.
  • IO — blocking I/O (disk, network, DB). Large elastic pool (default 64 threads).
  • Default — CPU-bound work (parsing, sorting, image processing). Pool sized to cores.
  • Unconfined — starts in the caller's thread, resumes wherever the suspending call resumes. Rarely used outside tests/edge cases.
Make suspend functions main-safe: shift threads inside the function with withContext(IO) rather than forcing callers to choose a dispatcher.
Q12blaunch vs async?Easy

launch fires a coroutine that returns a Job — use it for fire-and-forget work. async returns a Deferred<T> whose result you await() — use it for parallel decomposition.

coroutineScope {
    val user = async { api.user() }
    val feed = async { api.feed() }
    render(user.await(), feed.await())   // run in parallel
}

Note: an exception in async surfaces at await(), whereas launch throws into the scope immediately.

Q13coroutineScope vs supervisorScope?Mid

In coroutineScope, one child failing cancels its siblings and the scope. In supervisorScope (backed by a SupervisorJob), children fail independently — one crash doesn't tear down the others. Use supervisor scoping when you launch several independent jobs (e.g. loading widgets on a dashboard) and want partial success.

Q14How does coroutine cancellation actually work?Hard

Cancellation is cooperative. Cancelling sets the Job to cancelling; suspension points (from kotlinx.coroutines) check this and throw CancellationException. A tight CPU loop with no suspension points will not stop — you must cooperate via ensureActive(), yield(), or checking isActive.

while (isActive) { stepHeavyWork() }   // cancellable loop

Don't swallow CancellationException in a broad catch (e: Exception) — rethrow it. To run cleanup that must finish (e.g. closing a resource) use withContext(NonCancellable).

Q15How do you handle exceptions in coroutines?Hard
  • try/catch works around a suspending call for localized handling.
  • launch exceptions propagate up the job tree; async exceptions are deferred to await().
  • A CoroutineExceptionHandler installed on a root launch is the last-resort net (uncaught handler). It does not work for async nor in the middle of a tree.
  • A SupervisorJob stops a child failure from cancelling siblings.
Q16What is a Flow and what makes it "cold"?Mid

A Flow is an asynchronous, sequential stream of values built on suspension. It's cold: the producer block doesn't run until a terminal operator (collect) subscribes, and it re-executes for each new collector.

fun ticker() = flow {
    var i = 0
    while (true) { emit(i++); delay(1000) }
}   // nothing runs until someone collects
Q17StateFlow vs SharedFlow vs LiveData?Hard

All are hot holders, but they differ in semantics:

  • StateFlow — always has a current value, conflated, emits only on change (distinctUntilChanged). Perfect for UI state.
  • SharedFlow — configurable replay and buffer, emits every value (including duplicates). Perfect for one-off events (navigation, snackbars) with replay = 0.
  • LiveData — lifecycle-aware out of the box, main-thread only, Java-friendly. Flows need repeatOnLifecycle for safe collection.
For events, prefer SharedFlow(replay = 0) over StateFlow — a conflated StateFlow can drop or replay an event on config change.
Q18What does flowOn do and why not just withContext?Mid

flowOn(dispatcher) changes the context of the upstream (everything before it) without affecting the collector. You can't wrap a flow's emission in withContext because a flow must preserve context across emit boundaries (it'd throw IllegalStateException). The collector's context (e.g. Main) stays untouched.

repo.dataFlow()
    .map { transform(it) }
    .flowOn(Dispatchers.Default)   // upstream on Default
    .collect { render(it) }        // collected on caller (Main)
Q19Implement search-as-you-type with Flow operators.Hard 
queryFlow
    .debounce(300)                 // wait for typing to settle
    .filter { it.length >= 2 }
    .distinctUntilChanged()
    .flatMapLatest { q ->          // cancel stale request
        repo.search(q).catch { emit(emptyList()) }
    }
    .flowOn(Dispatchers.IO)
    .collect { results -> _state.value = results }

flatMapLatest is the key — when a new query arrives it cancels the in-flight inner flow, so you never render results for a stale keystroke. Use conflate() to drop intermediate values when the collector is slow, and buffer() to decouple a fast producer from a slow consumer.

Q20How do you turn a callback API into a Flow?Mid 
fun locationFlow(client: LocationClient) = callbackFlow {
    val cb = LocationCallback { loc -> trySend(loc) }
    client.register(cb)
    awaitClose { client.unregister(cb) }   // cleanup on cancel
}

callbackFlow + awaitClose bridges listener-based APIs and guarantees the listener is removed when the collector cancels — the canonical way to wrap sensors, Firebase listeners, etc.

Q21Why collect with repeatOnLifecycle instead of lifecycleScope.launch?Hard

A plain lifecycleScope.launch { flow.collect {} } keeps collecting while the UI is in the background, wasting work and risking updates to a stopped view. repeatOnLifecycle(STARTED) starts collection when the lifecycle reaches STARTED and cancels it at STOPPED, restarting on the next STARTED.

lifecycleScope.launch {
    repeatOnLifecycle(Lifecycle.State.STARTED) {
        viewModel.uiState.collect { render(it) }
    }
}

Or use flow.flowWithLifecycle(lifecycle) for a single flow.

03 Core & Lifecycle

The framework mechanics behind every screen.

Q22Walk the Activity lifecycle and where you'd do what.Easy

onCreate (inflate, bind, restore state) → onStart (visible) → onResume (foreground, interactive) → onPause (losing focus, release exclusive resources — keep it fast) → onStop (hidden, persist state) → onDestroy.

On rotation the activity is destroyed and recreated; survive transient UI data with SavedStateHandle/ViewModel, not member fields. onSaveInstanceState handles small, serializable UI state for process death.

Q23Why does a Fragment have two lifecycles?Hard

A Fragment instance can outlive its view (e.g. on the back stack the view is destroyed but the fragment isn't). So there's the fragment lifecycle and a separate view lifecycle (onCreateView → onDestroyView).

Always observe LiveData / collect flows tied to the UI with viewLifecycleOwner, not this. Using the fragment as owner re-attaches observers to a stale view after the view is recreated → duplicate callbacks and leaks. Also null out binding in onDestroyView.
Q24Explain Looper, Handler and MessageQueue.Hard

This trio is Android's per-thread event loop:

  • MessageQueue — a time-ordered queue of Message/Runnable items.
  • Looper — loops forever pulling the next due message off the queue and dispatching it. The main thread's Looper is what keeps the app alive and processes input, draw, lifecycle callbacks.
  • Handler — bound to a Looper; you post/send messages through it, and it runs their callbacks on that Looper's thread.
Handler(Looper.getMainLooper()).post { updateUi() }

To make a worker thread loop-able: Looper.prepare() → create handlers → Looper.loop() (or just use HandlerThread). Coroutines/Executors have largely replaced manual handlers, but the main-thread dispatch model is exactly this.

Q25Application Context vs Activity Context — which to pass where?Mid

Use the Activity context for UI things bound to that activity — inflating views, themed dialogs, starting activities. Use the Application context for things that outlive any screen — singletons, databases, long-lived caches. Holding an Activity context in a singleton or static field leaks the whole Activity (and its view tree). Rule of thumb: if it needs theming or a window, use Activity context; otherwise Application.

Q26How do you survive process death (not just rotation)?Hard

A ViewModel survives configuration changes but not process death — the OS can kill a backgrounded app and recreate it later. To restore essential UI state, use SavedStateHandle (it persists to the saved-instance bundle):

class SearchViewModel(val handle: SavedStateHandle) : ViewModel() {
    val query = handle.getStateFlow("q", "")
    fun setQuery(q: String) { handle["q"] = q }
}

Keep it small and serializable. Source-of-truth data should be re-fetchable from the repository.

Q27What causes an ANR and how do you debug one?Hard

An ANR fires when the main thread is blocked too long: ~5s for input dispatch, ~10s for a foreground BroadcastReceiver, or a slow Service start. Causes: heavy work on the main thread (DB/network/JSON), main-thread lock contention, a deadlock, or a binder call to a slow process.

Debug with the /data/anr/traces.txt stack dump (shows what the main thread was doing), Play Console ANR vitals, StrictMode to catch main-thread I/O in dev, and Macrobenchmark/Perfetto traces. Fix by moving work off-main with coroutines and keeping onCreate/onResume lean.

Q28WorkManager vs Service vs coroutine — how do you choose?Mid
  • Coroutine in viewModelScope — work tied to a screen, no persistence needed.
  • WorkManager — deferrable, guaranteed background work that must survive app death/reboot (sync, upload, periodic cleanup), with constraints (network, charging) and backoff.
  • Foreground Service — user-visible ongoing work that must run now and keep running (music playback, active navigation, live location) with a persistent notification.

Since Android 12+, background-start restrictions make WorkManager the default for most "do it later, reliably" work.

04 Views & Compose

From RecyclerView recycling to recomposition — the performance heart of the UI.

Q29How does RecyclerView recycle, and why DiffUtil?Mid

RecyclerView keeps a small pool of ViewHolders — roughly one per visible row plus a few. As rows scroll off, their holders are returned to the pool and onBindViewHolder rebinds them for new data, so inflation cost is paid once. DiffUtil (via ListAdapter) computes the minimal set of changes between old and new lists so only changed rows animate/rebind instead of notifyDataSetChanged() redrawing everything.

class Diff : DiffUtil.ItemCallback<Item>() {
    override fun areItemsTheSame(a: Item, b: Item) = a.id == b.id
    override fun areContentsTheSame(a: Item, b: Item) = a == b
}
Q30What is overdraw and how did you reduce jank on busy screens?Mid

The view pipeline is measure → layout → draw, run per frame. Overdraw is the GPU painting the same pixel multiple times (stacked opaque backgrounds). Deep nested hierarchies multiply measure/layout cost.

Levers that cut jank: flatten hierarchy (ConstraintLayout over nested LinearLayouts), remove redundant backgrounds, use DiffUtil to avoid full rebinds, avoid layout passes inside scroll, profile with the GPU Overdraw debug overlay and the Layout Inspector, and watch frame timing in Perfetto. On a chatroom feed this is how I took jank/re-rendering down ~80%.

Q31What is recomposition and when does it trigger?Mid

Recomposition is Compose re-running a composable to update the UI tree when a State it reads changes. It's smart — only composables that read the changed state recompose, and it can skip ones whose inputs are unchanged (if their parameters are stable). Composables must be side-effect-free and idempotent because they may run frequently, in any order, and be skipped.

Q32remember vs rememberSaveable, and state hoisting?Mid

remember caches a value across recompositions but is lost on config change. rememberSaveable additionally survives config change/process death via the saved-state bundle (needs a Saver for custom types).

State hoisting = move state up to the caller and pass value down + onValueChange up, making the composable stateless, reusable and testable (unidirectional data flow).

@Composable
fun Counter(count: Int, onInc: () -> Unit) {   // stateless
    Button(onClick = onInc) { Text("$count") }
}
Q33Why do unstable parameters hurt Compose performance?Hard

Compose skips a composable only if all its inputs are stable and unchanged. A type is stable if Compose can trust its equals and know when public properties change. List (the interface) is treated as unstable because the runtime can't prove it won't mutate — so passing a List can defeat skipping and cause needless recomposition.

Fixes: use ImmutableList from kotlinx-collections-immutable, mark types @Immutable/@Stable when you guarantee the contract, hoist lambdas with remember, and use the Compose compiler stability/recomposition reports to find offenders.

Q34When do you use LaunchedEffect vs DisposableEffect vs rememberCoroutineScope?Hard
  • LaunchedEffect(key) — run a suspend block when entering composition or when key changes; cancelled on leave. Use for one-shot loads, snackbars, animations.
  • DisposableEffect(key) — register/unregister non-suspending resources (listeners, observers) with an onDispose cleanup.
  • rememberCoroutineScope — a scope tied to the composition you launch from event callbacks (e.g. onClick), not during composition.
  • SideEffect — publish Compose state to non-Compose code after every successful recomposition.
Never launch coroutines directly in the composable body — use these effect APIs so work is scoped to the composition lifecycle.
Q35Two Compose gotchas: LazyColumn keys and Modifier order.Mid

Keys: give items(list, key = { it.id }) a stable key so Compose preserves item state and animates correctly on reorder/insert — without keys it falls back to position and mis-recomposes.

Modifier order matters — modifiers apply outside-in. padding().background() paints the background inside the padding; background().padding() paints it around. Same chain, different result.

Bonus: wrap derived UI flags in derivedStateOf so they only recompute (and trigger recomposition) when the derived result actually changes, e.g. val showFab by remember { derivedStateOf { listState.firstVisibleItemIndex > 0 } }.

05 Architecture

Patterns that keep a large codebase testable and maintainable.

Q36MVVM vs MVI vs MVP — trade-offs?Mid
  • MVP — Presenter holds UI logic and calls an explicit View interface. Testable but verbose, and the two-way View↔Presenter coupling is easy to get wrong on config change.
  • MVVM — ViewModel exposes observable state; the View reacts. Less boilerplate, survives config changes, the Android-default with Jetpack.
  • MVI — a single immutable state object updated by reducing intents; strict unidirectional flow. Great for complex screens (predictable, time-travel-debuggable) but more ceremony.

In practice: MVVM with a single immutable UiState (StateFlow) borrows MVI's predictability without the full reducer overhead — what I used for the ShareChat chatroom and intervention surfaces.

Q37Explain Clean Architecture layers and the dependency rule.Mid

Three layers: presentation (ViewModel/UI) → domain (use cases + entities, pure Kotlin, no Android) → data (repositories, network, DB). The dependency rule: dependencies point inward; domain knows nothing about Android or data sources. Repositories are defined as interfaces in domain and implemented in data, so you can swap a data source or test the domain with no framework.

Use cases (interactors) encapsulate one business action (GetFeedUseCase), keeping ViewModels thin. Don't over-apply it to trivial screens — the indirection has a cost.

Q38What does "single source of truth" mean in a repository?Mid

The UI observes one authoritative store — usually the local database — and the network only ever writes into that store. The repository exposes a Flow from the DB; a refresh fetches from network and upserts to the DB, which emits to observers automatically.

fun articles(): Flow<List<Article>> = dao.observeAll()
suspend fun refresh() { dao.upsert(api.fetch()) }

This gives offline support, consistent data after writes, and a clean reactive contract — no merging two streams in the UI.

06 Dependency Injection

Dagger/Hilt mechanics — frequently the make-or-break Android round.

Q39Why DI, and what does Hilt add over Dagger?Easy

DI gives a class its collaborators instead of constructing them — decoupling, easy mocking in tests, and one place to control lifetimes. Dagger generates this wiring at compile time (fast, type-safe, no reflection). Hilt is a layer on Dagger that provides standard Android components and scopes (@HiltAndroidApp, @AndroidEntryPoint, @HiltViewModel) so you write far less boilerplate and get a consistent component hierarchy out of the box.

Q40@Binds vs @Provides — when each?Mid

Both tell Dagger how to satisfy a type. @Provides is a method whose body constructs the object — use it when you need logic, third-party types, or builders. @Binds just maps an interface to an implementation Dagger already knows how to create — it's an abstract method, generates less code and is preferred for interface→impl.

@Module @InstallIn(SingletonComponent::class)
abstract class RepoModule {
    @Binds abstract fun bindRepo(impl: UserRepoImpl): UserRepo
}
@Module @InstallIn(SingletonComponent::class)
object NetModule {
    @Provides @Singleton fun retrofit(): Retrofit = Retrofit.Builder()...build()
}

Rule: can the impl be constructor-injected and you just need the interface? @Binds. Need to build it yourself? @Provides.

Q41Walk through Hilt components and scopes.Hard

Hilt has a fixed component hierarchy, each with a scope annotation and a lifetime:

ComponentScopeLives as long as
SingletonComponent@Singletonthe Application
ActivityRetainedComponent@ActivityRetainedacross config changes
ViewModelComponent@ViewModelScopedthe ViewModel
ActivityComponent@ActivityScopedthe Activity
FragmentComponent@FragmentScopedthe Fragment

A scope means "one instance per that component lifetime". Unscoped bindings give a new instance every injection. Key constraint: a binding can only depend on bindings from its own component or an ancestor — you can't inject an @ActivityScoped thing into a @Singleton. Don't over-scope; scoping a stateless object just pins memory.

Q42Two impls of the same type — qualifiers and multibinding?Mid

Qualifiers disambiguate two bindings of the same type:

@Qualifier annotation class Authed
@Provides @Authed fun authedClient(): OkHttpClient = ...
// inject:  @Authed client: OkHttpClient

Multibinding (@IntoSet, @IntoMap) collects many contributions into a Set/Map — perfect for plugin-style registries like a set of analytics trackers or a map of ViewModel factories, so feature modules add to a collection without the consumer knowing them.

07 Memory & Performance

Leaks, caching, startup and app-size — where senior engineers earn their title.

Q43Most common Android memory leaks and how you find them?Hard

A leak is a longer-lived object holding a shorter-lived one, blocking GC. Usual suspects:

  • A static/singleton holding an Activity or View context.
  • Non-static inner classes / anonymous listeners (implicit outer reference) outliving the Activity.
  • Handler with delayed messages keeping the Activity alive.
  • Unregistered listeners/observers, unclosed disposables, RxJava subscriptions.
  • Observing LiveData/flows with the fragment instead of viewLifecycleOwner.

Detect with LeakCanary (auto heap-dump + leak trace), the Android Studio Memory Profiler (allocations, heap dumps, retained set), and StrictMode. LeakCanary watches destroyed objects, forces GC, and if still reachable computes the shortest strong reference path to a GC root.

Q44How does an LRU cache work, and where would you use one?Mid

Least-Recently-Used eviction: when full, drop the entry untouched longest. Classic implementation is a HashMap + doubly-linked list (O(1) get/put, move-to-front on access). Android's LruCache caps by a size you define (e.g. bytes for bitmaps via sizeOf).

val max = (Runtime.getRuntime().maxMemory() / 1024 / 8).toInt()
val cache = object : LruCache<String, Bitmap>(max) {
    override fun sizeOf(k: String, b: Bitmap) = b.byteCount / 1024
}

Use it for in-memory bitmap/image caches (the memory tier of an image loader), backed by a disk LRU for the next tier.

Q45What are Baseline Profiles and how do they help startup?Hard

A Baseline Profile lists the hot code paths (startup, key journeys). At install time ART AOT-compiles those paths instead of relying on JIT warming up at runtime, cutting cold-start time and early-frame jank — commonly 20–30% startup improvement. You generate it with the Macrobenchmark library running a representative journey, ship it in the APK/AAB, and verify with cold-start traces. Pair with the App Startup library to order initializers and avoid a slow custom Application.onCreate.

Q46What does R8 do, and what breaks without keep rules?Mid

R8 (the default since AGP 3.4) does shrinking (drop unused code/resources), obfuscation (rename to short names), and optimization (inline, etc.) in one pass — smaller, faster APKs. It breaks code reached via reflection or serialization that R8 can't see statically (Gson/Moshi models, JNI-called methods, classes named in XML). Add -keep rules for those, or use library-provided consumer rules. Always test the minified release build, not just debug.

Q47How do you load a huge image without OOM?Mid

A bitmap costs width × height × bytesPerPixel in memory regardless of file size, so a 12MP photo can be tens of MB decoded. Downsample to the target view size: read bounds first with inJustDecodeBounds, compute a power-of-two inSampleSize, then decode. In practice use Coil/Glide which handle sampling, pooling, memory+disk caching and lifecycle for you — that's the right answer in an interview too.

08 Data, Build & Tooling

Persistence, modularization and the build pipeline I love optimizing.

Q48Room essentials: relations, @Transaction, reactive queries.Mid

Room maps SQLite to typed DAOs with compile-time-checked SQL. A DAO method returning Flow<T> emits automatically whenever the underlying tables change — the basis of single-source-of-truth. Use @Transaction for multi-step reads/writes that must be atomic and consistent (e.g. a @Relation query that runs several statements). Provide explicit Migrations (or fallbackToDestructiveMigration only in dev) and test them. DataStore replaces SharedPreferences for key-value because it's async, transactional and flow-based — no main-thread disk I/O.

Q49How does Paging 3 work end to end?Hard

A PagingSource loads pages on demand; the Pager exposes a Flow<PagingData> the UI collects (RecyclerView PagingDataAdapter or Compose LazyPagingItems). It handles prefetch, placeholders, retry and load states for you. For offline-first, a RemoteMediator fetches from network and writes to Room, and the PagingSource reads from Room — so the DB is the source of truth and the network just tops it up. Keys can be page indices or cursors.

Q50How did you cut build time 53%? (Gradle & modularization)Hard

The big levers, roughly in impact order:

  • Modularization — split the monolith into feature/data modules so Gradle compiles only what changed and parallelizes the rest. Depend on api sparingly; prefer implementation to shrink the recompilation graph.
  • Build & configuration cache — reuse task outputs and the configuration phase across builds (local + remote cache in CI).
  • Dependency hygiene — drop unused deps, avoid dynamic versions, enable parallel and adequate JVM/Gradle heap.
  • Annotation processing — KSP over KAPT (KSP skips the Java stub generation that makes KAPT slow).
  • CI specifics — warm remote build cache, shard tests, avoid clean builds where safe.

Dynamic Feature Modules also help install size: on-demand modules (e.g. FFMPEG/Agora) ship later, which is how we cut 15MB+ off the base APK.
